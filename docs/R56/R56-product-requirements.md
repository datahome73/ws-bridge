## 1. 问题背景

### 1.1 R55 自动驾驶管线上线后的实际状态

R55 实现了自动驾驶管线的 6 方向（A:放开角色校验 / B:退回命令 / C:git验证 / D:状态可视化 / E:模式开关 / F:定向发送），**30 项验收全绿，4 项审查警告全部修复**。

但实际生产运行效果：

| 问题 | 现象 | 根因线索 |
|:-----|:------|:---------|
| 🔴 管线推进后无人接手 | Step 交接系统通知发出去，但下一棒 bot 不响应 | 通知未到达 bot（离线/频道未切换） |
| 🔴 每 Step 需项目负责人 TG 私聊激活 | 项目负责人私聊转发 code block 后才动 | 手动中转是唯一可靠的触发方式 |
| 🔴 workspace 广播无回音 | 工作室内的消息其他 bot 收不到 | 通信层在频道切换或消息路由处断裂 |
| 🟡 调命令成功 ≠ 实际执行 | `!step_complete` 返回 ✅ 但无人行动 | 服务端状态推进成功，但通知层失败了 |

### 1.2 R55 审查 W-4 指向的根因

R55 代码审查已发现 1 个关键警告（W-4）：

> `_send_to_agent` 使用原始 WebSocket 帧发送定向通知。如果目标 bot 当时离线或连接已断开，通知消息**静默丢失**，没有任何回退机制（没有 `write_chat_log`、没有 `save_message`、没有 fallback 到广播）。

```python
conns = _connections.get(agent_id, set())
if not conns:
    return False  # 无人接收 → 静默丢
```

**这意味着：** R55 方向 F 的「定向发送」在实施上解决了回声问题，但引入了更严重的问题——通知可靠性从「广播（所有 bot 都收到）」降级为「定向（只有一个 bot 收，收不到就丢）」。

### 1.3 过渡期的核心矛盾

```
自动驾驶管线（想实现）                   实际状态（能实现）
┌──────────────────┐                  ┌──────────────────┐
│ Step N 完成       │                  │ Step N 完成      │
│   ↓               │                  │   ↓              │
│ 自动通知下一棒 bot │  ✗  通知静默丢失   │ 项目负责人 TG 私聊     │
│   ↓               │                  │   ↓              │
│ bot 自动接管工作   │                  │ 项目负责人转发 code    │
│   ↓               │                  │    block 到工作群  │
│ 全自动流转        │                  │ bot 手动接管     │
└──────────────────┘                  └──────────────────┘
     理想状态                             当前实际
```

**核心认识：** 在全自动通信层修复前，过渡期需要的是一个**可靠的半自动流程**，不是一个「看起来自动但实际经常断」的假自动流程。

## 2. 方向说明

### 方向 A：修复 Step 交接通知可靠性 🔴 P0

| 项目 | 说明 |
|:-----|:------|
| **问题** | Step 交接的定向通知（`_send_to_agent`）在目标 bot 离线或连接异常时静默丢失，没有持久化回退 |
| **目标** | Step 交接通知必须有可靠的送达机制：bot 在线时实时送达，离线时持久化到工作室频道消息，bot 恢复后能读到 |
| **设计哲学** | 过渡期的通知策略是 **"先不丢，再求静"**——宁可全体广播也不可无声丢失。在可靠性达到 100% 之前，不要为了减少回声而降低送达率 |

#### 技术方案方向

| 方案 | 说明 | 改动量 |
|:-----|:------|:------:|
| **A-a（推荐）** | `_send_to_agent` 先尝试定向发送；如果失败（`return False`），回退到 `write_chat_log` 写入工作室频道 + 工作室全广播（同旧行为）。这样：在线 bot 无回声，离线 bot 上线后能看到历史 | ~5 行 |
| **A-b** | 完全恢复 R55 前的全广播模式。工作室全广播 + `_admin` 频道日志。回声问题通过行为规则约束（规则 6：不回声） | ~0 行（回退方向 F） |
| **A-c** | 新建离线消息队列：`_offline_message_store[agent_id]` 持久化，bot 重连时推送 | ~30 行 |

> 技术方案（具体实现方式）由架构师决定。

#### 验收标准

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| A-1 | 目标 bot 在线时，`!step_complete` 的交接通知定向送达，其他 bot 不收到 | 实测 Step 交接 |
| A-2 | 目标 bot 离线时，通知回退到工作室频道广播，不静默丢失 | 断开目标 bot 再调 `!step_complete` |
| A-3 | 离线 bot 重新上线后，能在工作室频道历史中读到通知 | 重连后检查工作室消息 |
| A-4 | `_admin` 频道保留完整的 Step 交接日志（不受定向/回退影响） | 检查 admin 日志 |

### 方向 C：过渡期可靠协调流程 🟡 P1

| 项目 | 说明 |
|:-----|:------|
| **问题** | 在全自动通信不可靠时，缺乏一个**确定性可靠**的过渡期流程。当前的做法是「遇到问题再协调」（post-hoc），不是「预防性流程」（proactive） |
| **目标** | 设计并执行一套过渡期管线协调流程，确保每步都能可靠推进，同时为全自动积累条件 |

#### 过渡期流程设计哲学

```
全自动管线 = 系统100%触发 + 无人干预
  ↓ 过渡期目标
高可靠半自动 = 系统多数触发 + PM监督 + 项目负责人兜底
  ↓ 当前状态
人工拼凑 = 靠人发现卡住 → 项目负责人TG协调
```

**R56 的目标是从「人工拼凑」进化到「高可靠半自动」。**

#### 过渡期流程（初稿）

```
1. !pipeline_start R{N} 触发 → 建工作室 ✅（已有）
2. PM 验证点名全员到位（!pipeline_status + 工作室可见性检查）
   ↓
3. 各 Step 自动推进（方向 A 修复后，通知可靠性提升）
    PM 监控:
      - !pipeline_status 检查 Step 指针是否已前移
      - 工作室检视目标 bot 是否回复确认
   ↓
4. 如果 Step 在 5 分钟内无人响应:
    => PM 准备 code block 上下文（@接收人 + 任务描述）
    => TG 私聊项目负责人，请求转发激活
    => 项目负责人转发到工作群
   ↓
5. Step 完成确认:
    => !step_complete stepN --output <sha>（自动前移指针）
    => PM 验证 !pipeline_status 确认推进到下一 Step
   ↓
6. 循环 3→5 直到 Step 6 合并部署
```

#### 关键改进点

| 改进 | 当前行为 | 过渡期目标行为 |
|:-----|:--------|:--------------|
| 超时阈值 | 无明确标准（等人发现卡住） | 明确 5 分钟无响应 → 主动升级 |
| 升级链路 | 发现卡住 → 找项目负责人 | Step 完成 → PM 自动检视 → 超时主动升级 |
| 可靠通知 | `_send_to_agent` 静默丢失 | A-a: 定向+回退广播双保险 |

#### 验收标准

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| C-1 | PM 能在 Step 完成后 1 分钟内确认下一角色是否已响应 | 工作室检视 |
| C-2 | Step 在 5 分钟内无响应 → PM 自动准备升级 code block | 计时观察 |
| C-3 | 一轮管线全部 6 步能在合理时间内完成推进（不超过上一轮耗时） | 对比 R55 历史 |
| C-4 | 项目负责人 TG 私聊激活次数从「每步必发」降到「偶尔兜底」 | 统计触发次数 |

> 技术方案（流程细节和 PM 操作 SOP）由架构师与 PM 共同决定。

## 3. 过渡期行为规则（正式写入）

### 规则 1：PM 主动监控不是可选项

| Step 推进后 | PM 必须做的事 |
|:------------|:-------------|
| `!step_complete` 执行后 30 秒 | 检查 `!pipeline_status` Step 指针是否前移 |
| Step 指针前移后 1 分钟 | 检查工作室中目标 bot 是否回复确认 |
| Step 指针前移后 5 分钟 | 目标 bot 无响应 → 准备升级 code block |
| 连续 2 步需项目负责人激活 | 暂停管线并向项目负责人报告「通信链路可能断裂，建议先修方向 A」 |

### 规则 2：升级 code block 格式统一

```
@接收人 任务描述
- 需求文档：{URL}
- 技术方案：{URL}
- 期望产出：{具体要求}
- 完成后 !step_complete stepN --output <sha>
```

### 规则 3：完成就必须推进

| 谁 | 做什么 |
|:---|:-------|
| 编码者 | 推 dev 后立即在工作室内 `!step_complete step3 --output <sha>` |
| 方案师 | 推方案后立即 `!step_complete step2 --output <sha>` |
| 审查者 | 审查通过后立即 `!step_complete step4 --output <sha>` |
| PM | 任何产出推 git 后、即使不是自己做的，也可 `!step_complete` 推进 |

## 6. 设计哲学

### 可靠优先于静默

在通信可靠性达到 100% 之前，「宁可全部播，不可无声丢」——全广播可以被行为规则约束回声，但丢通知没有任何恢复手段。

### 先跑通，再优化

R56 的目标不是一步达到全自动，而是让管线**可靠地跑完一轮**——即使其中 3/6 步需要过渡期人工触达。从「人工拼凑」进步到「高可靠半自动」，再在 R57+ 逐步逼近全自动。

### 诊断不靠猜

在修通信层之前，先用实际数据确认断裂点在哪。每个「应该是通的」假设都必须用生产环境的 WS 直连验证来证明或推翻。
