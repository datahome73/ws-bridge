## 一、4 步通信流程的理解

### 协议设计的核心思想

R82 后 ws-bridge **所有消息统一为 inbox 消息**，不再区分"广播"和"收件箱"。每条消息明确指定接收者（`channel: "_inbox:<agent_id>"`），发送者身份可追溯（`agent_id` / `from_agent`），回复路径直接由发送者 ID 决定——整个通信模型从"频道广播"转向"点对点投递"。

### 标准 4 步流程

```
小谷（PM）                          Bot
  │                                 │
  ├─ Step 1：派活 ─────────────────→│  channel: _inbox:<bot_id>
  │                                  │  内容含 SENDER_INBOX
  │←─── Step 2：ACK ────────────────┤  回复 _inbox:<sender_id>
  │                                  │  "ACK ✅ R85 收到！"
  │←─── Step 3：完成回复 ───────────┤  回复 _inbox:<sender_id>
  │                                  │  "✅ 完成，已推 dev: xxxx"
  ├─ Step 4：确认 ─────────────────→│  channel: _inbox:<bot_id>
```

**Step 1 — 派活：** PM 向 bot 收件箱发送任务消息，内容包含任务描述 + `SENDER_INBOX` 字段指明回复目标。

**Step 2 — ACK：** Bot **立即**（5 秒内）回复一条简短确认。回复目标为 `_inbox:<发件人 agent_id>`，内容如 `"ACK ✅ R85 收到！"`。此步只确认"消息已收到"，不承诺完成时间。

**Step 3 — 完成回复：** Bot 实际完成任务（git push dev 等）后，**再发一条**消息到 `_inbox:<发件人 agent_id>` 告知结果。格式如 `"✅ 完成，已推 dev: abc1234"`。

**Step 4 — 确认闭环：** PM 收到完成回复后，向 bot 回一条确认。**Bot 收到此确认后不得再回复**——这是整轮通信的终点，再回复会开启无意义循环。

### 关键设计要点

| 要点 | 说明 |
|:-----|:------|
| 回复目标是**发件人** | 用 `msg.agent_id` / `msg.from_agent`，不是消息的 `channel`（那是自己的收件箱） |
| 回复分**两步** | ACK 先确认收到，干完活再发完成——不能合并成一句话 |
| 不回复自己 | Server 会拒绝向自己收件箱发消息 |
| 多 bot 通知 | 依次给每个 bot 的 inbox 发消息，＋1 秒间隔避限速 |

## 二、回复格式规则

### 规则定位

Inbox 消息是**工作任务通信**，不是日常聊天。回复只应包含 PM 需要知道的信息，不需要展示思考过程或推理步骤。

### ❌ 四不准

| 禁止行为 | 说明 |
|:---------|:------|
| 不准输出思考过程 | 不写"我看到你的消息，正在思考…""让我分析一下…"等内部推理 |
| 不准闲聊寒暄 | 不问候、不寒暄、不闲聊——每一条都有 token 成本 |
| 不准复述对方 | 不写"收到你的消息: XXXX"原样回放 |
| 不准拆条 | 一次说完，不把一句话拆成多条消息 |

### ✅ 三句式规范

| 步骤 | 正确回复 | 字数 |
|:-----|:---------|:-----|
| Step 2 ACK | `ACK ✅ R85 收到！` | 1 句，10 字内 |
| Step 3 完成 | `✅ 完成，已推 dev: abc1234` | 1 句，15 字内 |
| Step 4 确认 | `✅ 已收到你的完成通知。` | 1 句，12 字内 |

**核心原则：** 回复只包含 PM 需要知道的信息——确认 / 结果 / 完成。不需要 PM 看你的思考过程或推理步骤。

## 三、个人体会

协议设计非常清晰——把"如何回复"的选择权交给 bot（消息内容格式不限、由 LLM 决定），但**强制约束通信流程**（ACK 先于完成、Step 4 是终点）。这种"流程约束 + 内容自由"的组合既保证了工作流的一致性，又不限制各 bot 的表达方式。

4 步流程中最容易出错的点是：
1. **Step 2 / Step 3 混淆**——有人把 ACK 和完成合并成一条，失去"先确认再干活"的时序意义
2. **回复目标搞错**——回给自己收件箱（server 拒绝）或回给错误频道
3. **Step 4 后继续回复**——闭环后不停止，导致无意义循环

回复格式规则的核心精神是**尊重双方注意力**：PM 只想知道"收到没"/"做完没"，bot 的思考过程对 PM 来说是噪音。
