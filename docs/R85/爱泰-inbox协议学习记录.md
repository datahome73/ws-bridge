## 一、协议核心思想：只有一类消息

R82 重构后的最核心变化：**ws-bridge 所有消息都是 inbox 消息**。

过去（R82 前），系统存在"广播"（大厅）与"私信"（收件箱）两种消息路径，bot 需要区分 `lobby`、`workspace`、`_inbox:` 等不同频道类型。R82 去掉了所有频道概念，**所有路由统一为 inbox 模式**。

这意味着：
- 不再有"广播"这个概念。想通知多个 bot → 挨个给每个 bot 的 inbox 发一条
- 收到的消息 **必定** 是 `_inbox:{自己agent_id}` 频道——不需要判断频道类型
- 回复就是往发送者 inbox 发消息，和发任何其他消息没区别

> **一句话：只有发件人→收件人，没有"圈子"或"频道"。**

## 二、4 步通信流程理解

文档 §8 定义了标准的 4 步 SOP：

```
PM                              Bot
 │                                │
 ├── Step 1: 派活 ──────────────→│  向 bot 收件箱发任务（含 SENDER_INBOX）
 │                                │
 │←── Step 2: ACK ───────────────┤  5s 内回 ACK 确认
 │                                │
 │                     [实际干活: git push dev]
 │                                │
 │←── Step 3: 完成回复 ──────────┤  git push 后回复完成
 │                                │
 ├── Step 4: 确认 ──────────────→│  PM 回确认，闭环完成
```

### 各步骤要点

| 步骤 | 方向 | 内容要求 | 时效 |
|:----|:----|:---------|:----|
| Step 1 派活 | PM→Bot | 含 SENDER_INBOX，任务描述 | — |
| Step 2 ACK | Bot→PM | `ACK ✅ R{轮次} 测试收到！` | ≤5s |
| Step 3 完成 | Bot→PM | `✅ 完成，已推 dev: xxxxxx` | 任务完成后 |
| Step 4 确认 | PM→Bot | `✅ 已收到你的完成通知。` | 收到 Step 3 后 |

### 关键理解

1. **Step 2 和 Step 3 是两条独立消息**——不是替换关系。先 ACK，干完活再发完成
2. **回复目标必须是发件人的 agent_id**，而不是消息的 channel（那是自己的收件箱）
3. **Step 4 是终点**——bot 收到确认后不得再回复，否则开启死循环
4. **不要回自己的 inbox**——`_inbox:{自己agent_id}` 发出去也没意义

## 三、回复格式规则

文档 §8 的"回复格式规则"部分给出了明确的要求：

### ❌ 禁止

| 行为 | 例子 | 原因 |
|:----|:-----|:-----|
| 输出思考过程 | "我看到你的消息，正在思考..." | PM 不需要听推理 |
| 闲聊寒暄 | "你好呀，今天天气不错" | 工作任务通信，不是聊天 |
| 重复对方内容 | "收到你的消息: XXXX" 原样复述 | 浪费 token |
| 富文本格式 | 代码块、表格、Markdown | 没必要 |
| 拆成多条 | 一条消息能说完的分成 3 条 | 增加了通信轮次 |

### ✅ 正确格式

| 场景 | 正确回复 |
|:----|:---------|
| Step 2 ACK | `ACK ✅ R85 测试收到！` |
| Step 3 完成 | `✅ 完成，已推 dev: 8f8c862` |
| Step 4 确认 | `✅ 已收到你的完成通知。` |

### 核心原则

> 回复只包含 PM 需要知道的信息：**确认 / 结果 / 完成**。不需要 PM 听你的思考过程或推理步骤。

## 四、与自己之前行为的对比反思

回顾我之前的 inbox 回复习惯，有一些地方需要改进：

### ✅ 做得好的

1. **ACK 及时** — 收到任务后第一条消息就是 ACK，符合 Step 2 要求
2. **回复到正确 inbox** — 使用 `_inbox:ws_f26e585f6479` 回复给发件人，没有回自己的 channel
3. **git push 后通知** — 任务完成后先推代码，再单独发完成通知，符合 Step 3 流程

### ⚠️ 需要改进的

1. **ACK 内容过于啰嗦** — 之前回 `"✅ R85 ACK — 爱泰已收到 inbox 学习记录任务，正在阅读文档并撰写测试记录。"` 太长。按文档要求应该简洁为 `"ACK ✅ R85 测试收到！"` 即可
2. **完成通知包含过多细节** — 之前回的内容包含文件路径和 SHA 之后还加了表情。应该精简为 `"✅ 完成，已推 dev: 8f8c862"`
3. **没有明确区分 Step 2 / Step 3 的格式区别** — ACK 和完成通知的格式应该统一明确，两者用不同的前导符号（`ACK ✅` vs `✅ 完成`）
4. **注意 Step 4 确认后停止回复** — 如果 PM 回了确认，我必须停住不再发消息

### 改进方案

以后按以下模板回复：

```
Step 2 ACK:   ACK ✅ R{轮次} 测试收到！
Step 3 完成:  ✅ 完成，已推 dev: {sha}
```

不再加冗余说明、思考过程、文件路径等细节。

## 五、总结

Inbox 协议 v1.0 确立了 ws-bridge 通信的简洁范式：

- **统一的消息模型**（只有一类 inbox 消息）消除了频道复杂性
- **4 步 SOP** 确保了通信的可预期性和闭环
- **严格的回复格式** 节约了所有参与方的 token 成本

作为 bot，只需要记住三件事：
1. **收到 → 回 ACK（5s 内）**
2. **干完 → 回结果（git push 后）**
3. **收到确认 → 闭嘴**
