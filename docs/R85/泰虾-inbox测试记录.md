## 一、阅读体会

以前总觉得 ws-bridge 的消息体系很复杂——有 lobby 广播、工作室频道、admin 频道、inbox 收件箱好几种通道，bot 需要搞清楚当前在哪个频道才能正确收发。R82 Inbox-Only 架构重构之后，整个模型变得极其简洁：

**只有一种消息：inbox 消息。**

所有消息都是发给某个特定接收者的，不再有「广播」这个概念。想通知多个 bot，就挨个发。像发邮件一样自然。

这种简化带来的好处很实在：
- bot 不用管什么活跃频道切换、MSG_SET_ACTIVE_CHANNEL 协议
- 收件箱消息直达，不会被 nonsense/duplicate 过滤误拦
- 回复就是发一条消息到发送者的 inbox，没有特殊操作

## 二、4 步通信流程的理解

协议文档定义的标准处理流程，我拆成 4 步：

### 第 1 步：收到消息 → 确认是给我的

```python
msg = json.loads(raw_data)
channel = msg.get("channel", "")
if not channel.startswith("_inbox:"):
    return  # 不是我该处理的
```

`channel` 以 `_inbox:` 开头，表示这是定向给我的消息。R82 之后没有其他类型的频道消息了，但这个检查仍然是防御性编程的好习惯。

### 第 2 步：提取发送者信息

```python
sender_id = msg.get("agent_id") or msg.get("from_agent") or ""
sender_name = msg.get("from_name", "?")
```

注意 `agent_id` 和 `from_agent` 两个字段都携带了发送者 ID，`from_name` 是显示用的。

### 第 3 步：处理消息内容

交给 LLM 处理。根据 `content` 判断类型：
- **任务分配**（以 `📥` 开头）：先回 ACK 确认收到，完成后回结果
- **通知**（非任务）：按需回复
- **询问**（含问句）：处理完后回复

### 第 4 步：回复到发送者的 inbox

```python
if sender_id:
    await client.send_message(
        content="你的回复内容",
        channel=f"_inbox:{sender_id}",
    )
```

**回复不是特殊操作**——和发任何其他消息一样，就是往发送者的 inbox 发一条消息。

## 三、回复格式规则总结

| 规则 | 说明 |
|:-----|:------|
| **格式自由** | 回复内容没有限制，由 LLM 自行决定，建议简洁自然语言 |
| **任务 ACK** | 收到任务先回 ACK（`✅ 收到`），完成后回结果 |
| **回复目标** | 频道固定为 `_inbox:<发送者_agent_id>` |
| **发送者提取** | `msg.get("agent_id") or msg.get("from_agent")` |
| **多 bot 通知** | 逐个发 inbox 消息，中间间隔 1s 避免限速 |
| **ACK 超时** | ACK 超时不代表消息未送达，消息已投递到目标连接 |
| **消息去重** | 客户端通过 `seen_ids`（最多 500 条）自动去重 |

## 五、关键心得

1. **Inbox 协议的简洁之美**：从 4 种通道缩到 1 种，bot 代码心智负担大大降低
2. **回复即发消息**：回复不是特殊操作，就是往 `_inbox:{sender_id}` 发一条消息——这个设计很棒，统一了收发逻辑
3. **ACK 的时机**：任务消息先 ACK 再执行，让发送者知道我开始处理了；完成后通知结果，形成闭环
4. **header 字段兼容**：`agent_id` / `from_agent` 两个字段都看一遍，兼容老格式
