## 1. 背景与目标

### 1.1 问题

R117 修复了 `_auto_dispatch()` 的 card key → WS ID 桥接问题（`_resolve_card_key_to_ws_id()` 三策略 fallback），代码已合并到 main 并部署。但 **R117 从未在生产环境做全流程验证**：

- `##start##R{N}##requirements_url=...` 创建管线后，Step 1→6 是否真的能自动跑通？
- Bot 回复 `已完成 ✅ R{N} Step {M}` 后，`_try_advance_pipeline` 是否真的能正确推进到下一步并自动派活？
- 离线重试、PM 通知等机制是否存在未发现的断流点？

目前仍在靠 PM 手工派活，自动派活能力有等于无。

### 1.2 目标

| 优先级 | 目标 | 说明 |
|:------:|:-----|:------|
| 🥇 P0 | 全流程生产验证 | 在真实生产环境中跑通 `##start → Step 1→6 全自动` 一整套 |
| 🥈 P1 | 管线 Tab 倒序显示 | Web 仪表盘管线 Tab 卡片按最新在上排列，方便查看最新任务 |

### 1.3 基线

- R117 已在生产部署（`main.py` 含 `_resolve_card_key_to_ws_id` 三策略 fallback）
- `AUTO_DISPATCH_ENABLED = True`
- Bot 回复 `已完成 ✅ R{N} Step {M}` 格式正确即可触发自动推进

## 2. 功能需求

### 需求 A — 全流程生产验证

- **触发条件：** 用户审核通过本需求文档后
- **行为描述：** PM 在生产环境执行一次完整的自动派活管线验证：
  1. 写一个最小验证轮次（如 R118-test），改动不影响现有功能
  2. 用 `##start##R118-test` 创建管线（不带额外 kv，验证默认模板）
  3. 观察 Step 2 是否自动派活到 Arch
  4. Arch 完成（推简单改动）后观察 Step 3 是否自动派活到 Dev
  5. 逐步验证直到 Step 6 完成

- **验收标准：**
  - [ ] A-1：`##start##R118-test` 后 Step 2 自动派活到 Arch，无需 PM 确认
  - [ ] A-2：Arch 回复 `已完成 ✅ R118-test Step 2` 后 Step 3 自动派活到 Dev
  - [ ] A-3：Dev 回复格式正确后 Step 4 自动派活到 Review
  - [ ] A-4：全链路 Step 1→6 零人工介入完成
  - [ ] A-5：出问题时有明确的日志和错误信息（不静默吞错）
  - [ ] A-6：验证完成后记录 6 步各步的实际耗时、遇到的阻塞点

> **注：** 此需求是「验证」而非「开发」，不需要服务端代码改动。只测试已有功能在生产环境是否真的能跑通。

### 需求 B — Web 管线 Tab 倒序显示

- **触发条件：** 用户在 Web 端打开管线 Tab（Tab 4）
- **行为描述：** 管线列表卡片按创建时间倒序排列，最新的管线在最上面
- **当前问题：** 管线卡片按创建顺序正序排列，最新的管线在最下面，需滚动到底部才能看到

- **验收标准：**
  - [ ] B-1：新创建的管线出现在列表最顶部
  - [ ] B-2：倒序排列不影响已有管线数据的正确性
  - [ ] B-3：部署后 Ctrl+F5 刷新即可生效（无需清缓存）
  - [ ] B-4：改动仅限前端 HTML/JS 模板，不涉及后端 API

## 3. 方向决定

| 决定事项 | 选择 | 说明 |
|:--------|:----|:------|
| 验证轮次 | 用真实小改动验证 | 推一个简单文档或小注释到仓库，不影响任何功能 |
| 验证方式 | PM 在生产环境手动执行 | 逐个 Step 观察，记录结果，不需要自动化测试脚本 |
| 管线 Tab 改法 | 纯前端 JS 排序 | 前端一次性 sort，不修改后端 API 的返回顺序 |
| 不做事项 | 自动派活代码改动 | R117 代码已部署，本轮只验证不修改 |
| 不做事项 | 离线重试/PM 通知代码 | 这些属于 R118 后续功能，验证通过后再做 |
