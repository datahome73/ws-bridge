## 1. 问题背景

### 1.1 当前管线推进模式的问题

R53 上线了 ACK 确认制点名与派活，通信可靠性已解决。但 `!step_complete` 仍限制只有 step 负责人才能推进，导致以下痛点：

| 痛点 | 现象 | 影响 |
|:-----|:------|:------|
| 🔴 **角色卡住时无人能顶** | Step 负责人在线但不响应点名，其他人只能干等 | 管线阻塞，需要 PM 手动中转或等待超时 |
| 🔴 **产出完成但无人推进状态** | 某人的产出已推 git，但没调 `!step_complete`，管线状态停在原地 | R48/R53 反复出现，所有人以为在正常推进但实际已停摆 |
| 🔴 **退回无命令支持** | Step N 产出有缺陷，但只能口头在大厅反馈，无正式退回机制 | 缺少正式的审查→修复循环，问题易遗漏 |
| 🔴 **git 产出无人知道** | 不知道远程 git 是否有新 commit，需要人肉眼检查 | 推进时机依赖人工觉察，有时滞 |

### 1.2 已有设计基础

R53 已产出 **自动驾驶管线协作设计原则**（`docs/R53/R53-self-driving-pipeline-design.md`），定义了核心理念和行为规则：

> **核心原则：** 任何人都可以推进 step。共享 WORK_PLAN + step 状态 + git，相邻 step 互为审查，无产出则退回。

但设计原则中的 **5 项技术改动尚未编码实现**，且行为规则尚未写入 WORK_PLAN 和工作流程。

## 2. 方向说明

R55 将 [R54 产品需求](docs/R54/R54-product-requirements.md) 中的 5 项技术改动方向全部实现，同时将行为规则正式纳入 WORK_PLAN。

### 方向 A：放开 `!step_complete` 角色校验 🔴 P0

| 项目 | 说明 |
|:-----|:------|
| **当前** | `!step_complete stepN` 只允许 step 负责人执行，其他人调了权限不足 |
| **目标** | 允许任意成员推进任一处于 **pending** 的 step |
| **场景** | ① PM 看到开发已推 git 但干等 → 直接 `!step_complete step3 --output <sha>` 推进 ② 审查看到代码没问题 → 顺手推进到测试 |
| **约束** | ① 仅允许推进 pending 的 step（已完成/已退回应报错） ② 不可跳过当前活跃 step（防止抢活冲突） ③ 如果当前 step 的负责人刚调了命令，2 秒内的重复推进应被拒绝（序列化） |
| **服务端系统层** | ✅ — 纯规则校验，零 token |

> 技术方案（具体如何判定「pending」「活跃 step」、序列化控制）由架构师决定。

### 方向 B：新增 `!step_reject stepN --reason <原因>` 🔴 P0

| 项目 | 说明 |
|:-----|:------|
| **当前** | Step N 产出有问题时没有正式退回机制，只能口头说 |
| **目标** | 新增 `!step_reject` 命令，退回上一 step 并附理由 |
| **场景** | 开发提交了有 Bug 的代码，审查看到后在工作室内执行 `!step_reject step3 --reason "变量名阴影，见审查报告 §2.1"` → 系统自动将 step 指针回退到 step3，通知开发者修正 |
| **约束** | ① 退回必须附 reason（不能无声退回） ② 退回后自动更新 `!pipeline_status` 显示退回状态 ③ 单 step 最多退回 2 轮，超时自动升级给 PM 协调 |
| **服务端系统层** | ✅ — 纯规则状态机操作，零 token |

> 技术方案（退回后的状态转换逻辑、存量 task 处理、通知方式）由架构师决定。

### 方向 C：基于 git 的自动验证 🟡 P1

| 项目 | 说明 |
|:-----|:------|
| **当前** | `!step_complete stepN --output <sha>` 不会验证 commit 是否存在 |
| **目标** | `!step_complete` 收到 `--output` 参数时，自动检查远程 git 是否存在该 commit |
| **场景** | 开发说「编码完成了」但没推 git → 调 `!step_complete` 时系统检查远程不存在该 sha → 报错「❌ Commit xxx 不存在于远程 dev 分支」→ 阻止状态机空转 |
| **约束** | ① 检查 `origin/dev` 分支（非本地分支） ② `--output` 参数改为可选——如果不传，跳过 git 检查，直接推进 ③ 超时 10 秒，失败时降级为警告（不阻断推进） |
| **纯规则逻辑** | ✅ — `git ls-remote origin dev` 的 HTTP 请求，零 token |

> 技术方案（远程 git 验证的具体实现方式、仓库 URL 配置）由架构师决定。

### 方向 D：Step 状态关联可视化 🟡 P1

| 项目 | 说明 |
|:-----|:------|
| **当前** | `!pipeline_status` 只显示当前活跃 step，不显示哪些可推进、哪些被退回 |
| **目标** | `!pipeline_status` 输出显示：哪些 step 已完成（✅）、哪些可推进（▶）、哪些被退回（🔄）、哪些待启动（⏳）。被退回的 step 附退回次数和最近一次理由 |
| **场景** | PM 查看管线状态时一目了然：Step 3 显示 **🔄 退回（第 1 轮）— 变量名阴影** → 知道需要介入协调 |
| **约束** | ① 状态信息从持久化层读取（管线 state + task 状态） ② 退回次数和理由在退回时记录到管线 state 中 |
| **纯规则逻辑** | ✅ — 数据查询+格式化输出，零 token |

### 方向 E：自动驾驶模式开关 🟢 P2

| 项目 | 说明 |
|:-----|:------|
| **当前** | 所有管线按传统模式（角色限制推进）运行 |
| **目标** | 每个管线可设 `auto`（自动驾驶：任何人可推进、相邻审查）或 `manual`（传统模式：角色限制推进）模式，默认为 `auto` |
| **场景** | `!pipeline_start R55 --mode auto` 或在管线启动后 `!pipeline_mode auto` |
| **约束** | ① 模式存储在管线 state 中 ② 方向 A/B/C/D 的自动行为仅在 auto 模式下生效 |
| **纯规则逻辑** | ✅ — 配置开关，零 token |

## 3. 自动驾驶行为规则（正式写入）

以下规则为 R54 设计原则中确定的 **行为层规则**，不需要代码实现，直接在 WORK_PLAN 和 WORKFLOW.md 中声明，全体 bot 遵守：

### 规则 1：共享上下文，主动推进

任何人都可以推进 step。不需要等 step 负责人主动调 `!step_complete`。

| 条件 | 操作 |
|:-----|:------|
| Git 上有该 step 的新 commit | 任意角色直接调 `!step_complete stepN --output <sha>` |
| 当前 step 无人正在工作 | 跳过已完成中间 step |
| 当前 step 有人刚调了命令 | 等 2 秒序列化缓冲，避免冲突 |

### 规则 2：相邻审查机制

| 顺序 | 规则 |
|:----|:------|
| Step N → Step N+1 | Step N+1 的人先检查 Step N 是否有产出 |
| Step N 无产出 | 退回 Step N，附「等待 Step N 产出」 |
| Step N 有缺陷 | 退回 Step N 并附审查意见 |
| Step N 产出正确 | 直接推进到 Step N+1 开始工作 |

### 规则 3：退回规范

| 要点 | 说明 |
|:-----|:------|
| 退回必须附理由 | `!step_reject stepN --reason <明确原因>`，不能无声退回 |
| 退回次数限制 | 单 step 最多退回 2 轮，超时升级给 PM 协调 |
| 退回后自动通知 | 退回后系统自动在工作室通知上一步负责人 |

### 规则 4：保证始终有人工作

| 场景 | 行为 |
|:-----|:------|
| Step N 退回修正 | 修正完统一 push，然后推进到下一 step |
| Step N+1 审查退回 | 退回后 Step N 的人修正，Step N+1 的人等待 |
| 长时间无推进（>10min） | PM 主动介入协调 |
| 卡点无法解决 | 按超时升级链路：PM → 项目负责人 |

### 规则 5：不抢占活跃 Step

自动驾驶不等于所有人都能抢活。当前活跃 step 的负责人在工作时，其他人不要 `!step_complete` 覆盖推进，除非负责人已推 git 产出但不调命令。

### 规则 6：Step 交接不回声

Step 交接（`!step_complete` / `!step_reject`）产生的系统广播消息，**仅被 @mention 指定接手的 bot 回复确认**。其他角色看到此类消息 **零输出**，不发表态度、不回复「收到」「确认」「明白」等空表态消息。

| 场景 | 行为 |
|:-----|:------|
| Step N 完成 → 点名 Step N+1 | 只有 Step N+1 的角色可回复 ACK 确认接管 |
| Step N 被退回 → 通知 Step N 负责人 | 只有被退回的角色可回复「收到，修正中」 |
| 系统广播 `📋进度` 消息 | 全体静默（系统消息不需要各自确认） |
| 其他人的 ACK 确认回复 | 全体静默（不要重复确认别人的确认） |

## 方向 F：减少 Step 交接复读机回声 🟡 P1

| 项目 | 说明 |
|:-----|:------|
| **问题** | Step 交接时 `!step_complete` 结果广播到工作室，所有 bot 看到后多条回复「收到推进」「确认」「明白」等空表态消息，污染工作室信息流，但只抢话不动作 |
| **目标** | Step 交接产生的系统消息应明确标记为「无需回复」，bot 不应对此类消息做出表态回复 |
| **当前行为** | 各 bot 对 `!step_complete` 返回的系统文本逐个回复确认，一条完成公告引发 3~6 条空表态回复 |
| **目标行为** | 只有被 @mention 的目标 bot 回复 ACK 接管，其他角色零输出 |

### 技术方案方向

| 方案 | 说明 | 改动量 |
|:-----|:------|:-------|
| **F-a（推荐）** | `!step_complete` 的 `MSG_BROADCAST_TASK` 只发给目标角色，不广播给全工作室。完成通知改为 `_admin` 频道记录 + 工作室内**短提示**（不触发 bot 回复） | ~15 行 |
| **F-b** | step_complete 返回消息中附加 `[SYSTEM]` 前缀，bot agent prompt 增加规则「看到 [SYSTEM] 前缀的消息不回复」 | ~3 行 + prompt 更新 |
| **F-c** | 纯行为约束 — 仅在 WORK_PLAN 和 WORKSPACE_RULES 中增加规则，不修改代码 | 0 行 |

### 约束

- 被 @mention 的目标 bot 仍应回复 ACK 确认（不退化为完全不回复）
- 仅减少「无关 bot 的复读式回复」
- 不改变 `_admin` 频道的进度通知（那是给 PM 看的日志，不触发 bot 回复）

> 技术方案（具体实现方式）由架构师决定。

## 4. 验收标准

### 方向 A：放开角色校验

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| A-1 | member 角色在工作室内对 pending step 执行 `!step_complete step2 --output <sha>`，应成功推进 | 在实际管线中测试 |
| A-2 | 非工作区成员尝试推进，应被拒绝 | 用户权限校验 |
| A-3 | 对已完成的 step 执行 `!step_complete`，应报错「已完成的 step 不可重复推进」 | 状态校验 |
| A-4 | 同一 step 2 秒内被两人同时推进，第二次应被拒绝 | 序列化缓冲 |

### 方向 B：`!step_reject` 退回命令

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| B-1 | 在工作室内执行 `!step_reject step3 --reason "变量名阴影"`，step 指针回退到 step3 | `!pipeline_status` 显示退回到 step3 |
| B-2 | 退回时不带 `--reason` 参数，应报错「退回必须附理由」 | 参数校验 |
| B-3 | 被退回的 step 名称出现在 pipeline_status 的退回记录中 | 状态可视化 |
| B-4 | step 连续被退回 2 次后，第 3 次被拒绝并升级通知 | 轮次计数 |

### 方向 C：git 自动验证

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| C-1 | `!step_complete step2 --output 不存在的sha` → 报错「按远程不存在此 commit」 | 远程验证 |
| C-2 | `!step_complete step2 --output 存在的sha` → 正常推进 | 验证通过 |
| C-3 | `!step_complete step2`（不传 `--output`） → 跳过 git 检查，正常推进 | 兼容旧用法 |
| C-4 | `!step_complete step2 --output sha` 时远程不可达 → 降级为警告，仍推进 | 故障容错 |

### 方向 D：状态可视化

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| D-1 | `!pipeline_status` 显示每个 step 的状态：✅ ▶ 🔄 ⏳ | 查看输出 |
| D-2 | 被退回的 step 显示退回次数和最近一次理由 | 验证输出含退回信息 |
| D-3 | 管线在 auto/manual 模式下显示模式标记 | 模式切换后验证 |

### 方向 E：模式开关

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| E-1 | `!pipeline_start R55 --mode auto` → 管线自动模式运行 | 创建后查看状态 |
| E-2 | `!pipeline_start R55`（不传 mode）→ 默认 auto 模式 | 默认值 |
| E-3 | `!pipeline_mode manual` → 切换为手动模式，方向 A/B/C 行为不可用 | 验证切换 |

### 方向 F：减少 Step 交接回声

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| F-1 | `!step_complete` 广播的消息只有被 @mention 的角色回复 ACK，其他 bot 零输出 | 实测 Step 交接，观察工作室消息数 |
| F-2 | `!step_reject` 广播同样只触发被退回角色回复，其他 bot 不表态 | 实测退回场景 |
| F-3 | 系统 `📋进度` 系统消息不触发任何 bot 回复 | 观察 admin 频道通知输出 |
| F-4 | 被 @mention 的目标 bot 仍能正常回复 ACK 确认 | 确认接管流程不受影响 |
