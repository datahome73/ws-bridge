## 🔴 自动驾驶行为规则（全体必读 — 提醒一起执行）

> R55 起，管线正式进入**自动驾驶模式**。以下规则所有角色必须遵守。

### 规则 1：谁看到 git 产出，谁就是推进者

不需要等 step 负责人调 `!step_complete`。只要通过 git 确认远程有产出（`git ls-remote origin dev` 有新 commit），**任意角色都可以直接推进管线**。

```
✅ 我看到 Git 有产出 → 确认 commit 内容正确 → !step_complete stepN --output <sha>
❌ 我等 step 负责人调命令 → 等半天没动静 → 管线阻塞
```

### 规则 2：相邻 step 互为审查

| 你当前 Step | 你要检查 | 如果没产出 | 如果有缺陷 | 如果 OK |
|:-----------|:---------|:-----------|:-----------|:--------|
| Step N+1 | Step N 的产出 | `!step_reject stepN --reason "无新产出"` | `!step_reject stepN --reason "缺陷说明"` | 直接推进到 Step N+2 或开始工作 |
| Step N | Step N+1 的审查 | 修正后再推 | 回复审查意见 | 继续推进 |

### 规则 3：退回必附理由，最多 2 轮

```
!step_reject step3 --reason "变量名阴影 shadow 了内置函数 len，见审查报告§2"
                                                            ↑ 必须写理由
```

- 退回必须附具体原因，不能「声退回」
- 同一 step 最多退回 2 轮 → 第 3 次自动升级给 PM 协调
- 退回后修正完再 push → 调 `!step_complete` 推进

### 规则 4：始终有人在干活

| 场景 | 行为 |
|:-----|:------|
| Step N 退回修正 | 修正完推 git，推进到下一 step |
| 10 分钟无推进 | PM 主动介入核查 |
| 卡点无法解决 | PM → 项目负责人（TG 私聊） |

### 规则 5：不抢占活跃 Step

自动驾驶 ≠ 抢活。当前活跃 step 的负责人正在**工作中**（工作室有消息动态）时，其他人不要覆盖推进。只有在**负责人已完成产出但不调 `!step_complete`** 时，其他人再主动推进。

### 规则 6：Step 交接不回声

Step 交接产生的系统消息，**仅被 @mention 的目标 bot 回复 ACK**。其他角色 **零输出**——不发表态度、不回复「收到」「确认」「明白」等空表态。

| 场景 | 行为 |
|:-----|:------|
| `!step_complete` 完成公告 | 只有下一角色回复 ACK 接管，其他人静默 |
| `!step_reject` 退回通知 | 只有被退回的角色回复，其他人静默 |
| 系统 `📋` 进度消息 | 全体静默——系统消息不需要各自确认 |
| 其他人的 ACK 回复 | 全体静默——不要重复确认别人的确认 |

## 管线步骤

### Step 1：管线启动 🧐 PM

```bash
!pipeline_start R55 --from step2 --mode auto
```

- 默认 `--mode auto`（自动驾驶模式）
- 管线启动后所有 `!step_complete` 放开角色校验
- 各角色活跃频道自动切到 `ws:R55-dev`

### Step 2：技术方案 🏗️ 架构师 ✅ `ea7f65f`

**产出：** `docs/R55/R55-tech-plan.md`

**覆盖方向：**
- 方向 A：`!step_complete` 放开角色校验（PIPELINE_STEP_MAP 校验逻辑）
- 方向 B：新增 `!step_reject` 命令（状态机 + 退回计数 + 升级通知）
- 方向 C：git 自动验证（`urllib` 请求 `git ls-remote` 或 GitHub API）
- 方向 D：`!pipeline_status` 增强输出（退回记录 + 状态标记）
- 方向 E：`--mode auto/manual` 参数 + `!pipeline_mode` 切换命令

**关键决策（方案中需明确）：**
| 决策项 | 说明 |
|:-------|:------|
| `_channel_ack_state` 是否复用 | R53 已有 ACK 状态结构，是否直接扩用于退回/推进序列化？ |
| 退回后的 task 处理 | 原 task 关闭 + 新 task 创建？还是原 task 状态重置？ |
| git 验证方式 | `git ls-remote` HTTP 协议 vs GitHub API vs 本地 git 命令？ |
| 序列化策略 | 2 秒缓冲 vs 数据库锁 vs 消息队列？ |

**自动驾驶模式提醒（编码者）：**
> 你现在处于自动驾驶模式中。看到 git 产出就顺手推进，不用等人点名。
> 技术方案完成后推 git，不一定非得你调 `!step_complete`——PM 或其他人看到产出也会帮你推进。

**完成后：** 推 `docs/R55/R55-tech-plan.md` → 工作室讨论通过 → 推进管线

### Step 3：编码 💻 开发工程师 ✅ `ea9d0ce`

**产出：** `server/handler.py` + `server/config.py` 改动

| 方向 | 预估改动量 | 说明 |
|:----:|:---------:|:------|
| A 放开角色校验 | ~10 行 | 修改 `_cmd_step_complete` 的角色校验逻辑 |
| B `!step_reject` | ~40 行 | 新命令：参数解析、退回逻辑、通知、计数 |
| C git 验证 | ~20 行 | HEAD 请求远程 repo，验证 commit 存在性 |
| D 状态可视化 | ~15 行 | `_cmd_pipeline_status` 输出增强 |
| E 模式开关 | ~10 行 | `--mode` 参数解析 + `!pipeline_mode` 命令 |
| F 减少回声 | ~15 行 | `MSG_BROADCAST_TASK` 定向发送给目标 bot，完成通知 _admin 频道记录 |
| **合计** | **~110 行** | 全部新增/修改 |

**自动驾驶模式提醒（编码者）：**
> R55 本身就在测试自动驾驶模式。你的编码产出推 git 后，不需要等点名——PM 或审查看到 `git log` 有新 commit 就会直接推进管线。
>
> **干活顺手调命令：** 编码完成后，推 dev → 顺手 `!step_complete step3 --output <sha>`，不用等到被点名。如果看到技术方案已推 git 但还没调命令，你也可以直接推进 Step 2。

**完成后：** 推 dev → `!step_complete step3 --output <sha>`

### Step 4：代码审查 🔍 审查工程师

**审查重点：**

| 方向 | 审查要点 |
|:----|:---------|
| A | 角色校验逻辑：是否所有 pending step 都放开？已完成 step 是否被正确拦截？ |
| B | 退回状态机：退回后 step 指针正确回退？退回次数自增？第 3 次报错？ |
| C | git 验证超时：远程不可达时降级为警告？`--output` 不传时跳过？ |
| D | `!pipeline_status` 输出格式：退回记录清晰？状态标记准确？ |
| E | 模式切换：auto/manual 下行为是否正确隔离？ |
| F | 减少回声：`MSG_BROADCAST_TASK` 只发给目标 bot 还是全广播？完成通知是否触发 bot 回复？ |

**自动驾驶模式提醒（审查者）：**
> 看到代码推了 git 就拉下来审查，不需要等开发者调 `!step_complete`。
> 发现 Bug 直接用 `!step_reject step3 --reason "xx"` 退回，不必等角色正式指派。

**完成后：** `!step_complete step4 --output <sha>`

### Step 5：测试验证 🦐 测试工程师

**测试范围：** 方向 A~F 的验收标准共 20 项（A-1~A-4, B-1~B-4, C-1~C-4, D-1~D-3, E-1~E-3, F-1~F-4）

**自动驾驶模式关键验证：**

| 测试点 | 方法 |
|:-------|:------|
| 非负责人也能 `!step_complete` | member 角色在工作室推进 pending step |
| `!step_reject` 退回 | 退回后 pipeline_status 显示退回记录 |
| git 验证正常/降级 | 存疑 sha 报错，存在 sha 通过，不传 `--output` 跳过 |
| mode auto/manual 切换 | auto 下放开校验，manual 下回退到旧限制 |
| 减少回声 | Step 交接消息只触发目标 bot ACK，其他 bot 零输出 |

**自动驾驶模式提醒（测试者）：**
> 看到代码已部署（`!pipeline_status` 正常返回）就直接跑测试，不需要等审查调命令。
> 发现 bug 直接用 `!step_reject step4 --reason "验收不通过：V-X 失败"` 退回。

**完成后：** `!step_complete step5 --output <sha>` → 测试报告推 dev

### Step 6：合并部署 + 归档 🦸 管理员

| 操作 | 说明 |
|:-----|:------|
| 合并 dev → main | 代码上线 |
| 部署生产容器 | 重启服务使新命令生效 |
| 更新 TODO.md | 完成项标记 🟢 |
| 归档工作室 | 关闭 `ws:R55-dev` |

## 自动驾驶模式行为速查表

| 角色 | 传统模式（manual） | 自动驾驶模式（auto ✅ 默认） |
|:-----|:------------------|:----------------------------|
| 🧐 PM | 每一步人工调度，等各角色调命令 | 触发管线后看到 git 产出就推进，只在退回/阻塞时介入 |
| 🏗️ 架构师 | 出方案后等人推进 | 出方案后自己或别人看到产出就推进 |
| 💻 开发 | 编码后等调度去审查 | 编码后推 git → 看到审查也在推 step |
| 🔍 审查 | 被点名后才开始 | 看到 git 有编码产出就拉下来审，有问题直接退回 |
| 🦐 测试 | 等审查完成才部署 | dev 容器部署后直接跑验收，不等人 |
| 🦸 管理员 | 等测试完成才合并 | 监控整体状态，最终合并部署 |
