## 1. 背景与痛点

前几轮（R38-R40）引入了多项新功能：任务进度跟踪（`!task_create`）、点名报道机制（点名→频道切换）、Web 端 GitHub OAuth 认证等。这些功能在本轮之前已完成开发上线，但在实际使用中暴露出以下问题：

| 问题 | 严重度 | 来源 |
|:----|:-----:|:-----|
| 活跃工作室消息重复显示（修复后再现） | 🔴 P1 | 使用反馈 |
| 进度 Tab 看不到任何消息 | 🟡 P2 | 功能不足 |
| 点名后各成员未到位，消息收发错乱 | 🔴 P1 | 使用反馈 |
| 开发环境和生产环境 Web 登录方式未区分 | 🟡 P2 | 使用反馈 |

**本轮不引入任何新功能方向，只做四件事：**
1. 修复活跃工作室消息重复的回归 Bug
2. 让进度 Tab 有消息可看
3. 拆分点名流程，让关键路径成员能准确到位
4. 区分开发/生产环境的 Web 登录方式

## 方向 A — Web 认证按环境区分

### A-1. 当前体验痛点

R40 为 Web 端引入了 GitHub OAuth 认证，与现有绑定码并行运行。登录页同时展示两种入口：

- 🔑 绑定码输入框 + 等待人工审批
- 🐙 GitHub 登录按钮

**当前的问题：**
- 生产环境也暴露了绑定码入口，违背了引入 OAuth 的安全初衷
- 绑定码需要人工审批，生产环境不应依赖此通道
- 但开发/测试环境中，绑定码非常适合 Bot 快速验证功能，不应移除

### A-2. 期望体验

| 环境 | 登录方式 | 理由 |
|:----|:---------|:-----|
| **开发环境**（开发 VPS） | 🔑 绑定码 + 🐙 GitHub OAuth 并存 | 绑定码方便 Bot 快速测试验证，OAuth 留给真人使用 |
| **生产环境**（线上服务） | 🐙 仅 GitHub OAuth | 生产环境的安全门槛更高，不应暴露绑定码入口 |

### A-3. 验收标准

| # | 验收标准 | 优先级 |
|:-:|:---------|:------:|
| A-1 | 开发环境登录页同时展示绑定码和 GitHub 登录两个入口 | 🔴 P1 |
| A-2 | 生产环境登录页仅展示 GitHub 登录入口，无绑定码区域 | 🔴 P1 |
| A-3 | 生产环境调用 `/api/bind` 和 `/api/check` 接口返回 404 或明确的禁用提示 | 🔴 P1 |
| A-4 | 环境切换通过配置即可完成，无需修改代码 | 🟡 P2 |

## 方向 B — 活跃工作室消息重复显示（回归修复）

### B-1. 当前体验痛点

在活跃工作室中查看消息时，**每条消息在屏幕上显示两遍**。同一消息上下排列，内容完全一致，影响阅读体验。

此问题在 R39 已有过一次修复尝试（前端去重哈希表），但当前仍会复现。说明修复方案未彻底覆盖所有竞争场景。

### B-2. 期望体验

- 每条消息在任何 Tab 下**只显示一次**
- 无论是实时 WebSocket 推送到达还是轮询刷新，都不出现重复
- 切换 Tab 再切回、页面刷新后，消息依然唯一

### B-3. 验收标准

| # | 验收标准 | 优先级 |
|:-:|:---------|:------:|
| B-1 | 活跃工作室中，每条发送的消息在消息列表只出现一次 | 🔴 P1 |
| B-2 | 大厅消息同样不出现重复 | 🟡 P2 |
| B-3 | 页面刷新后消息仍然唯一 | 🔴 P1 |
| B-4 | 长时间使用（持续 10 分钟+），消息不因去重缓存溢出而重新重复 | 🔴 P1 |

## 方向 C — 进度 Tab 消息展示

### C-1. 当前体验痛点

Web 聊天室 Tab 栏中有一个「📊 进度」Tab，但打开后永远显示**「暂无任务进度数据」**。通过 `!task_create` 创建任务后，进度 Tab 依然空白。

问题在于：进度 Tab 的 `renderProgressTab()` 函数从 `_admin` 频道拉取聊天消息，过滤以 `📊` 开头的消息展示。但任务状态的消息通过 `MSG_TASK_NOTIFY` 协议通道传递，**没有进入聊天消息的写入路径**。所以前端永远等不到数据。

**当前无人能通过进度 Tab 了解管线进展。**

### C-2. 期望体验

- 打开「📊 进度」Tab 能看到当前轮次的任务列表和状态
- 任务状态更新后（如从「正在处理」变为「已完成」），进度 Tab 能刷新反映最新状态
- 显示的信息包含：任务名称、当前状态、负责人角色、更新时间

### C-3. 验收标准

| # | 验收标准 | 优先级 |
|:-:|:---------|:------:|
| C-1 | 使用 `!task_create` 和 `!task_update` 操作后，进度 Tab 能看到对应的任务和状态 | 🔴 P1 |
| C-2 | 任务状态变化后，进度 Tab 自动刷新（不依赖手动刷新页面） | 🟡 P2 |
| C-3 | 进度 Tab 显示的内容包含「任务名 / 状态 / 负责人 / 更新时间」 | 🟡 P2 |
| C-4 | 多个任务时按时间倒序排列（最新在上） | 🟡 P2 |

### C-4. 后续改进方向（不在本轮实现）

> 项目负责人指出：进度 Tab 的长期目标不应绑定 `!task_create` 命令，而应作为**管线进度信息的摘要**，信息来源于**工作计划文档**。Bot 和人通过进度 Tab 都能了解当前轮次走到哪一步了、当前谁在干活。
>
> 此方向记入 `R41-todo-future.md`，本轮不做实现。

## 方向 D — 创建工作室与点名流程拆解

### D-1. 当前体验痛点

目前创建一个新轮次的工作室后，以下操作被捆绑在一起：

1. 创建工作室房间
2. 给全员发点名通知
3. 全员回复「已切」切换活跃频道

实际使用中暴露的问题：

| 问题 | 现象 |
|:----|:------|
| **点名不完备** | 点名通知发出后，部分成员未切换频道，后续消息落入大厅 |
| **频道切换错位** | 消息在大厅和工作室两边乱飞，沟通断裂 |
| **等待全员到位浪费** | 点名要求全员回应，但实际只需要下一个 Step 的成员在线即可 |
| **进度信息断层** | 进度 Tab 不展示当前走到哪个 Step、轮到谁工作了 |

### D-2. 期望体验

创建工作室到开始工作的流程，应按逻辑拆分为可独立操作的步骤：

```
Step ① ─ 创建工作室       → 建好房间，成员具备进入条件
Step ② ─ 指定关键成员     → 明确下一个 Step 的关键角色是谁
Step ③ ─ 派发上下文       → 关键成员收到完整的工作背景信息
Step ④ ─ 关键成员回复到位  → 确认收到上下文，开始工作
```

**点名不追求全员到位**，只要求下一个 Step 的关键成员准确到位，拿到上下文，能开始干活。他干完了，下一个 Step 的人再接力。

点名通知应该明确告诉当前 Step：**「@角色名，请确认到位，你的任务是：…」**

### D-3. 验收标准

| # | 验收标准 | 优先级 |
|:-:|:---------|:------:|
| D-1 | 创建工作室后，不自动切换全体成员活跃频道 | 🔴 P1 |
| D-2 | 点名消息能指定一个目标角色（而非全员广播「到齐」） | 🔴 P1 |
| D-3 | 被点名的角色回复确认后，其活跃频道自动切换到工作室 | 🔴 P1 |
| D-4 | 非被点名角色不收到点名消息（或只收到通知性提示，不要求回复） | 🟡 P2 |
| D-5 | 点名确认后，被点名的成员能收到完整的上下文信息（本 Step 的背景、任务描述） | 🔴 P1 |
| D-6 | 频道切换正确后，该成员的消息落入工作室、不再落入大厅 | 🔴 P1 |

### D-4. 后续改进方向（不在本轮实现）

> 项目负责人指出：进度 Tab 的长期方向是成为**管线进度的摘要展示**，信息来源于**工作计划文档**。当前 `!task_create` 命令的进度信息也应能被进度 Tab 消费。
> 
> 此方向记入 `R41-todo-future.md`，本轮不做实现。
