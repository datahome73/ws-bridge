## 1. 问题背景

### 1.1 R56 交付后管线运行实际状态

R56 实现了通信层修复三方向（A: `_send_to_agent` 定向+回退广播双保险 / B: 7 段通信链路诊断 / C: 过渡期 SOP），部署上线后管线推进可靠性有了提升：

| 场景 | R55 状态 | R56 状态 |
|:-----|:---------|:---------|
| 目标 bot **在线**时 Step 交接通知 | 全广播（有回声） | ✅ 定向送达，零回声 |
| 目标 bot **离线**时 Step 交接通知 | ❌ 静默丢失，通知消失 | 🟡 回退到工作室频道广播 + 写入日志（不丢但无人处理） |
| 工作室内各 bot 可见性 | ❌ 多数收不到消息 | ✅ 活跃频道切换正常，在线 bot 可见 |

### 1.2 残余卡点（R56~R55 实操暴露）

| # | 卡点 | 现象 | 根因 |
|:-:|:-----|:------|:-----|
| 🔴 P0 | **主角离线时管线卡死** | Step 交接时目标角色（主角）不在线/不响应，通知写入回退广播但无人执行 Step。每轮需要项目负责人 TG 转发激活或重启 gateway | `!step_complete` 点名后只定向通知主角一人。主角不在时没有备用接替机制，系统不会尝试找别人 |
| 🔴 P0 | **系统消息展示 agent ID 不直观（F-19）** | `!pipeline_start` / `!pipeline_status` 的成员列表使用原始 agent ID（如 `01KTNJ2QQ...`），Web 端观察者不直观且暴露隐私 | 系统消息渲染时直接将 agent_id 填入，未经名称/角色翻译 |

### 1.3 核心理念：全员点名发现是最可靠的

> **项目负责人的经验总结：**
> - 服务端重连 hook 不可靠（bot 重连了也不知道该干什么）
> - **全员点名发现才是最可靠的** — 点名阶段如果主角不出现，立刻知道有问题
> - 每个 Step 应该有备用人员：点名时发现主角不在 → 让备用上场干这个 Step 的活
> - 角色分工固定不能乱 — 各人有主任务，上下文不混、质量高、skill 有沉淀
> - 但点名发现主角不出现时，**不换人管线肯定卡死**
> - 所以正确的方向是：**协调机制 + 后备换人机制**

```
R55: 管线代码正确但通信断裂 → 每 Step 需人工激活
  ↓
R56: 定向+回退双保险 + 过渡期SOP → 在线bot全自动，离线bot通知不丢
  ↓
R57: Step 交接点名发现 + 备用自动换人 + 角色名显示
      → 主角不在时自动换备用上场，不给管线卡死的机会
```

## 2. 方向说明

### 方向 A（核心）：Step 交接点名发现 + 备用自动换人 🔴 P0

| 项目 | 说明 |
|:-----|:------|
| **问题** | `!step_complete` 触发 Step 交接时，只定向通知下一 Step 的主角。主角如果离线或不响应，管线永久卡死。没有后备接替机制 |
| **目标** | 每个 Step 在 PIPELINE_STEP_MAP 中配置主角色和备用角色。`!step_complete` 触发后：先点名发现主角是否在 → 主角在则正常交接 → 主角不在则自动切换给备用，附完整上下文 |
| **设计哲学** | 全员点名发现是最可靠的检测手段。主角在不在，点名就知道。点名阶段不换人，管线必卡。**不依赖 WebSocket 重连 hook，依赖主动确认机制** |

#### 设计详述

##### B1：PIPELINE_STEP_MAP 扩展 — 每个 Step 配主角 + 备用

当前映射（仅主角）：
```python
PIPELINE_STEP_MAP = {
    "step2": {"role": "arch"},
    "step3": {"role": "dev"},
    "step4": {"role": "review"},
    "step5": {"role": "qa"},
    "step6": {"role": "admin"},
}
```

扩展后：
```python
PIPELINE_STEP_MAP = {
    "step2": {"primary": "arch", "backup": "dev"},
    "step3": {"primary": "dev", "backup": "arch"},
    "step4": {"primary": "review", "backup": "dev"},
    "step5": {"primary": "qa", "backup": "review"},
    "step6": {"primary": "admin", "backup": "arch"},
}
```

> 具体的主/备角色映射由架构师和项目负责人协商确定。上述仅为示例。

##### B2：Step 交接点名流程

```
!step_complete stepN --output <sha>
  ↓
① 服务端确认 Step N 完成，推进 Step 指针到 Step N+1
  ↓
② 查 PIPELINE_STEP_MAP 获取 Step N+1 的主角 + 备用
  ↓
③ 点名主角（`@主角 Step N+1 轮到你了，请回复确认`）
  ↓
④ 等待主角确认（超时阈值：30 秒）
  ↓
┌─ 主角回复 ✅ → 正常交接给主角（同当前行为）────┐
│   ↓                                              │
│  Step N+1 正常推进                                │
└───────────────────────────────────────────────────┘
  ↓（主角 30 秒无响应）
⑤ 系统自动切换给备用人员
   ┌─ 广播到工作室：`⚠️ 主角未响应，Step N+1 由备用接替`
   └─ 定向通知备用：`@备用 Step N+1 轮到你了！`
      上下文：需求文档 URL + 工作平面 URL + Step N 产出 SHA
  ↓
⑥ 备用回复确认 → Step N+1 正常推进
```

##### B3：备用接替后的记录

- `_admin` 频道日志记录：`📋 R57 | Step N+1 | 主角未响应 → 备用接替`
- `!pipeline_status` 展示当前角色时标注入选角色名：`▶ step3 — dev(备用接替)`

##### B4：主角回归后的处理

如果 Step N+1 进行中途主角重新上线：
- 不抢占当前执行者（备用已在工作中）
- 主角自动进入「待命」状态，等待下一 Step 的轮次

##### B5：备用也离线的情况

如果主角和备用同时都不在线（极端情况），回退到 R56 过渡期流程：
1. 系统广播到工作室：`🔴 Step N+1 主角和备用均不在线，等待项目负责人协调`
2. PM 按方向 B 流程 TG 汇报项目负责人
3. 项目负责人协调重启或转发

> 这种极端情况很少见（两个角色同时离线），不需要为它过度设计。

> 技术方案（具体实现方式：PIPELINE_STEP_MAP 的结构设计、点名超时阈值、角色查找逻辑）由架构师决定。

#### 验收标准

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| A-1 | `!step_complete` 触发后，系统点名下一 Step 的主角，30 秒内主角回复 → 正常交接给主角 | 实测 Step 交接，主角在线 |
| A-2 | 主角 30 秒未回复 → 系统自动切换给备用，备用收到带上下文的通知 | 断开主角连接再调 `!step_complete` |
| A-3 | 备用接替后，工作室收到换人公告（含角色名） | 检查工作室广播 |
| A-4 | `_admin` 频道记录换人日志：管线名、Step、主角、备用 | 检查 admin 频道 |
| A-5 | `!pipeline_status` 标注当前执行角色（含「备用接替」标记） | 执行 `!pipeline_status` |
| A-6 | 主角在 Step 进行中重新上线 → 不抢占，自动进入待命状态 | 模拟重连确认无干扰 |
| A-7 | 主角和备用同时离线 → 系统广播 + 过渡期流程触发 | 双 bot 断线后验证 |
| A-8 | Option B 用户无备用配置时，行为不变（仅通知主角，不尝试换人） | 配置 PIPELINE_STEP_MAP 无 backup 字段验证 |

### 方向 B（辅助）：PM 主动监控 + TG 即时汇报 🟡 P1

方向 A 覆盖了「主角不在线 → 自动换备用」的场景。但极端情况（主角和备用同时离线、服务器异常等）仍需人工兜底。方向 B 是 PM 的兜底工作流程。

| 项目 | 说明 |
|:-----|:------|
| **问题** | 极端卡死场景（方向 A 无法覆盖的 edge case）需要有人及时发现并通知项目负责人 |
| **目标** | PM 建立主动监控纪律，发现卡死后即时 TG 私聊项目负责人，附标准汇报格式 |

#### 监控纪律

| 时机 | PM 动作 | 超时阈值 |
|:-----|:--------|:--------:|
| Step 交接后 | 观察工作室是否出现目标 bot 的确认回复 | 1 分钟内无回复 → 进入超时计时 |
| 工作室静默 | 从最后一条工作消息算起 | **10 分钟**无任何 bot 输出 → 判断为卡死 |
| 连续 2 次 Step 需项目负责人协调 | 暂停管线，TG 汇报「建议先排查通信链路」 | 立即汇报 |

#### TG 即时汇报格式

```
🔴 R57 管线卡死报告
├─ 工作室：ws:R57-dev
├─ 当前 Step：Step 3（编码），已静默 12 分钟
├─ 目标角色：dev-bot
├─ 备用状态：已尝试换人但仍无响应
└─ 需要项目负责人：重启 dev-bot gateway 或转发激活消息
```

PM 汇报后不等待回复，继续监控下一 Step。项目负责人看到后自行处理。

#### 验收标准

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| B-1 | 工作室安静 10 分钟 → PM 主动 TG 私聊项目负责人汇报卡死 | 检查 TG 历史 |
| B-2 | TG 汇报格式包含：管线名、当前 Step、静默时长、推测原因、需要项目负责人做什么 | 模板检查 |
| B-3 | PM 汇报后不等待回复，继续监控 | 观察 PM 行为 |

> 方向 B 为零代码改动，纯工作流程改进。

## 5. 设计哲学

### 点名发现 > 事后重连

全员点名是最可靠的成员状态检测手段——名字点到、有回复就是在线，没回复就是不在。不依赖 WebSocket 底层状态判断，不依赖重连时的 hook 触发。**点名是主动确认，重连是被动推测。**

### 不换人管线必卡

每个 Step 有主角有备用。点名时主角不出现，自动换备用上场。线不够粗就并线。

### 角色固定，但角色不等同于一个人

各角色有固定分工（arch/dev/review/qa/admin），上下文明确、质量高、skill 有沉淀。但每个角色可以由不同的人担任——主角不在时备用顶上，管线不卡。**角色和人不绑定**。

### 角色名是信息，agent ID 是隐私

系统消息对 Web 端观察者可读，agent ID 只在调试日志中保留。
