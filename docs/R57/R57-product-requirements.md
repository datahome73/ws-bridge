## 1. 问题背景

### 1.1 R56 交付后管线运行实际状态

R56 实现了通信层修复三方向（A: `_send_to_agent` 定向+回退广播双保险 / B: 7 段通信链路诊断 / C: 过渡期 SOP），部署上线后管线推进可靠性有了提升：

| 场景 | R55 状态 | R56 状态 |
|:-----|:---------|:---------|
| 目标 bot **在线**时 Step 交接通知 | 全广播（有回声） | ✅ 定向送达，零回声 |
| 目标 bot **离线**时 Step 交接通知 | ❌ 静默丢失，通知消失 | 🟡 回退到工作室频道广播 + 写入日志（不丢但无人处理） |
| 工作室内各 bot 可见性 | ❌ 多数收不到消息 | ✅ 活跃频道切换正常，在线 bot 可见 |

### 1.2 残余卡点（R56~R55 实操暴露）

| # | 卡点 | 现象 | 根因 |
|:-:|:-----|:------|:-----|
| 🔴 P0 | **主角离线时管线卡死** | Step 交接时目标角色（主角）不在线/不响应，通知写入回退广播但无人执行 Step。每轮需要项目负责人 TG 转发激活或重启 gateway | `!step_complete` 点名后只定向通知主角一人。主角不在时没有备用接替机制，系统不会尝试找别人 |
| 🔴 P0 | **系统消息展示 agent ID 不直观（F-19）** | `!pipeline_start` / `!pipeline_status` 的成员列表使用原始 agent ID（如 `01KTNJ2QQ...`），Web 端观察者不直观且暴露隐私 | 系统消息渲染时直接将 agent_id 填入，未经名称/角色翻译 |

### 1.3 核心理念

> **项目负责人的经验总结：**
> - 服务端重连 hook 不可靠（bot 重连了也不知道该干什么）
> - **全员点名发现是最可靠的** — 点名阶段如果主角不出现，立刻知道有问题
> - 但点名前可以先查在线状态：Web 端能看到各 bot 是否在线。**离线的先解决连接状态，再来点名，就可靠了**
> - 每个 Step 应该有备用人员：点名时发现主角不在 → 让备用上场干这个 Step 的活
> - 角色分工固定不能乱 — 各人有主任务，上下文不混、质量高、skill 有沉淀
> - 但点名发现主角不出现时，**不换人管线肯定卡死**
> - 所以正确的方向是：**在线状态预检 + 后备换人机制**

```
R55: 管线代码正确但通信断裂 → 每 Step 需人工激活
  ↓
R56: 定向+回退双保险 + 过渡期SOP → 在线bot全自动，离线bot通知不丢
  ↓
R57: 在线状态预检 + Step 交接点名发现 + 备用自动换人
      → 离线 bot 不需要等 30s 超时，直接换人
      → 不给管线卡死的机会
```

## 2. 方向说明

### 方向 A（核心）：在线状态预检 + Step 交接点名发现 + 备用自动换人 🔴 P0

| 项目 | 说明 |
|:-----|:------|
| **问题** | `!step_complete` 触发 Step 交接时，只定向通知下一 Step 的主角。主角如果离线或不响应，管线永久卡死。没有后备接替机制 |
| **目标** | `!step_complete` 触发后：① 查 `_connections` 检测主角是否在线 → ② 主角离线则直接切备用（0 秒等待）→ ③ 主角在线则 30 秒点名确认 → ④ 主角不响应则切备用。备用接替后收到完整上下文 |
| **设计哲学** | 点名前先做状态预检。离线不花时间等——直接换人。在线才点名——30 秒足够确认。**查 `_connections` 比等超时快 30 秒** |

#### 设计详述

##### A1：PIPELINE_STEP_MAP 扩展 — 每个 Step 配主角 + 备用

由项目协调人（PM）在 WORK_PLAN 中配置主备角色映射。唯一约束：**不能自己审自己**——即编写技术方案的人不能同时是同一 Step 的编码者。

示例映射：
```python
PIPELINE_STEP_MAP = {
    "step2": {"primary": "arch", "backup": "dev"},       # arch 写方案 → dev 可替补
    "step3": {"primary": "dev", "backup": "arch"},        # dev 编码 → arch 可替补
    "step4": {"primary": "review", "backup": "dev"},      # review 审查 → dev 可替补
    "step5": {"primary": "qa", "backup": "review"},       # qa 测试 → review 可替补
    "step6": {"primary": "admin", "backup": "arch"},      # admin 部署 → arch 可替补
}
```

> 注意：只要不是同一个人写方案又编码，就是合理的。如 arch 写方案 → dev 编码 ✅，arch 写方案 → 其他角色编码 ✅。
> 具体映射由项目协调人（PM）在 WORK_PLAN 或配置中确定，不在代码中硬编码。

##### A2：Step 交接点名流程（含在线状态预检）

```
!step_complete stepN --output <sha>
  ↓
① 服务端确认 Step N 完成，推进 Step 指针到 Step N+1
  ↓
② 查 PIPELINE_STEP_MAP 获取 Step N+1 的 主角 + 备用
  ↓
③ 查 _connections 检测主角是否有活跃 WS 连接
  ↓
┌─ 主角 OFFLINE（无活跃连接）──── 直接换人 ────────────┐
│    ↓                                                    │
│  ⚠️ 系统广播：`主角离线，Step N+1 由备用接替`            │
│  定向通知备用 → 带完整上下文                              │
└──────────────────────────────────────────────────────────┘
  ↓（主角 ONLINE）
④ 点名主角（`@主角 Step N+1 轮到你了，请回复确认`）
  ↓  等待 30 秒
┌─ 主角回复 ✅ ── 正常交接 ──────────────────────────┐
│    ↓                                                │
│  Step N+1 正常推进                                   │
└──────────────────────────────────────────────────────┘
  ↓（主角 30 秒无响应）
⑤ 系统自动切换给备用人员
   ├─ 广播：`⚠️ 主角未响应，Step N+1 由备用接替`
   └─ 通知备用：完整上下文（需求文档 URL + WORK_PLAN + Step N 产出 SHA）
  ↓
⑥ 备用回复确认 → Step N+1 推进
```

##### A3：备用接替后的记录

- `_admin` 频道日志记录：`📋 R57 | Step N+1 | 主角未响应 → 备用接替`
- `!pipeline_status` 展示当前角色时标注入选角色：`▶ step3 — dev(备用接替)`

##### A4：主角回归后的处理

如果 Step N+1 进行中途主角重新上线：
- 不抢占当前执行者（备用已在工作中）
- 主角自动进入「待命」状态，等待下一 Step 的轮次

##### A5：备用也离线的情况（极端情况）

如果主角和备用同时离线（查 `_connections` 双方都无连接）：
1. 系统广播到工作室：`🔴 Step N+1 主角和备用均不在线，等待协调`
2. PM 按方向 B 流程 TG 汇报项目负责人
3. 项目负责人安排重启 gateway 或调整人员

> 这种极端情况很少见（两个角色同时离线），不需要为它过度设计。

> 技术方案（具体实现方式）由架构师决定。

#### 验收标准

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| A-1 | `!step_complete` 触发后，主角有活跃 WS 连接 → 点名主角，30 秒内回复 → 正常交接 | 实测 Step 交接，主角在线 |
| A-2 | 主角 `_connections` 为空（离线）→ **不点名**，直接切备用，0 秒等待 | 断开主角连接，调用 `!step_complete` |
| A-3 | 主角在线但不回复 → 30 秒超时后自动切备用，备用收到带上下文的通知 | 主角在线但不回，等待 30 秒 |
| A-4 | 备用接替后，工作室收到换人公告（含角色名） | 检查工作室广播 |
| A-5 | `_admin` 频道记录换人日志：管线名、Step、主角、备用 | 检查 admin 频道 |
| A-6 | `!pipeline_status` 标注当前执行角色（含「备用接替」标记） | 执行 `!pipeline_status` |
| A-7 | 主角在 Step 进行中重新上线 → 不抢占，自动待命 | 模拟重连确认无干扰 |
| A-8 | 主角和备用同时离线 → 系统广播 + PM 按过渡期流程处理 | 双 bot 断线后验证 |
| A-9 | 无 backup 配置的 Step 行为不变（仅通知主角，不尝试换人） | 删除 PIPELINE_STEP_MAP backup 字段后验证 |

### 方向 B（辅助）：PM 主动监控 + TG 即时汇报 🟡 P1

方向 A 覆盖了「主角不在线 → 自动换备用」的主流场景。但边缘情况（主角和备用同时离线、服务器异常等）仍需人工兜底。

| 项目 | 说明 |
|:-----|:------|
| **问题** | 极端卡死场景（方向 A 无法覆盖的 edge case）需要有人及时发现并通知项目负责人 |
| **目标** | PM 建立主动监控纪律：10 分钟工作室静默 → 即时 TG 私聊项目负责人，附标准汇报格式 |

#### 监控纪律

| 时机 | PM 动作 | 超时阈值 |
|:-----|:--------|:--------:|
| Step 交接后 | 观察工作室是否出现目标 bot 的确认回复 | 1 分钟内无回复 → 进入超时计时 |
| 工作室静默 | 从最后一条工作消息算起 | **10 分钟**无任何 bot 输出 → 判断为卡死 |
| 连续 2 次 Step 需项目负责人协调 | 暂停管线，汇报「建议先排查通信链路」 | 立即汇报 |

#### TG 即时汇报格式

```
🔴 R57 管线卡死报告
├─ 工作室：ws:R57-dev
├─ 当前 Step：Step 3（编码），已静默 12 分钟
├─ 目标角色：dev-bot
├─ 备用状态：已尝试换人但仍无响应
└─ 需要项目负责人：重启 dev-bot gateway 或调整人员
```

PM 汇报后不等待回复，继续监控。

#### 验收标准

| # | 验收标准 |
|:-:|:---------|
| B-1 | 工作室安静 10 分钟 → PM 主动 TG 私聊项目负责人汇报卡死 |
| B-2 | TG 汇报格式包含：管线名、当前 Step、静默时长、推测原因、需要项目负责人做什么 |
| B-3 | PM 汇报后不等待回复，继续监控 |

> 方向 B 为零代码改动，纯工作流程改进。

## 5. 设计哲学

### 点名发现 > 事后重连

全员点名是最可靠的成员状态检测手段。**点名前先查 `_connections`**，离线不花时间等——直接换人。先解决连接状态，再来点名。

### 不换人管线必卡

每个 Step 有主角有备用。主角不在时自动换备用上场。约束：不能自己审自己（编写方案≠编码同一人）。

### 角色固定，但角色不等同于一个人

各角色有固定分工（arch/dev/review/qa/admin），上下文明确、质量高、skill 有沉淀。但每个角色可以由不同的人担任——主角不在时备用顶上。

### 主备角色由 PM 配置

主备映射不在代码中硬编码。由项目协调人（PM）在 WORK_PLAN 或配置中指定。每轮可灵活调整。

### 角色名是信息，agent ID 是隐私

系统消息对 Web 端观察者可读，agent ID 只在调试日志中保留。
