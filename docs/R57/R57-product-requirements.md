# R57 产品需求 — 自动驾驶管线最后一段：离线自愈 + PM-TG 协调机制 + 角色名显示

> **版本：** v0.2（基于项目负责人反馈修正）
> **状态：** 📝 草稿（待项目负责人审核）
> **产品经理：** 🧐 PM
> **日期：** 2026-06-30
> **本轮改动范围：** 仅第①类（服务器代码 `server/handler.py`） + PM 工作流程
> **上轮反馈：** v0.1 → `_admin` 频道超时告警无效（项目负责人不看 `_admin`）；角色名显示为必修；PM 协调力度需增强

---

## 1. 问题背景

### 1.1 R56 交付后管线运行实际状态

R56 实现了通信层修复三方向（A: `_send_to_agent` 定向+回退广播双保险 / B: 7 段通信链路诊断 / C: 过渡期 SOP），部署上线后管线推进可靠性有了提升：

| 场景 | R55 状态 | R56 状态 |
|:-----|:---------|:---------|
| 目标 bot **在线**时 Step 交接通知 | 全广播（有回声） | ✅ 定向送达，零回声 |
| 目标 bot **离线**时 Step 交接通知 | ❌ 静默丢失，通知消失 | 🟡 回退到工作室频道广播 + 写入日志（不丢但无人处理） |
| 工作室内各 bot 可见性 | ❌ 多数收不到消息 | ✅ 活跃频道切换正常，在线 bot 可见 |

### 1.2 残余卡点（R56 实操暴露，未完全解决）

| # | 卡点 | 现象 | 根因 |
|:-:|:-----|:------|:-----|
| 🔴 P0 | **离线 bot 需要人工激活** | 即使通知已写入工作室频道日志，bot 重连后不会自动检查是否有挂起的 Step 任务。每轮仍需要项目负责人通过 TG 私聊转发 code block 激活目标 bot | 回退广播只解决了「不丢失」，未解决「离线后自动恢复」。bot 重连 WS 后没有触发 pending task 检查机制 |
| 🔴 P0 | **PM 协调力度不够，卡死发现不及时** | 工作室静默 10 分钟无消息 → 实际上已卡死。admin-bot 可以通过重新点名让不响应的 bot 回到工作室，或重启离线的 bot 的 gateway 来恢复连接。但 PM 没有 admin-bot 的权限，无法执行这些操作。当前 R56 过渡期 SOP 的「5 分钟升级→准备 code block→等项目负责人发现」链路太长 | PM 不具备系统级操作能力（重新点名、重启 gateway），卡死的发现依赖于项目负责人主动关注工作室。需要从「项目负责人主动发现」变为「PM 主动发现 → 即时 TG 汇报 → 项目负责人决策执行」|
| 🔴 P0 | **系统消息展示 agent ID 不直观（F-19）** | `!pipeline_start` 的广播在 `_admin` 频道列出成员时使用原始 agent ID（如 `01KTNJ2QQ...`），对 Web 端观察者不直观且暴露隐私 | `_cmd_pipeline_start` 构建系统消息时直接使用 `auth.get_users()` 返回的 agent_id 字段，未经翻译 |

### 1.3 R56 设计哲学的延续

```
R55: 管线代码正确但通信断裂 → 每 Step 需人工激活
  ↓
R56: 定向+回退双保险 + 过渡期SOP → 在线bot全自动，离线bot通知不丢
  ↓
R57: 离线bot重连自愈 + PM主动监控TG汇报 + 角色名显示
      → 减少人工触达次数，不追求零人工
```

**关键认识：** 在全自动通信层到位之前，最务实的路径不是「加一个 nobody 看的超时告警」，而是让 PM 主动监控、即时 TG 汇报给项目负责人、项目负责人决策执行（重新点名、重启 gateway 等）。**PM 发现 → TG 汇报 → 项目负责人行动**，这是当前最可靠的协调链路。

## 2. 方向说明

### 方向 A：离线 bot 自动唤醒 — WS 重连时拉取挂起任务 🔴 P0

| 项目 | 说明 |
|:-----|:------|
| **问题** | bot 离线重连 WS 后，不会自动检查是否有挂起的 pipeline Step 任务。即使 R56 的回退广播已写入工作室频道日志，bot 不会主动读取，仍需要人工转发激活 |
| **目标** | bot 重连 WS 后自动检测活跃 pipeline 中是否有分配给自己的 pending Step，如有则自动进入工作模式并确认接管 |
| **设计哲学** | 重连 = 自动归队。不是等人工唤醒，也不是等下一轮点名。bot 像「进办公室自动查待办」一样 |

#### 技术方案方向

| 方案 | 说明 | 改动量 |
|:-----|:------|:------:|
| **A-a（推荐）** | `handler.py` 的 WS 连接建立入口（`handler()` / `ws_handler()`）在完成身份认证后，检查活跃 pipeline 的当前 Step 是否分配给此 agent。如果有 → 通过 `_send_to_agent` 发送任务接替通知。这样 bot 重连后 1 秒内就知道自己该干什么 | ~15 行 |
| **A-b** | 在 pipeline 层增加 `_check_pending_tasks(agent_id)` 周期性扫描函数。bot 重连后调用此函数，返回所有未完成的 Step 列表 | ~30 行 |

> 技术方案（具体实现方式）由架构师决定。

#### 验收标准

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| A-1 | bot 断开 WS 连接后又重新连接 → 自动收到分配给的 pipeline Step 通知，不需要人工转发 | 模拟 bot 断线重连 + `!pipeline_status` 确认 Step 指针 |
| A-2 | 同时有多个活跃 pipeline 时，只检查当前活跃 pipeline（不跨 pipeline 干扰） | 开 2 个 pipeline，bot 只收到自己正在跑的管线任务 |
| A-3 | 没有被分配 pending step 的 bot 重连后零干扰（不产生噪声） | 重连后查 `_admin` 频道日志，只有自己相关的通知 |
| A-4 | `!pipeline_status` 显示的 Step 指针与重连后 bot 收到的任务一致 | 对比系统通知内容与 `!pipeline_status` 输出 |
| A-5 | 重连后通知中带需求文档 URL + 当前 Step 上下文（与原始点名通知一致） | 检查通知格式是否包含 URL 和上下文 |

### 方向 B：PM 主动监控 + TG 即时汇报机制 🟡 P1

| 项目 | 说明 |
|:-----|:------|
| **问题** | 当前卡死发现机制是被动的——项目负责人自己关注工作室聊天发现 10 分钟静默才意识到卡死。PM 虽然按 R56 SOP 监控，但发现卡死后缺乏「即时通知项目负责人」的纪律，导致项目负责人需要自己盯进度 |
| **目标** | PM 建立主动监控纪律：工作室静默达到阈值（10 分钟无消息）→ 立即 TG 私聊项目负责人汇报卡点，附上下文。不等项目负责人发现，不等下一个时间点 |
| **设计哲学** | PM 是项目负责人的眼线，不是旁观者。PM 发现卡死后不自己纠结「怎么解决」——权限不够（无重新点名/重启 gateway 权限），直接 TG 汇报，让项目负责人决策执行 |

#### 为什么不是服务器端超时检测？

上一版（v0.1）提出在 `_admin` 频道加服务端 watchdog 超时告警。项目负责人纠正：

> 发送到 `_admin` 频道其实没有作用——之前遇到了很多次超时，都是项目负责人通过 TG 唤醒的。
> 项目负责人关注工作室的聊天，通常 10 分钟聊天室没有消息发出，其实就是卡死了。

**结论：** 服务端 watchdog → `_admin` 频道这条路径对项目负责人不可见。真正有效的发现机制是工作室的消息流——工作室安静了就是卡死了。而最有效的响应链路是 **PM 发现 → TG 即时汇报 → 项目负责人行动**。

#### 方向 B 包含的内容

##### B1：PM 监控纪律（正式写入，强制执行）

| 时机 | PM 动作 | 超时阈值 |
|:-----|:--------|:--------:|
| Step 交接后 | 观察工作室是否出现目标 bot 的确认回复 | 1 分钟内无回复 → 进入超时计时 |
| 工作室静默 | 从最后一条工作消息算起，静默计数器开始 | **10 分钟**无任何 bot 输出 → 判断为卡死 |
| 连续 2 次 Step 需项目负责人激活 | 暂停管线，向项目负责人报告「通信链路可能断裂」 | 立即报告 |

##### B2：TG 即时汇报格式

发现卡死后，PM 立即在 TG DM 汇报，格式如下：

```
🔴 R57 管线卡死报告
├─ 工作室：ws:R57-dev
├─ 当前 Step：Step 3（编码），已静默 12 分钟
├─ 目标角色：dev-bot
├─ 推测原因：离线 / 不响应
└─ 需要项目负责人：重启 dev-bot gateway 或转发激活消息
```

##### B3：PM 汇报后不等待

PM 汇报后不需要在 TG 等待项目负责人回复。项目负责人看到后自行处理（重启 gateway / 转发消息 / 重新点名）。PM 继续监控下一 Step。

#### 验收标准

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| B-1 | Step 交接后 1 分钟目标 bot 无回复 → PM 进入超时计时，不静默等待 | 实际执行观察 |
| B-2 | 工作室安静 10 分钟 → PM 主动 TG 私聊项目负责人汇报卡死 | 检查 TG 历史 |
| B-3 | TG 汇报格式包含：管线名、当前 Step、静默时长、推测原因、需要项目负责人做什么 | 模板检查 |
| B-4 | PM 汇报后不等待回复，继续监控下一 Step | 观察 PM 行为 |

> 方向 B 为零代码改动，纯工作流程改进。

## 5. 设计哲学

### 重连 = 自动归队

离线不是终点。bot 重连 WS 后，系统应当像「早上进办公室」一样——自动检查自己的待办事项并发起工作。不等人工喊开始。

### PM 是眼线，不是决策者

PM 发现卡死后，不自己想办法解决（没有 admin-bot 的权限）。即时 TG 汇报给项目负责人，由项目负责人决策执行（重新点名、重启 gateway）。**PM 发现 → TG 汇报 → 项目负责人行动**是最可靠的协调链路。

### 工作室安静 = 卡死，10 分钟是阈值

项目的负责人通过工作室消息流感知进度——有消息就是活着，无声就是卡死。PM 监控以工作室消息为信号，10 分钟静默就是 TG 汇报的触发点。

### 先减少人工触达次数，不求零人工

方向 A 减少离线 bot 需要激活的次数（重连自愈）。方向 C 提升信息可读性。方向 B 让残余的人工触达更高效。R57 不是「零人工」版本——是「人工触达次数减少 + 剩下的人工触达更高效」的版本。
