## 1. 问题背景

### 1.1 R56 交付后管线运行实际状态

R56 实现了通信层修复三方向（A: `_send_to_agent` 定向+回退广播双保险 / B: 7 段通信链路诊断 / C: 过渡期 SOP），部署上线后管线推进可靠性有了提升：

| 场景 | R55 状态 | R56 状态 |
|:-----|:---------|:---------|
| 目标 bot **在线**时 Step 交接通知 | 全广播（有回声） | ✅ 定向送达，零回声 |
| 目标 bot **离线**时 Step 交接通知 | ❌ 静默丢失，通知消失 | 🟡 回退到工作室频道广播 + 写入日志（不丢但无人处理） |
| 工作室内各 bot 可见性 | ❌ 多数收不到消息 | ✅ 活跃频道切换正常，在线 bot 可见 |

### 1.2 残余痛点（R56 实操暴露，未完全解决）

| # | 痛点 | 现象 | 根因 |
|:-:|:-----|:------|:-----|
| 🔴 P0 | **离线 bot 需要人工激活** | 即使通知已写入工作室频道日志，bot 重连后不会自动检查是否有挂起的 Step 任务。每轮仍需要项目负责人通过 TG 私聊转发 code block 激活目标 bot | 回退广播只解决了「不丢失」，未解决「离线后自动恢复」。bot 重连 WS 后没有触发 pending task 检查机制 |
| 🟡 P1 | **管线超时后无声阻塞（F-17）** | 某 Step 负责人忘了调 `!step_complete` 或离线了，管线状态永久停留在该 Step，无人知道卡住了。没有系统级超时检测机制向 `_admin` 频道汇报 | 管线状态机是事件驱动的（被动等待 `!step_complete`），缺乏定期巡检的 watchdog |
| 🟢 P2 | **系统消息展示 agent ID 不直观（F-19）** | `!pipeline_start` 的广播在 `_admin` 频道列出成员时使用原始 agent ID（如 `01KTNJ2QQ...`），对 Web 端观察者不直观且暴露隐私 | `_cmd_pipeline_start` 构建系统消息时直接使用 `auth.get_users()` 返回的 agent_id 字段，未经翻译 |

### 1.3 R56 设计哲学的延续

```
R55: 管线代码正确但通信断裂 → 每 Step 需人工激活
  ↓
R56: 定向+回退双保险 + 过渡期SOP → 在线bot全自动，离线bot通知不丢
  ↓
R57: 离线bot重连自愈 + 超时检测 → 真正无人工干预的自动驾驶管线
```

R56 的规划已明确提出「R57+ 逐步逼近全自动」。R57 的目标是 **消除最后的人工触达点**，使管线在大多数情况下不需要项目负责人转发激活。

## 2. 方向说明

### 方向 A：离线 bot 自动唤醒 — WS 重连时拉取挂起任务 🔴 P0

| 项目 | 说明 |
|:-----|:------|
| **问题** | bot 离线重连 WS 后，不会自动检查是否有挂起的 pipeline Step 任务。即使 R56 的回退广播已写入工作室频道日志，bot 不会主动读取，仍需要人工转发激活 |
| **目标** | bot 重连 WS 后自动检测活跃 pipeline 中是否有分配给自己的 pending Step，如有则自动进入工作模式并确认接管 |
| **设计哲学** | 重连 = 自动检查任务。不是等人工唤醒，也不是等下一轮点名 |

#### 技术方案方向

| 方案 | 说明 | 改动量 |
|:-----|:------|:------:|
| **A-a（推荐）** | `handler.py` 的 WS 连接建立入口（`handler()` / `ws_handler()`）在完成身份认证后，检查活跃 pipeline 的当前 Step 是否分配给此 agent。如果有 → 通过 `_send_to_agent` 发送任务接替通知。这样 bot 重连后 1 秒内就知道自己该干什么 | ~15 行 |
| **A-b** | 在 pipeline 层增加 `_check_pending_tasks(agent_id)` 周期性扫描函数。bot 重连后调用此函数，返回所有未完成的 Step 列表。A-b 比 A-a 更干净但需要新函数 + 定时逻辑 | ~30 行 |
| **A-c** | bot 侧主动拉取（在 bot 的 Hermes agent 中加入 reconnection hook，主动调 `!pipeline_status` 检查自己的任务）。不需要服务端改动，但依赖 bot 侧实现 | 0 行服务端 |

> 技术方案（具体实现方式）由架构师决定。

#### 验收标准

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| A-1 | bot 断开 WS 连接后又重新连接 → 自动收到分配给的 pipeline Step 通知，不需要人工转发 | 模拟 bot 断线重连 + `!pipeline_status` 确认 Step 指针 |
| A-2 | 同时有多个活跃 pipeline 时，只检查当前活跃 pipeline（不跨 pipeline 干扰） | 开 2 个 pipeline，bot 只收到自己正在跑的管线任务 |
| A-3 | 没有被分配 pending step 的 bot 重连后零干扰（不产生噪声） | 重连后查 `_admin` 频道日志，只有自己相关的通知 |
| A-4 | `!pipeline_status` 显示的 Step 指针与重连后 bot 收到的任务一致 | 对比系统通知内容与 `!pipeline_status` 输出 |
| A-5 | 重连后通知中带需求文档 URL + 当前 Step 上下文（与原始点名通知一致） | 检查通知格式是否包含 URL 和上下文 |

### 方向 B：管线 Step 超时自动检测 🟡 P1

| 项目 | 说明 |
|:-----|:------|
| **问题** | 当前管线状态机是事件驱动的——只有 `!step_complete` 才能推动 Step 前进。如果负责该 Step 的角色离线或忘记调命令，管线永久阻塞，无任何人知道 |
| **目标** | 服务端增加定时巡检机制：活跃 pipeline 的当前 Step 启动后超过阈值未完成 → 自动向 `_admin` 频道汇报超时，触发升级链路 |
| **设计哲学** | 超时检测是服务端纯粹的规则逻辑，零 token 消耗。参考已有 Hot Standby 看门狗模式（`references/r43-hot-standby-watchdog-pattern.md`） |

#### 技术方案方向

| 方案 | 说明 | 改动量 |
|:-----|:------|:------:|
| **B-a（推荐）** | 在 `_cmd_step_complete` 执行时记录 Step 开始时间戳（存入 `_PIPELINE_STATE`），新增 `_pipeline_watchdog()` 定时检查函数（如每 5 分钟扫描一次），超时阈值可配置（默认：技术方案/编码 30 分钟，审查/测试 15 分钟） | ~25 行 |
| **B-b** | 复现 R43 看门狗模式（`_HOT_STANDBY_WATCHDOG_INTERVAL`），在已有 watchdog 循环中加入 pipeline Step 超时检查。利用现有基础设施 | ~15 行 |
| **B-c** | 仅纯行为约束——在 WORK_PLAN 中声明超时规则，PM 按 C-2（R56 过渡期流程）手动监控。零代码改动 | 0 行 |

> 技术方案（具体实现方式和超时阈值默认值）由架构师决定。

#### 验收标准

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| B-1 | 活跃 pipeline 当前 Step 超时 → `_admin` 频道收到超时告警消息 | 启动管线后不调 `!step_complete`，等待超时阈值 |
| B-2 | 超时消息格式含：管线名称、当前 Step、已用时、目标角色 | 验证管道消息字段 |
| B-3 | 超时检测执行后不自动推进 Step（仅告警，不自动跳过） | 确认 Step 指针未移动 |
| B-4 | 有多个活跃 pipeline 时，每个 pipeline 独立超时检测 | 启动 2 个管线，验证各自超时报文 |
| B-5 | 超时前正常完成 Step 的 pipeline 不触发告警 | 正常跑完一轮管线，检查零误报 |
| B-6 | 超时阈值可通过配置项调整（不要在代码中硬编码 30 分钟） | 修改配置后重启，验证超时时间变化 |

## 5. 设计哲学

### 重连 = 自动归队

离线不是终点。bot 重连 WS 后，系统应当像「早上进办公室」一样——自动检查自己的待办事项并发起工作。不等人工喊开始。

### 超时检测是安全网，不是推进器

方向 B 的超时检测**只告警，不自动跳过 Step**。跳过需要人工决策（`!step_reject` 退回或 `!step_handoff` 手动推进）。安全网可以提醒人，但不能替人做决定。

### 先跑通完整管线，再优化剩余噪声

R57 的通过标准不是「零人工干预」，而是「管线可以完整跑一轮，90%+ Step 不需要项目负责人转发」。偶发的离线场景由超时告警兜底，PM 按 R56 过渡期流程处理残余的 10%。
