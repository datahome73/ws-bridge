# 自动驾驶管线协作设计原则

> **状态：** 🔶 待开发
> **提出人：** 大宏
> **日期：** 2026-06-29
> **基于：** R53 管线实践

---

## 核心理念

### 共享上下文，主动推进

管线中的所有角色共享以下信息：
- **WORK_PLAN** — 各 step 负责人、产出、验收标准
- **工作区 Step 状态** — 当前在哪个 step，谁在干活
- **Git 仓库** — 代码/文档立即可拉

**原则：任何人都可以推进 step。**

不需要等上一 step 的人主动调 `!step_complete`。只要通过 git 确认了远程产出（commit），就可以直接推进管线到下一步。

### 相邻审查机制（Built-in Adjacent Review）

```
Step N ──→ Step N+1
    ↑          │
    └──────────┘
    (退回审查)
```

- **Step N 和 Step N+1 互为审查**
- 如果 Step N 没有产出（git 上没有新 commit），Step N+1 的人暂不干活，退回 Step N
- 如果 Step N 有产出但有缺陷，Step N+1 的人可以退回并附上审查意见
- **核心效果：** 始终有人在干活，所有人都推动 step 前进

### 流程示意图

```
# 传统模式（串行阻塞）
Step1完成 → 点名 → 等确认 → Step2开始 → Step2完成 → 点名 → 等确认 → ...

# 自动驾驶模式（主动推进+相邻审查）
         ┌──────────────────────────────┐
         │ Git 上看到 Step N 产出 ✅      │
         │ → 直接推进到 Step N+1 (任意人) │
         │ → 开始 Step N+1 的工作         │
         └──────────────────────────────┘

         ┌──────────────────────────────┐
         │ Git 上 Step N 无产出/有缺陷   │
         │ → 退回 Step N (附审查意见)     │
         │ → Step N 人修正后重推          │
         └──────────────────────────────┘
```

## 具体规则

### 1. Step 推进条件

| 条件 | 操作 |
|:----|:-----|
| Git 上有该 step 的新 commit | 任意角色可直接调 `!step_complete stepN --output <sha>` |
| 技术方案与代码逻辑一致 | 自动验证（检查 commit 消息是否匹配方案） |
| 当前 step 无人正在工作 | 跳过已完成的中间 step（如 Step 1/2 已完成，直接推进到 Step 3） |

### 2. 退回规则

| 条件 | 操作 |
|:----|:-----|
| Step N 无产出（git 上无新 commit） | Step N+1 退回，附「等待 Step N 产出」 |
| Step N 有缺陷 | Step N+1 退回并附审查意见 |
| Step N 代码不匹配方案 | Step N+1 退回并附差异分析 |

### 3. 角色特化

自动驾驶模式下，每个人的角色能力不变，但工作方式变化：

| 原角色 | 自动驾驶模式行为 |
|:-------|:----------------|
| 🧐 PM | 触发管线 + 遇到退回/阻塞时协调，不再负责推进每一步 |
| 🏗️ 架构师 | 出方案后自己或让别人推进 Step 2→3，不用等手动点名 |
| 💻 开发 | 编码完成后推代码，看到代码落地后自动推进审查 |
| 🔍 审查 | 直接拉代码审查，不需要等专人点名。看到问题退回 Step 3 附意见 |
| 🦐 测试 | dev 部署后跑验收，不需要等 Step 4 完成。发现问题退回上一步 |
| 🦸 管理员 | 监控整体状态，最终合并部署 |

## 收益

| # | 收益 | 说明 |
|:-:|:-----|:------|
| 1 | ⏱️ **消除等待时间** | 不用等点名、等确认、等人调命令 |
| 2 | 🔄 **始终有人在工作** | 退回了就修正，修正完继续推 |
| 3 | 🔍 **内建审查机制** | 相邻 step 互为审查，不需要独立审查阶段 |
| 4 | 🚀 **并行效率** | 可多个 step 同时在不同轨道上推进 |
| 5 | 📉 **降低协调成本** | 不需要 PM 每一步的人工调度 |

## 实现需求

### 技术改动

| # | 改动 | 说明 |
|:-:|:-----|:------|
| 1 | `!step_complete` 放开角色校验 | 允许任意成员推进任一处于 pending 的 step（当前限制为 step 负责人） |
| 2 | 基于 git 的自动验证 | `!step_complete` 检查远程 git 是否存在指定 commit |
| 3 | Step 退回命令 | 新增 `!step_reject stepN --reason <原因>` |
| 4 | 相邻 step 状态关联 | `!pipeline_status` 显示哪些 step 可被推进、哪些被退回 |
| 5 | 自动驾驶模式开关 | 每个管线可设 `auto` / `manual` 模式 |

### 非技术要点

| # | 要点 | 说明 |
|:-:|:-----|:------|
| 1 | 信任模型 | 自动驾驶依赖成员自觉——谁看到 git 有产出就直接推进 |
| 2 | 退回规范 | 退回必须附审查意见，不能「无声退回」 |
| 3 | 超时兜底 | 如果某个 step 长时间没推进，PM 主动介入协调 |
| 4 | 混合模式 | 允许在关键 step（如 merge 部署）走传统手动模式 |

## 与 R53 ACK 的关系

R53（ACK 确认制）提供了自动驾驶管线的基础设施：
- ACK 保证了「消息已送达、任务已确认」
- 但 ACK 本身不解决「谁来推进 step」的问题
- **自动驾驶模式**是在 ACK 之上的协作层设计

两者互补：
```
R53 ACK ──→ 确保通信可靠
               ↓
自动驾驶 ──→ 确保推进高效
               ↓
效率最大化
```

## 附录：对比当前行为

| 维度 | 当前（R53 前） | 自动驾驶模式 |
|:-----|:--------------|:------------|
| Step 推进 | 只有负责人能调 `!step_complete` | 任何人都能推进 |
| 审查 | 独立 Step 4 审查阶段 | 相邻 step 内建审查 |
| 等待 | 等点名、等确认、等人完成 | 看到 git 产出就推进 |
| 退回 | 无机制 | `!step_reject` 退回到上一步 |
| 并行度 | 单线程串行 | 可并行（不同轨道） |
