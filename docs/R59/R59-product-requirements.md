# R59 产品需求 — arch/dev 自动触发修复 + PM 自动兜底机制

> **版本：** v0.1（初稿）
> **状态：** 📝 草稿（待项目负责人审核）
> **产品经理：** 🧐 PM（需求分析师）
> **日期：** 2026-06-30
> **本轮改动范围：** 全栈（`server/handler.py` + bot 网关配置调优 + `PIPELINE_STEP_MAP` 弹性化）
> **R58 遗留项：** 🔴 arch/dev 持续需额外 TG 触发（确认 R58 from_name 改造不生效于此二者）

## 1. 问题背景

### 1.1 R58 交付后管线运行实际状态

R58 实现了 **from_name 系统→PM 身份 + 工作室广播 @mention**（方向 A）+ **ACK 软检查**（方向 B）+ **通知状态跟踪**（方向 C），代码级 38/38 验收全绿，已合并部署 `ws-bridge:r58`。

但 R58 实操中暴露了一个**致命残留问题**：

| 角色 | bot | R58 方向 A 后的触发情况 | 状态 |
|:----|:---:|:-----------------------|:----:|
| 🧐 PM | 需求分析师 | ✅ 始终正常响应 | — |
| 🦸 admin | 项目管理 | ✅ 方向 A 后能触发（需 @mention + 上下文） | 🟢 |
| 🦐 qa | 测试工程师 | ✅ 方向 A 后能触发 | 🟢 |
| 🔍 review | 审查工程师 | ✅ 方向 A 后能触发（备用 qa 测试正常） | 🟢 |
| 🏗️ arch | 架构师 | ❌ **方向 A 无效，仍需 TG 转发** | 🔴 |
| 💻 dev | 开发工程师 | ❌ **方向 A 无效，仍需 TG 转发** | 🔴 |

> **关键数据：** R58 管线全流程实操验证中发现，架构师(arch) 和 开发工程师(dev) 在 R58 代码部署后仍然不响应工作室内的 PM 身份 @mention 通知。每轮 Step 2（技术方案）和 Step 3（编码）都需要项目负责人通过 TG 私聊手动转发指令才能触发 bot 工作。

### 1.2 管线自动化断点矩阵

```
管线全自动的理想状态（6 步全部自动累加）:

  Step 1  管线启动（服务端自动）               ✅ 自 R48
  Step 2  技术方案（arch）                     ❌ 需项目负责人 TG 转发 ← R59 要修的
  Step 3  编码实现（dev）                      ❌ 需项目负责人 TG 转发 ← R59 要修的
  Step 4  代码审查（review）                   ✅ R58 后自动
  Step 5  测试验证（qa）                       ✅ R58 后自动
  Step 6  合并部署（admin）                    ✅ R58 后自动

管线自动化完成度：4/6 = 67%  ✅
手动介入率：2/6 = 33%（集中在 arch/dev）
```

> **影响：** 管线虽然能跑，但每轮至少需要项目负责人介入 2 次（Step 2 和 Step 3），无法实现「启动后不用管」的最终目标。

### 1.3 根因定位方向

R58 的 `from_name` 改造（系统→PM 身份 + 工作室广播 @mention）在 **review/qa/admin** 三个 bot 上验证有效，但在 **arch/dev** 上无效。这排除了「ws-bridge 发消息格式」这个变量——同一个格式、同一个广播路径，不同 bot 的响应不同。

#### 推测根因树

```
arch/dev 不响应 PM @mention 的根本原因
├── A. bot 网关配置层面（最可能 🔴）
│   ├── A1. 架构师/开发工程师的 Hermes 网关配置了「仅响应项目负责人 ID」的 @mention 过滤
│   ├── A2. 架构师/开发工程师的 LLM prompt/system prompt 不同，对「系统通知伪装成人」的处理方式不同
│   ├── A3. 架构师/开发工程师的 bot 配置了不同的消息预处理 pipeline（丢弃了某些格式的消息）
│   └── A4. 架构师/开发工程师的活跃频道配置特殊，工作室广播未送达（但 R57 的 _connections 预检认为在线）
│
├── B. ws-bridge 消息层面（次要 🟡）
│   ├── B1. 当前 PM @mention 模板对 arch/dev 的触发词不够明确（缺少特定关键词）
│   ├── B2. 当前 PM @mention 模板缺少 arch/dev 所需的特定格式（如 code block 标记）
│   └── B3. 当前消息长度/结构触发 arch/dev 的某种防 spam 逻辑
│
└── C. 消息路径层面（可能性低 🟢）
    ├── C1. 工作室广播的 from_name 虽为 PM，但 bot 内部识别出了「这不是真实人类」
    └── C2. 架构师/开发工程师在线但活跃频道指向了其他地方，没收到工作室消息
```

#### 对比实验：什么有效 vs 什么无效

| 触发方式 | from_name | 格式 | arch/dev 结果 | review/qa/admin 结果 |
|:---------|:---------:|:-----|:-------------:|:--------------------:|
| R57 系统通知 | 系统 | 简略文本 | ❌ | ❌ |
| R58 PM @mention (工作室广播) | PM | `@bot名 🚨 Step「X」到你了！\n\n📄 需求...\n📋 WORK_PLAN...\n🔗 上一步产出...` | ❌ | ✅ |
| 项目负责人 TG 私聊转发 | 项目负责人(真人) | 自定义 code block | ✅ | ✅ |

> **核心观察：** 有效和无效的差距不在于消息内容的丰富度（R58 已经补全了上下文），而在于 `from_name` 是否被 bot 识别为「真人」。
> 项目负责人的 TG 转发中 `from_name` 是 [项目负责人 TG 用户 ID]（项目负责人的 TG 用户 ID），bot 网关视之为真人消息。
> R58 的 PM 身份中 `from_name` 是 `"PM"` 字符串——bot 网关虽然不再视作「系统」，但可能视作「另一个 bot 发的消息」。

### 1.4 核心理念

> **R58 确认：** `from_name="PM"` 对 review/qa/admin 有效，但对 arch/dev 无效。
> **R59 假设：** arch/dev 的 bot 网关配置了更严格的触发条件——只响应特定 `from_name`（如人类 ID）或特定消息格式。
> **修复方向：** 不猜——先探测，再适配。

```
R55: 代码正确但通信断裂                 → 每 Step 需人工激活
  ↓
R56: 在线定向+离线回退广播              → 在线全自动，离线通知不丢
  ↓
R57: 点名发现+主备换人                  → 主角离线自动换备用
  ↓
R58: from_name 系统→PM + 工作室广播     → review/qa/admin 自动触发
  ↓
R59: 修复 arch/dev 自动触发残件         → 6 步管线全自动的最后 2 步
```

## 2. 方向设计

### 方向 A（核心调查 🔴 P0）：探测 arch/dev 触发临界条件

> **Why not fix directly?** 根因最可能是 bot 网关配置（非 ws-bridge 代码），直接改代码可能做无用功。先通过实验确定「什么东西能触发 arch/dev」，再针对性适配。

| 项目 | 说明 |
|:-----|:------|
| **问题** | 不知道 arch/dev 的 `from_name` / 消息格式 / 触发词临界条件——无法定位修复方向 |
| **目标** | 通过系统化实验，确认「什么消息格式 + 什么 from_name + 什么内容」能触发 arch/dev 工作模式 |
| **方法** | 通过 WS 直连发送不同变体的消息，观察哪个变体被 arch/dev 响应 |

#### 探测实验方案

| 1c | `"项目负责人"`（与项目负责人 TG 名字一致） | ? | ⬜ |
| 1d | `"[项目负责人 TG 用户 ID]"`（项目负责人 TG 用户 ID） | ? | ⬜ |
| 1e | `"项目负责人"` + 额外字段（如 `agent_id` 或 `user_id`） | ? | ⬜ |

#### 项目环境信息（重要）

本次实验得益于项目基础设施的完善——同一 VPS 上运行着两套独立容器：

| 环境 | 用途 | 数据 | 谁可操作 | 项目负责人是否观察 |
|:----|:-----|:----|:--------|:-----------------:|
| **dev** | 开发测试 🧪 | 独立于 main | 各 bot 均可（测试工程师可随时部署） | ❌ 不看 |
| **main** | 生产 🏭 | 独立于 dev | 仅 admin 有凭证 | ✅ 只观察 Web 端 |

- 两套环境数据完全分离，dev 随便折腾不会影响 main
- 各 bot 客户端已配置同时连接 dev 和 main 服务器
- **方向 A 实验全部在 dev 环境执行，零生产风险**
- 方向 B 编码后先部署到 dev 验证，确认通过后再合入 main

> **影响：** 实验可以大胆做、频繁做、在任何时候做。不需要挑生产空闲窗口，不需要担心污染生产消息流。

### 方向 B（编码适配 🟡 P1）：根据实验结果修改通知格式

> **依赖：** 方向 A 的实验结果。如果没有找到有效触发条件，方向 B 降级为理论方案。

| 项目 | 说明 |
|:-----|:------|
| **目标** | 根据方向 A 的实验结论，修改 `_cmd_step_complete` 和 `_cmd_pipeline_start` 中的通知格式，使 arch/dev 能自动触发 |
| **改动范围** | `server/handler.py` — PM @mention 通知格式 + 发送路径 |

#### B1：通知格式按角色差异化

如果实验发现 arch/dev 需要特定的消息格式才能触发（如带 code block、或特定 `from_name`）：

```python
# 当前（R58 统一 PM 格式）
mention_msg = f"@{primary_name} 🚨 Step「{next_step}」到你了！\n\n📄 需求：{req_url}\n..."

# 改造后（按角色差异化）
if next_role in ["arch", "dev"]:
    # arch/dev 需要特殊格式（如带 code block 或特定触发词）
    mention_msg = f"@{primary_name} 🚨 Step「{next_step}」到你了！\n\n📄 需求：{req_url}\n📋 WORK_PLAN：{plan_url}\n🔗 上一步产出：{output_ref}\n\n```\n请确认收到后开始工作。完成后调用 !step_complete {next_step} --output <sha>\n```"
else:
    # 其他角色保持 R58 格式
    mention_msg = f"@{primary_name} 🚨 Step「{next_step}」到你了！\n\n📄 需求：{req_url}\n..."
```

#### B2：通知发送路径按角色差异化

如果实验发现 arch/dev 需要 `_send_to_agent` 直连（而非工作室广播）：

```python
if next_role in ["arch", "dev"]:
    # arch/dev 走 _send_to_agent 直连（from_name 用实验确定的特殊值）
    await _send_to_agent(agent_id, special_mention_msg, ws_id=sender_ch, 
                         from_name=experiment_found_name)
else:
    # 其他角色走工作室广播 PM @mention（R58 路径）
    _persist_broadcast(sender_ch, pm_name, mention_msg)
    for conn in conns:
        await _send(conn, broadcast_payload)
```

#### B3：PM 自动兜底机制（如果方向 A 找不到有效触发条件）

如果方向 A 的实验结论是「arch/dev 的 bot 网关完全过滤了非项目负责人 ID 的消息」，ws-bridge 代码层面无法强制触发，则引入 PM 自动兜底：

- `_cmd_step_complete` 在通知 `arch/dev` 后，启动一个后台监视任务（5 分钟超时）
- 如果 5 分钟内没有检测到该 bot 的 ACK 或任务推进信号
- PM 自动在工作室内输出一条 `@架构师/开发工程师：请确认收到，任务详情见上方 @mention`
- 同时 PM 通过 TG 发送通知给项目负责人：「Step X 的 @mention 已送达但 bot 未响应，请确认是否需要 TG 转发」

> **设计哲学：** 如果 ws-bridge 不能直接触发 arch/dev，至少让 PM 自动做人类 PM 会做的事——催一下、报告一下。不让项目负责人在无人通知的情况下等待。

#### 验收标准

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| B-1 | arch 在 Step 2 交接时自动收到可触发的消息（不依赖 TG 转发） | 实操 Step 2 交接 |
| B-2 | dev 在 Step 3 交接时自动收到可触发的消息（不依赖 TG 转发） | 实操 Step 3 交接 |
| B-3 | 其他角色（review/qa/admin）的通知格式不变（不退化） | 对比 R58 格式 |
| B-4 | 如果兜底机制存在（B3）：bot 不响应时 PM 自动催 + TG 通知项目负责人 | 检查输出 |
| B-5 | 方向 A 的实验报告锁定触发条件后被 B 代码正确实现 | 审查代码 vs 报告 |

### 方向 C（角色弹性 🟢 P2）：PIPELINE_STEP_MAP 角色解耦

> 如果方向 A/B 都无法解决 arch/dev 的自动触发问题，引入「哪些角色能做哪些 Step」的弹性配置，绕开不能自动触发的 bot。

| 项目 | 说明 |
|:-----|:------|
| **问题** | arch 和 dev 是固定角色绑定到 Step 2/3，如果它们永远不能自动触发，管线在这两步永远需要人工介入 |
| **目标** | 让 PM 可以在 RUN_WORK_PLAN 或 PIPELINE_STEP_MAP 中配置「谁是架构师」「谁是开发者」，不再硬绑定 arch=Step 2、dev=Step 3 |

#### C1：角色映射动态配置

当前 `PIPELINE_STEP_MAP` 格式：

```python
PIPELINE_STEP_MAP = {
    "step1": {"name": "管线启动", "role": "auto", ...},
    "step2": {"name": "技术方案", "role": "arch", "primary": "arch-bot", "backup": "dev-bot"},
    "step3": {"name": "编码", "role": "dev", "primary": "dev-bot", "backup": "arch-bot"},
    ...
}
```

改造后：角色名和 Step 编号解耦，允许 PM 在 WORK_PLAN 中指定「谁做什么」：

```python
# PM 可在 RUN_WORK_PLAN 中覆盖角色映射
R59_ROLE_MAP = {
    "arch_role": "review",     # 本轮 arch 的工作交给 review 角色
    "dev_role": "qa",          # 本轮 dev 的工作交给 qa 角色
    # review 和 qa 的工作交换给... (后续处理)
}
```

或者更灵活的：允许一个 bot 担任多个角色：

```python
# 如果 review 能自动触发，且能兼任 arch 的写作方案工作
R59_ROLE_MAP = {
    "arch": "review",    # Step 2 由 review 角色执行
    "dev": "qa",         # Step 3 由 qa 角色执行
    "review": "admin",   # Step 4 由 admin 角色执行
    "qa": "admin",       # Step 5 由 admin 角色执行
}
```

> **约束：** 不能自己审自己、谁能写方案不等于谁能编码（且不能是同一个人），这些约束需要维护。

#### C2：一次性的「绕过 arch/dev」管线

如果方向 A/B 都失败且方向 C 来不及完整实现，引入一次性的最短路径：

```
R59 管线（最短路径版）:
  Step 1  管线启动（服务端自动）
  Step 2  技术方案 → 由 测试工程师(qa) 执行（qa 已经验证可自动触发）
  Step 3  编码实现 → 由 审查工程师(review) 执行（review 已经验证可自动触发）
  Step 4  代码审查 → 由 项目管理(admin) 执行（admin 已经验证可自动触发）
  Step 5  测试验证 → 由 测试工程师(qa) 执行（qa 自己测自己方案/代码？有约束问题）
  Step 6  合并部署 → 由 项目管理(admin) 执行（仅 admin 有凭证）
```

> ⚠️ **约束冲突：** 自己不能审自己、写方案的人 ≠ 编码的人、编码的人 ≠ 审查的人。如果绕过 arch/dev 导致约束冲突，需要在 WORK_PLAN 中明确豁免。

#### 验收标准

| # | 验收标准 | 验证方法 |
|:-:|:---------|:---------|
| C-1 | PM 可在 WORK_PLAN 中配置 Step 的角色分配（不限于 arch/dev） | 检查配置生效 |
| C-2 | 一步最短路径管线能全自动跑通（不依赖项目负责人 TG 转发） | 实操完整管线 |
| C-3 | 约束冲突（自审/自测）在 WORK_PLAN 中显式标记，不会被静默绕过 | 检查输出 |

## 3. 方向间依赖关系

```
方向 A（探测实验）← 无条件开始，独立执行，产出一份实验报告
     ↓
方向 B（编码适配）← 依赖方向 A 的实验结论（或至少实验 1-3 开始后有初步结果）
     ↓
方向 C（角色弹性）← 独立于 A/B，但最有用的情况是 A/B 都失败后的兜底路径
```

**推荐执行顺序：**

1. **方向 A（先做）** — 通过 WS 直连发出 10+ 种消息变体，5 分钟内可获得初步结果
2. **方向 B（并行开始理论设计）** — 不等方向 A 全完成，有初步结论就开始编码
3. **方向 C（备选）** — 如果 A+B 两周后仍无进展，启动方向 C

## 6. 设计哲学

### 先探测，再编码

不清楚 arch/dev 的触发临界条件就写代码是盲改。先通过实验锁定条件，再针对性编码。实验代价低（5 分钟发几条 WS 消息），但能避免走错方向。

### 按角色差异化是合理的

review/qa/admin 用 PM @mention 格式触发，arch/dev 可能需要特殊格式。这不是代码设计不对称——而是现实不对称。不管黑猫白猫，抓到老鼠就是好猫。

### 如果 ws-bridge 做不到，PM 兜底

在「ws-bridge 代码不能直接触发 arch/dev」的极端情况下（bot 网关完全过滤非项目负责人 ID），不放弃——PM 自动生成转发 code block + TG 通知项目负责人。**进步是相对的：** 从「项目负责人等半小时才发现卡了」变成「PM 自动告诉项目负责人来转发一下」，效率提升。

### 角色不绑定 Step，才是真弹性

如果 arch/dev 永远不能自动触发，不让修复卡在「它是 arch 所以必须做 Step 2」上。谁自动触发谁做。角色是标签，Step 是任务——解耦才能全自动。
