## 0. 本轮行为规则（全体必读）

### 0.1 本轮核心任务

> R59 的目标是**填上管线全自动的最后两个缺口**——arch 和 dev 在 Step 2/3 不能自动触发的问题。
> R58 的 from_name=PM 改造在其他 4 个 bot 上验证有效，但在 arch/dev 上无效。
> 本轮采用**先探测、再编码**的策略：先通过 WS 直连实验锁定触发条件，再针对性编码。
> 如果实验发现 ws-bridge 代码层面无法强制触发 arch/dev，则走兜底路径——PM 自动通知项目负责人。

### 0.2 主备映射（本轮）

| Step | 角色 | 主角 | 备用 | 约束 |
|:----:|:----|:----:|:----:|:-----|
| Step 2 | 🏗️ 技术方案 | arch | dev | — |
| Step 3 | 💻 编码 | dev | arch | 写方案的人 ≠ 编码的人 ✅ |
| Step 4 | 🔍 审查 | review | qa | 编码者 ≠ 审查者 ✅ |
| Step 5 | 🦐 测试 | qa | review | 编码者 ≠ 测试者 ✅ |
| Step 6 | 🦸 合并部署 | admin | ⛔ 无备用 | 仅 admin 有部署凭证 |

> **⚠️ 关于 arch/dev 的主备：** 如果方向 A 实验发现 arch/dev 完全不能自动触发，且方向 C（角色解耦）未实现，PM 将在本步的 `!step_complete` 中手动通知项目负责人触发 arch/dev。这不是备用换人（备用也是 dev/arch，同样不触发），而是 PM 自动兜底。
>
> **⚠️ Step 6 重要限制：** 合并部署环节**无法换人**——仅 admin-bot（项目管理）拥有生产容器的凭证，其他角色无部署权限。卡住时 PM 直接 TG 通知项目负责人处理。

> 主备由项目协调人（PM）在本 WORK_PLAN 中指定。项目负责人审核通过。各角色按照主备规则执行，**不能自己审自己**。

### 0.3 方向 A 实验的静默执行规则

方向 A（探测实验）通过 PM 私下的 WS 直连脚本执行，**不在工作室内发冗余消息**。

- 实验消息全部通过原始 WebSocket 发送到目标 bot 的私有频道或 `_admin` 频道
- 实验期间不在工作室频道产生干扰消息
- 实验结束后 PM 汇总结果，产出实验报告到工作室

### 0.4 本条行为规则的确认

当管线启动后点名到你的 Step 时，回复 ACK 后读取本条规则全文，按规则执行。

## 2. 管线步骤

### Step 1：管线启动（服务端自动）

| 项目 | 说明 |
|:-----|:------|
| **触发** | PM 在 `_admin` 频道执行 `!pipeline_start R59 --from step2 --mode auto` |
| **执行者** | 🤖 服务端自动 |
| **操作** | ① 暂停大厅接收 ② 建工作室 `ws:R59-dev` ③ `MSG_SET_ACTIVE_CHANNEL` 切频道 ④ 广播 PM 身份 @mention 点名 ⑤ 点名架构师附上下文 |
| **产出** | 管线状态 📊 + 工作室就绪 |

> **验证：** 触发后立即 `!pipeline_status` 确认 Step 指针在 step2

#### 同时：方向 A 探测实验（PM 私下执行，在 dev 环境）

在 Step 1 启动后、Step 2 点名架构师之前，PM 通过 WS 直连 **dev 容器** 执行探测实验：

1. 连接到 **dev 环境** WS（非生产，零风险）
2. 依次发送 from_name 变体（系统/PM/项目负责人/项目负责人ID）+ 消息格式变体 + 发送路径变体
3. 观察目标 bot（架构师/开发工程师）是否响应
4. 记录有效/无效到实验表

> **环境信息：** dev 环境独立于 main 生产环境，数据完全分离，有独立域名和端口。各 bot 客户端已配同时连接 dev+main。测试工程师可随时操作 dev 部署。详见需求文档「项目环境信息」章节。

**实验不阻塞管线推进**——PM 并行执行，实验结果用于方向 B 编码。

### Step 2：技术方案

| 项目 | 说明 |
|:-----|:------|
| **主角** | 🏗️ arch-bot（架构师） |
| **备用** | dev-bot（开发工程师） |
| **产出** | `docs/R59/R59-tech-plan.md` (commit sha) |
| **内容** | 方向 A 实验结果分析 + 方向 B（通知差异化改造）+ 方向 C（角色解耦）的技术方案 |
| **触发风险** | ⚠️ arch 可能不响应 PM @mention，需要项目负责人 TG 转发 |
| **完成条件** | ✅ 推 dev + 工作室 QA 讨论通过 |

**架构师须知：**

1. **方向 A（探测实验）** 是 PM 执行的工作，但架构师需要：
   - 审阅实验方案（from_name 变体 / 消息格式变体 / 发送路径变体）
   - 提供 **dev 环境的 WS URL 和连接信息**（架构师知道 dev 容器的域名和端口）
   - 确认实验不会对 dev 环境造成影响（虽然后者随便折腾）

2. **方向 B（编码适配）** 的核心设计：
   - 根据实验结论，在 `_cmd_step_complete` 中增加按角色差异化的通知格式
   - 如果 arch/dev 需要特殊 `from_name`，该值从配置读取（不硬编码）
   - 推荐实现方式：`if next_role in ["arch", "dev"]:` 分支（或更通用的配置驱动模式）
   - PM 自动兜底机制（5 分钟超时 → TG 通知项目负责人）

3. **方向 C（角色解耦）** 的设计：
   - `PIPELINE_STEP_MAP` 的 `role` 字段从固定值改为 PM 可配置
   - 约束检查（不能自审自测）的自动化
   - 优先级：方向 B 优先，方向 C 是兜底，无需深入设计

4. **需求文档参考：** `docs/R59/R59-product-requirements.md` §2-A/B/C

### Step 3：编码

| 项目 | 说明 |
|:-----|:------|
| **主角** | 💻 dev-bot（开发工程师） |
| **备用** | arch-bot（架构师） |
| **输入** | `docs/R59/R59-tech-plan.md`（架构师方案） |
| **产出** | 方向 B + B3 的代码提交 |
| **触发风险** | ⚠️ dev 可能不响应 PM @mention，需要项目负责人 TG 转发 |
| **完成条件** | ✅ 推 dev + 附带改动文件清单 |

**开发者须知：**
- 严格按技术方案的行号和函数名编写
- 方向 B 改动集中在 `_cmd_step_complete` 的 Step 交接通知段（R58 A2 标记附近，~L1615-1641）
- `from_name` 差异化值必须从配置读取——绝不硬编码
- PM 自动兜底（B3）是后台任务，注意 `asyncio.create_task()` 或 `ensure_future()` 的使用
- 改动前先 `git log --oneline -5` 看 dev 最新状态
- **⚠️ Post-fix 未部署 = 无效（R58 教训）：** 如果编码阶段发现 Bug 需要修复，推完 commit 后必须让 PM 手动绕过（`!step_complete` 推进），等 Step 6 部署时一并带上修复

### Step 4：代码审查

| 项目 | 说明 |
|:-----|:------|
| **主角** | 🔍 review-bot（审查工程师） |
| **备用** | qa-bot（测试工程师） |
| **输入** | 开发者提交的代码 commit |
| **产出** | `docs/R59/R59-code-review.md` |

**审查者须知：**
- 方向 A 实验报告中的触发条件是否被正确反映在代码中
- `from_name` 差异化值是否从配置读取（不是硬编码）
- 方向 B 的差异化分支是否不影响其他角色（review/qa/admin）的现有触发机制
- B3（PM 自动兜底）的后台任务是否使用 `asyncio.create_task()`，不会阻塞命令响应
- PM 自动兜底的 TG 通知是否安全（不泄漏内部信息）

### Step 5：测试验证

| 项目 | 说明 |
|:-----|:------|
| **主角** | 🦐 qa-bot（测试工程师） |
| **备用** | review-bot（审查工程师） |
| **输入** | 开发者提交的代码 |
| **产出** | `docs/R59/R59-test-report.md` |

**测试者须知：**
- 核心验收项：
  - 方向 B 差异化通知不影响 review/qa/admin 的现有触发
  - 按角色差异化分支正确触发不同角色
  - B3 PM 自动兜底的超时逻辑和 TG 通知格式
- 特别检查：如果实验发现 `from_name=PM` 仍无效，差异化分支是否正确使用实验确认的有效值

### Step 6：合并部署归档

| 项目 | 说明 |
|:-----|:------|
| **主角** | 🦸 admin-bot（项目管理） |
| **备用** | ⛔ **无备用** — 仅 admin 有部署凭证 |
| **操作** | ① dev 环境部署验证 → ② dev→main 合并 → ③ 构建+部署生产容器 → ④ 健康检查 → ⑤ 更新 TODO.md → ⑥ 关闭工作室 |
| **产出** | 部署成功 + 管线完结 |
| **卡死处理** | admin 不响应 → PM TG 通知项目负责人，不走备用换人 |

**执行者须知：**
- 先确认 `docs/R59/` 下所有文档齐全
- 再确认 TODO.md 版本号 bump + 变更记录更新
- 合并前确认代码审查+测试已通过
