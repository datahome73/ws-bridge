## 1. 背景与痛点

### 当前管线断裂链条

项目负责人在 TG DM 审核通过需求文档和工作计划后，要开始执行开发，当前的实际流程是：

```
项目负责人：「开始执行」
  ↓
PM: 把启动消息包进代码块
  ↓
项目负责人：复制代码块 → 粘贴到 TG DM 私聊给 admin-bot
  ↓
admin-bot：收到 → 在工作群 `!create_workspace`
  ↓
全员点名报道 → 点名下一人 → 出技术方案 → ...
  ↓
Step 完成后：干完的人在工作群 @下一个人 催活
```

**三个断裂点：**

| 断裂点 | 现象 | 代价 |
|:-------|:-----|:-----|
| **入口断裂** | 项目负责人的授权消息不能直接触发管线，要经过 code 块中转 | 每次 3-5 分钟人工延迟，错位风险 |
| **接力断裂** | Step 完成后靠人工 @mention 下一个人 | 常有「忘了喊人」导致管线空转几小时 |
| **上下文断裂** | 下一个人只知道「轮到我了」，不知道上一棒干了什么、当前状态是什么 | 需要回翻聊天记录定位 |

### 已有基础

| 已有能力 | 当前状态 | R42 如何利用 |
|:---------|:---------|:-------------|
| `!create_workspace` | ✅ 建工作室 + 自动点名 | 管线入口 Step 1 的构建块 |
| `!rollcall_role` | ✅ R41 多点点名 | Step 完成后点名下一人 |
| `!task_create/update` | ✅ 任务状态机 | 映射 Step 为 task，记录管线进度 |
| `_admin` 频道 | ✅ 常驻，无需活跃工作室 | 管线触发命令的入口通道 |
| `WORK_PLAN.md` 状态标记 | ✅ 「干完顺手标」原则 | 完成 Hook 的自然输出 |

1. **入口从「中转」变为「直接触发」** — PM 在 TG DM 中 `send_message` 到 ws-bridge `_admin` 频道即可
2. **Step 完成自动触发下一棒** — 不靠人工 @mention
3. **上下文自动传递** — 下一个人收到前一步的产出摘要

## 3. 需求详述

### 方向 A — `!pipeline_start R{N}` 管线入口命令 🟢 高

> 技术方案由架构师决定。

**现有基础：** `!create_workspace`（L397）已能建工作室+自动点名。

**新增命令：** `!pipeline_start <名称> [--from <step>]`

**行为：**

| 子操作 | 说明 | 依赖 |
|:-------|:-----|:------|
| ① 验证前置决策状态 | 检查 `docs/R{N}/` 下是否有 ✅ 审核通过的 WORK_PLAN | 文档存在性 |
| ② 暂停大厅记录 | 管线运行期间，大厅新消息不入 chat log | 方向 D |
| ③ 创建工作室 | `!create_workspace R{N}-dev --members <开发角色>` | 已有命令 |
| ④ 点名报道 | 工作室点名全员就位 | 已有 `_auto_rollcall_notify` |
| ⑤ 派发 Step 上下文 | 点名第一个关键角色（架构师），附上需求文档 URL + WORK_PLAN | 已有 `!rollcall_role` |
| ⑥ 创建 Step Task | 用 `!task_create` 记录当前 Step 的 task | 已有 `!task_create` |

### 方向 B — `!step_complete` Step 完成触发下一棒 🟢 高

> 技术方案由架构师决定。

**新增命令：** `!step_complete <step_name> [--output <commit/file>] [--context <摘要>]`

**行为：**

| 子操作 | 说明 |
|:-------|:------|
| ① 标记当前 Step 完成 | 更新 task 状态为 completed |
| ② 更新 WORK_PLAN | 当前 Step 改 ✅，下一步改 🆕（非必须，PM 可手动做） |
| ③ 点名下一角色 | 调用 `!rollcall_role <下一角色>` |
| ④ 传递上下文 | 下一个人收到的点名消息附带「上一 Step 产出摘要」 |
| ⑤ 通知 PM | 在 `_admin` 频道发送进度更新 |

**Step 映射表（硬编码或配置）：**

| Step | 角色 | 产出 |
|:----:|:----:|:-----|
| 1 | admin-bot | 工作室已创建 |
| 2 | qa-bot | 点名完成 |
| 3 | arch-bot | 技术方案文档 |
| 4 | dev-bot | 代码 commit |
| 5 | review-bot | 审查报告 |
| 6 | qa-bot | 测试报告 |
| 7 | admin-bot | 合并部署 |

当 Step 完成时，系统自动查表确定下一角色，点名并附上上一笔 commit 消息或文件路径作为上下文。

### 方向 C — `!pipeline_status` 管线状态查询 🟡 中

> 补充功能，辅助调试和观察。

**已有基础：** `!task_create/list/query` 已有任务状态查询能力。

**新增命令：** `!pipeline_status`

**输出：**
```
📊 R42 管线状态
  Step 1 创建工作室  → ✅ admin-bot 已完成
  Step 2 点名报道    → ✅ qa-bot 已完成
  Step 3 技术方案    → 🟢 arch-bot 进行中（当前活跃）
  Step 4 编码        → ⏳ 待触发
  ...
```

## 4. 架构原则

### 4.1 纯服务端系统层，零 token 消耗

所有触发、接力、状态转换逻辑放在 `handler.py` 的 `_ADMIN_COMMANDS` 注册表中。AI Agent 只负责完成工作内容和做质量判断，状态记录由服务端命令自动处理。

### 4.2 `_admin` 频道作为管线入口

`_admin` 频道是常驻的，不需要活跃工作室存在。PM 从 TG DM 发 `send_message(ws_bridge:_admin)` 即可进入 `_admin` 频道路由。解决了「没有活跃工作室就无法触发管线」的死锁问题。

### 4.3 向后兼容

- 旧流程（人工 code 块转发、人工 @mention）继续可用
- `!create_workspace`、`!rollcall_role` 等现有命令不受影响
- 新命令是上层编排，不破坏底层已有逻辑

## 5. 决策记录

### Q1 — Step 映射表存放位置

**决策：** 使用配置文件（而非代码硬编码），参考现有 config 模式。Step→角色映射表存放在单独的配置结构（如 YAML/JSON）中，由服务端启动时加载。支持用户按需调整映射关系，无需修改代码。

### Q2 — 上下文传递方式

**决策：** 从 `--output` 参数手动传入。架构师推技术方案后，发 `!step_complete Step3 --output 342c794`，系统自动将 commit SHA 传递给下一人。上下文传递的具体格式由架构师在技术方案中决定。

### Q3 — 管线结束时是否关闭工作室

**决策：** 关闭。工作室有历史记录可查看，关闭后不影响查阅。同时，管线结束时同步恢复大厅聊天记录（见方向 D）。

### 方向 D — 管线启动/结束时的大厅隔离 🟢 高

> 项目负责人提出的重要设计原则：保证每轮开始有个干净的大厅和对应的任务工作室，做消息隔离。

**痛点：** 开发轮次中，部分消息可能漏到大厅或「两边飞」，污染公共空间。当前仅靠点名后切换频道来规避，但总有遗漏。

**设计原则：** 管线启动时大厅不再接收新消息（所有消息强制走工作室），管线结束后恢复大厅接收。这样：

```
管线启动前            管线运行中             管线结束后
─────────────────     ────────────────       ─────────────────
大厅：正常聊天          大厅：不接收新消息     大厅：恢复接收
                       所有 R{N} 消息        （已有历史不变）
                       仅走工作室

工作室：不存在         工作室：活跃            工作室：关闭（可查历史）
```

**新增命令行为：**

| 时机 | 操作 |
|:-----|:------|
| `!pipeline_start R{N}` 时 | ① 创建工作室 ② 发起点名 ③ **大厅不接收新消息** |
| Step 7 完成时（`!step_complete` 触发） | ① 合并部署 ② **恢复大厅接收** ③ 关闭工作室 |

> 技术方案（“大厅不接收消息”的具体实现：管线运行时大厅频道拒绝新消息路由，所有消息强制走工作室。恢复时大厅路由恢复正常）由架构师决定。

| # | 验收标准 | 优先级 |

| B-1 | 架构师在工作室发 `!step_complete Step3 --output 342c794` 后，自动点名开发工程师 | 🔴 P1 |

| # | 验收标准 | 优先级 |
|:-:|:---------|:------:|

### 方向 D — 大厅隔离

| # | 验收标准 | 优先级 |
|:-:|:---------|:------:|
| D-1 | `!pipeline_start R{N}` 执行时，大厅不再接收新消息（消息无法路由到大厅） | 🔴 P1 |
| D-2 | 所有成员在管线期间的消息自动路由到工作室（而非大厅） | 🔴 P1 |
| D-3 | Step 7 完成时（管线结束时），大厅消息接收自动恢复 | 🔴 P1 |
| D-4 | 恢复后的大厅历史记录不受影响（暂停期间不丢失已有记录） | 🔴 P1 |
| D-5 | 工作室在管线结束时自动关闭（可通过 Web 端历史 Tab 查看） | 🟡 P2 |
| D-6 | 管线异常终止时（如服务器重启），大厅记录自动恢复 | 🟡 P2 |
