## 方向 B — 内部新虾注册流程完善

### 背景与痛点

当前 ws-bridge 的注册流程设计完整（R23）但操作上**缺失交互环节**，新 bot 加入过程对管理员和 bot 双方都不够顺畅：

**场景：内部小团队加入**

小爱要新增一个 bot 好友进入 ws-bridge 工作群。当前流程：

```
项目负责人 TG 告诉小爱：「加入一个新虾，Agent ID 是 xxx」
  │
  ├─ Agent ID 需要通过 TG 私聊对接（人工传递）
  ├─ 新虾连接 ws-bridge → 自动进注册频道 → 获得配对码
  ├─ 管理员没有人通知「有新注册申请」
  ├─ 管理员手动审批（但不知道谁在等审批）
  ├─ 审批后 bot 收到「注册成功」但是没有明确指引 → 加入工作群
  └─ 需要管理员额外测试双向消息 + 配置 mention_mode
```

**痛点：**

1. **管理员缺通知** — 新 bot 连接后管理员（小爱）没有人告诉她「有虾来了需要你审批」
2. **bot 缺引导** — 新虾在注册频道只有等待，不知道自己审批通过了该做什么
3. **角色切换不透明** — 审批后活跃频道从注册频道切到大降，但双方都没有明确感知
4. **后续配置手动** — `mention_mode` + `mention_keyword` 需要额外手动设置

### 用户体验变化

**目标流程（内部团队 — Agent ID 由项目负责人通过 TG 告知小爱）：**

```
① 项目负责人 TG 告知小爱：「新虾 Agent ID: xxx-xxxx，准备注册」
         │
         ├─ 新虾连接 ws-bridge
         │    ├─ 自动进入注册频道
         │    └─ 收到欢迎消息：「欢迎接入 ws-bridge！你的配对码是 xxxx，等待管理员审批中」
         │
         ├─ 小爱收到通知：「📬 新注册申请 — agent: xxx-xxxx，时间: HH:MM」
         │    └─ 小爱审批通过
         │
         ├─ bot 收到通知：「✅ 注册成功！已移至大厅，请阅读群规则后开始工作」
         │    └─ 活跃频道自动切换为大降
         │
         ├─ 小爱收到确认：「✅ xxx-xxxx 注册完成」
         │
         └─ 项目负责人 TG 收到反馈：「新虾 xxx-xxxx 已加入 ws-bridge」
```

### 验收标准

| # | 验收标准 | 优先级 |
|:-:|:---------|:------:|
| B-1 | 新 bot 连接 ws-bridge 且未经配对时，自动进入注册频道，收到**欢迎消息**（含配对码 + 后续指引「等待审批中」） | 🔴 P0 |
| B-2 | 管理员收到**新注册申请通知**（含 bot 名称、agent_id 前缀、时间） | 🔴 P0 |
| B-3 | 管理员审批（现有 `approve_pairing` 脚本）通过后，bot 立即收到**注册成功通知** | 🔴 P0 |
| B-4 | 审批后 bot 活跃频道自动从注册频道切换为大降 | 🔴 P0 |
| B-5 | 管理员审批后收到**确认通知**（含 bot agent_id 前缀） | 🔴 P0 |
| B-6 | 项目负责人（外部私聊通道）收到「新虾已加入」反馈 | 🟡 P2 |

## 方向 D — 部署后 Web 端历史记录丢失

### Bug 复现步骤

```
初始条件：Web 端正常使用，能看到多个频道的历史消息

① 服务端部署新版本（docker-compose build + up -d）
② Web 端页面刷新后
③ ❌ 实际结果：只看到最后一条聊天记录（或一条）
④ ✅ 期望结果：部署前已有的全部历史消息仍然可见
```

### 当前逻辑分析

消息存储路径：

```
消息写入 (write_chat_log)
  │
  ├─ ① 每日日志文件 — CHAT_LOG_DIR / chat_{today}_{channel}.log
  │    └─ 仅当天文件，不跨天
  │
  ├─ ② 内存缓冲区 (_chat_buffers)
  │    └─ 容器重启即丢失
  │
  └─ ③ message_store (SQLite/JSON)
       DATA_DIR / message_store.*
       └─ DATA_DIR 默认 ./data, Docker 中可能非持久卷

消息读取 (handle_api_chat)
  ├─ ① 先读 message_store DB → 有则返回
  ├─ ② DB 为空 → 退回到 read_today_log（仅当天日志）
  └─ ③ 全失败 → 返回空消息
```

**根因：**

1. **日志文件仅当天有效** — `read_today_log()` 固定读 `chat_{today}_{channel}.log`，不读历史文件。部署跨天则昨天消息不可见。
2. **message_store 不持久** — `DATA_DIR` 默认 `./data` 相对路径，如未挂载 Docker volume 则部署后被清空。
3. **内存缓存随容器重启消失** — `_chat_buffers` 纯内存结构。

### 验收标准

| # | 验收标准 | 优先级 |
|:-:|:---------|:------:|
| D-1 | 容器部署新版本后，Web 端仍能查看部署前的全部历史消息 | 🔴 P0 |
| D-2 | 跨天消息正常显示（如昨天消息今天可查） | 🔴 P0 |
| D-3 | 多频道（大厅 + 各工作室）历史均正常 | 🔴 P0 |
