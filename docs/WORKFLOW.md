## 1. 团队角色分工

| 角色 | 成员 | 职责 |
|------|------|------|
| **项目管理 / 调度员** | 🦸 项目管理 | 任务编排、调度、消息中转、汇总报告给项目负责人 |
| **研究专家（需求分析）** | 需求分析师 🧐 | 需求调研、现状分析、可行性方案报告、方向审查 |
| **架构师（技术方案）** | 架构师 🏗️ | 技术方案设计、实施方案制定、技术风险评估 |
| **开发工程师** | 开发工程师 💻 | 代码编写、功能实现、单元测试 |
| **审查工程师** | 审查工程师 🔍 | 代码审查、审计报告、质量把关、向后兼容确认 |
| **测试工程师** | 测试工程师 🦐 | 部署验证、生产环境测试、E2E 测试、提 Issue |

> 所有代码/配置变更需审查工程师审查通过后才进入测试。

## 3. 各环节详细说明

### Step 1 — 创建工作室

**目的：** 为本轮次创建独立的工作空间，隔离团队协作。

**流程：**
1. 项目管理在 ws-bridge 中创建新的 workspace
2. 指定 workspace 名称（如 `R25开发工作室`）
3. 指定成员列表（参与本轮的 bot）
4. **项目负责人审批 WORK_PLAN** — 项目管理将 WORK_PLAN 提交后，项目负责人在创建工作室时即确认审批通过
5. 工作室创建后自动通知所有成员
6. 各成员确认收到工作室创建通知

**经验总结：**
- 工作室命名的轮次编号必须准确，如 `R25开发工作室`
- 成员列表包含所有参与本轮的 bot，遗漏需及时补加
- 项目管理（项目管理）默认是工作室管理员

### Step 2 — 点名（全员确认在线）

**目的：** 确认所有参与本次轮次的 bot 在线且就绪。

- 项目管理在工作室发点名消息
- 各成员在 **工作室频道** 回复确认（不在大厅回复）
- 点名超时（默认 30 秒无响应）→ 项目管理判定是否需要角色互换
- 点名完成后，各成员回到活跃频道等待派活

**经验总结（R13-R25）：**
- 📋 点名路由到 @ 目标，点名消息发到对应的 workspace 而非大厅
- 点名时各成员的活跃频道保持在工作室，点名消息仍正常送达

### Step 3 — 需求文档 🔑（关键审核闸门）

> **这是整个开发流程中最关键的审核环节。** 只有项目负责人审核通过后，流水线才能推进到 Step 4 方案阶段。

**目的：** 需求分析师阅读 WORK_PLAN 了解开发范围后，基于 docs 背景资料 + 与项目负责人讨论 + 调研结果编写产品需求文档，项目负责人审核确认方向正确。

**流程：**
1. 需求分析师调研当前状态 → 出调研报告
2. 需求分析师编写产品需求文档初稿（`R{轮次}-product-requirements.md`）
3. 需求文档提交到仓库 `docs/R{轮次}/`
4. **项目负责人 外部私聊通道审核** — 项目负责人阅读全文后批准（✅通过）或要求修改（❌退回）
5. ✅ **通过** → 进入 Step 4 方案阶段
6. ❌ **退回** → 需求分析师根据项目负责人反馈修正需求文档，出下一稿 → 重新提交项目负责人审核

**审核方式：** 通过外部私聊通道与项目负责人讨论——项目负责人指出问题，需求分析师改文档，再推给项目负责人看，直到通过。

**为什么这是关键闸门：**
- 需求错了，后面所有环节（方案、编码、测试）都是错的
- 项目负责人作为产品决策者，在此环节确认方向后才值得投入开发资源
- 此环节通过后，项目管理方可推进后续流水线，项目负责人不再打断

**产出：**
- `R{轮次}-product-requirements.md`（需求分析师 — 产品需求文档）

### Step 4 — 方案阶段

**目的：** 需求分析师调研现状，架构师设计技术方案。

**流程：**
1. 需求分析师调研当前状态 → 出调研报告
2. 架构师根据调研报告出技术实施方案
3. 方案提交到仓库 `docs/R{轮次}/`

**产出：**
- `R{轮次}-tech-plan.md`（架构师 — 技术方案）

### Step 5 — 方向审查

**目的：** 需求分析师审查技术方案是否满足需求，确认方向正确性。

**审查结论：**
| 结论 | 含义 | 后续动作 |
|:----|:-----|:---------|
| 🟢 **无条件通过** | 方案完全满足需求 | 直接进入编码 |
| 🟡 **有条件通过** | 方案基本满足，有 🔴 必修项待修复 | 修复后进入编码 |
| 🔴 **不通过** | 方案不符合需求或存在重大问题 | 退回 Step 4 重新设计 |

**审查报告结构：**
- 🔴 **必修** — 必须修复后才能进入 Step 6 编码
- 🟡 **建议** — 可延后到后续轮次
- 🟢 **可延后** — 非阻塞，记录即可

**产出：** `R{轮次}-direction-review.md`

### Step 6 — 编码阶段

**目的：** 开发工程师根据技术方案实现代码，推送到 dev 分支。

**流程：**
1. 开发工程师拉取最新 dev 分支
2. 按技术方案实现功能
3. 运行单元测试
4. 推送到 dev 分支

**产出：** 代码推送到 GitHub dev 分支

**Git 提交规范：**
- 提交信息格式：`{type}({scope}): {描述}`
- 示例：`feat(handler): 添加大厅消息前缀路由`
- 示例：`docs(R25): Step 7 测试工程师测试报告`
- Commit 署名必须使用 bot 自己的名字

### Step 7 — 开发审查

**目的：** 审查工程师审查代码变更，确保实现符合设计、向后兼容。

**流程：**
1. 审查工程师审查 dev 分支上的所有变更（代码 + 配置 + 文档）
2. 审查维度：功能完整性、代码质量、安全性、向后兼容、文档同步
3. 出审查报告

**审查通过条件：**
- 所有 🔴 阻塞项已修复或明确理由不修复
- 实现符合设计文档
- 无向后兼容问题

**不通过处理：**
- 出正式审计文档到 `docs/` 目录，列出：
  - 每个问题的位置（文件名 + 行号）
  - 问题描述
  - 严重度（🔴 阻塞 / 🟡 建议 / 🟢 微小）
  - 修复要求
- 不通过的代码不回退，在 dev 分支修复后重新审查
- 审查退回不超过 2 轮，超过则上报项目负责人决策

**产出：** `R{轮次}-code-review.md`

### Step 8 — Dev 部署

**目的：** 将 dev 分支代码部署到 dev 服务器，供测试工程师测试。

**流程：**
1. 项目管理将 dev 分支代码部署到 dev 服务器
2. 确保 dev 服务器上的容器使用最新的 dev 代码
3. 验证 dev 服务正常启动（health check）
4. 在工作群通知测试工程师 dev 环境就绪

**注意：** dev 部署仅在 dev 服务器上进行，不影响生产环境。

### Step 9 — Dev 测试

**目的：** 测试工程师在 dev 环境执行全面测试。

**流程：**
1. 测试工程师在 dev 服务器上执行测试用例
2. 覆盖 P0 核心功能、P1 兼容性、P2 边界情况
3. 各成员按测试用例在 dev 环境逐项验证

**测试类型：**

| 测试类型 | 执行者 | 覆盖内容 |
|:--------|:------|:---------|
| P0 核心功能 | 测试工程师 | 本轮核心改动的主流程 |
| P1 兼容性 | 测试工程师 | 与现有功能的兼容性（回归）|
| P2 边界情况 | 测试工程师 | 异常输入、错误处理路径 |
| 交叉验证 | 全员 | 各成员自身体验报告 |

**测试结论：**
- ✅ **全部通过** → 进入 Step 10 上线测试验证
- ❌ **有失败** → 退回对应环节修复

**产出：** `R{轮次}-test-report.md`

**注意：** 此阶段的测试在 **dev 服务器** 上进行，不涉及生产环境。

### Step 10 — 上线测试验证

**目的：** 在 dev 服务器上做预发布验证，确认代码可以上线到生产环境。

**这是关键的「上线前闸门」环节。** dev 测试通过后，不直接上线生产，而是在 dev 服务器做一次完整的上线前验证。

**验证内容：**
1. 本轮功能完整流程走通
2. 与现有功能的兼容性无退化
3. 数据完整性检查（无数据丢失）
4. 回滚方案确认（必要时）
5. 各成员确认各自的体验报告已完成

**通过条件：**
- ✅ 完整流程验证通过
- ✅ 各成员体验报告已提交到 `docs/R{轮次}/`
- ✅ 无未解决的 🔴 阻塞项

**验证结论：**
- ✅ **通过** → 进入 Step 11 合并 main + 更新容器
- ❌ **不通过** → 退回对应环节修复

**产出：** `R{轮次}-release-verification.md`（项目管理 — 上线验证报告，含验证项逐条结果 + 结论标记 ✅/❌）

### Step 11 — 合并 main + 更新容器

**目的：** 上线验证通过后，合并 dev→main，更新生产容器。

**流程：**
1. 项目管理执行合并：`git checkout main && git merge dev && git push origin main`
2. 更新生产容器（docker restart 或 Railway redeploy）
3. 确认生产服务正常启动
4. 在工作群报告上线状态


### 备份清单（合并前必须执行）

合并 dev→main 前，项目管理必须完成以下数据备份：

| # | 备份内容 | 来源 | 说明 |
|:-:|:---------|:-----|:-----|
| 1 | **workspace 数据** | `workspaces.json` | 所有活跃/归档 workspace 的成员和状态 |
| 2 | **配对信息** | `_pairing_codes.json`、`_approved_users.json` | 所有已认证 bot 的永久配对数据 |
| 3 | **活跃频道状态** | `_agent_active_channels.json` | 各成员的当前活跃频道 |
| 4 | **Web 会话** | `_web_sessions.json`、`_web_bind_codes.json` | Web 端绑定状态 |
| 5 | **环境变量/配置** | `.env`、容器环境变量 | 部署配置快照 |

**备份操作：**
1. 复制上述文件到安全备份路径
2. 截图或导出容器环境变量
3. 确认备份文件可读、完整
4. 备份完成后才能执行 `git merge`

**恢复验证：** 合并后至少抽查 1 个 workspace 状态和 1 个 bot 配对信息是否正常。

---

**合并规则：**
- 上线验证通过 → **项目管理合并 dev→main**（无需等项目负责人审批）
- 合并后立即更新生产容器
- 文档与代码同时合并，确保 main 有完整文档记录
- 合并前备份关键数据（workspace、配置等）

### Step 12 — 关闭工作室

**目的：** 开发完成，归档工作室。

**流程：**
1. 项目管理发起关闭工作室请求
2. 各成员发 ACK 确认（全员确认后才关闭）
3. 工作室归档，消息不再发送到该 workspace
4. 各成员切换活跃频道回到大厅待命

**注意：**
- 关闭 workspace 使用 WebSocket `workspace_close`（auth 需 agent_id + app_id）
- 收到 `workspace_closing` 消息 → 立即发 ACK → 不再在该 workspace 发任何消息
- 关闭后归档的工作室数据保留，可查阅历史记录

## 4. 环节流转与汇报机制

### 流程控制 — 通过/退回对照

| 环节 | 审核/闸门 | ✅ 通过 → 下一环节 | ❌ 退回 → 修复后重审 |
|:----|:---------|:-----------------|:-------------------|
| Step 1 创建工作室 | 项目负责人审 WORK_PLAN | 工作室就绪，点名启动 | 项目负责人调整后项目管理重提 |
| Step 2 点名 | 全员确认在线 | 全员到位 → Step 3 | 缺席者替代调度 |
| **Step 3 需求文档** 🔑 | **项目负责人审需求分析师需求文档** | **✅ 通过 → Step 4 方案** | **❌ 退回 → 需求分析师修正 → 项目负责人再审** |
| Step 4 方案 | 需求分析师+架构师提产出 | 方案就绪 → Step 5 | — |
| Step 5 方向审查 | 需求分析师审方案方向 | 🟢/🟡 通过 → Step 6 | 🔴 退回 → 架构师修正 → 重审 |
| Step 6 编码 | 开发工程师实现 | 代码推 dev → Step 7 | — |
| Step 7 开发审查 | 审查工程师审代码 | 🟢 通过 → Step 8 | 🔴 退回 → 开发工程师修复 → 重审 |
| Step 8 Dev 部署 | 项目管理部署 | dev 就绪 → Step 9 | — |
| Step 9 Dev 测试 | 测试工程师测试 | ✅ 全通过 → Step 10 | ❌ 失败 → 对应环节修复 |
| Step 10 上线验证 | 项目管理+全员 | ✅ 通过 → Step 11 | ❌ 不通过 → 对应环节修复 |
| Step 11 合并 main | 项目管理执行 | main 合并+容器更新 → Step 12 | — |
| Step 12 关闭工作室 | 全员 ACK | 归档完成 | 未全员确认则等待 |

### 交付物存放规则

- **所有调研报告、技术方案、文档必须提交到对应项目的 GitHub 仓库 `docs/` 目录**
- 不允许私聊发文件给项目负责人，必须走 GitHub
- 提交后在工作群通知（@管理员 确认已提交）
- 禁止在工作群直接发长报告内容，只发摘要

### 汇报通道

- 各成员完成任务后 → 在工作群 **@管理员** 汇报（简短摘要）
- 项目管理汇总 → 通过 **外部私聊通道** 报告给项目负责人
- 外部私聊通道消息用 code block 包装，方便项目负责人复制粘贴转发各成员

### Issue 路由机制

测试工程师提 Issue 后，工作群广播 @项目管理，项目管理判定：

| 类型 | 标签 | 处理 |
|------|------|------|
| 🔧 实现问题 | `bug` | 派回开发工程师修 |
| 📐 设计缺陷 | `design` | 回架构师改方案 |
| ❓ 需求模糊 | `clarify` | 回来问项目负责人 / 退回需求文档 |
| ✅ 验证通过 | `verified` | 归档，下一轮 |

## 5. 其他通用规范

### 5.1 会话生命周期

会话生命周期分两种模式，取决于角色是否为**工作室管理员**。

#### 模式 A：短任务模式（非管理员 — 架构师/开发工程师/审查工程师/测试工程师）

非管理员角色的各成员执行的是**单步任务**：拿到上下文 → 干完 → 结束会话 → 释放资源，等下一个任务再重新激活。

```
🦸 项目管理派活（给上下文 + 工作背景知识）
  |
  ▼
执行当前环节任务
  |
  ▼
汇报结果（在工作群 @管理员）
  |
  ▼
会话结束 → 释放资源 → 回到静默待命
      |
      ▼
🦸 项目管理派下一个任务 → 重新激活
```

- 非管理员不跨环节保持会话，不持有整轮上下文
- 干完当前任务后立即结束会话，释放 token 和上下文资源
- 下一个任务来时重新获得准确的上下文和背景知识
- **禁止将未完成的任务拖到下一轮** — 如有阻塞在工作群说明

#### 模式 B：整轮模式（工作室管理员 — 🦸 项目管理 + 🧐 需求分析师）

项目管理（项目管理）和需求分析师（工作室管理员）因承担跨环节的调度/方向把控职责，会话贯穿整轮开发：

```
点名 → 接收整轮 WORK_PLAN
  |
  ▼
跨环节持续值守（调度、催人、汇总、方向审查）
  |
  ▼
轮次结束 → 关闭工作室 → 会话结束
```

- 整轮期间不清除上下文，保持对全局进度的追踪
- 项目管理负责编排调度和合并部署，需求分析师负责方向审查
- 轮次结束后关闭工作室时一同结束会话

### 5.2 角色互换规则

| 场景 | 处理 |
|:-----|:------|
| **主力不在线 >5 分钟** | 在项目管理判定后备选接替 |
| **项目负责人主动要求换人** | 立即执行，无需等待 |
| **主力恢复上线** | 自动退回原角色，备选回到待命 |

- 只有项目负责人主动提出或主力不在线时才能换角色
- 任何 bot **不得擅自假设**自己可以代行其他 bot 的职责
- 如果主力 bot 掉线 → 先在工作群 @该 bot 确认无响应（等 30 秒）→ 项目管理判定后才换人
- 角色互换后，原主力 bot 恢复上线时自动退回原角色

### 5.3 Git 署名规则

- 所有 Git commit 必须使用 bot 自己的名字（`git config user.name`）
- 署名不对 → 项目管理退回，要求重提
- 禁止冒用其他 bot 的名义提交代码或文档
- 允许的署名格式：中文名（如"开发工程师"）、英文名（如"aite"）、或用户名+描述（如"审查工程师 🔍"）

### 5.4 Gateway 配置要求

每个 bot 的 Gateway 必须确保：

1. **ws_bridge 平台正确连接** — agent_id 写入永久 pairing 文件（项目管理协助）
2. **mention_mode: true** — 工作群消息通过 @mention 激活，不消耗沉默 token
3. **mention_keyword 用自己名字** — 如开发工程师用"开发工程师"，需求分析师用"需求分析师"，项目管理用"项目管理"
4. **prompt 中配好授权路由规则**
5. **完成一次双向消息测试** — 项目管理发测试消息 → bot 回复确认
6. **主通道为 外部私聊通道** — 项目管理的私聊是默认通信通道

### 5.5 自检原则

Gateway 遇到问题时，bot 必须优先自检：

| 问题 | 自检项 |
|------|--------|
| Gateway 崩溃/断开 | 查自身容器日志（Railway logs / `journalctl`）|
| 收不到工作群消息 | 检查 mention_mode 配置、ws_bridge 插件配置、配对状态 |
| 消息发不出 | 检查 Gateway 日志和 ws_bridge 连接状态 |
| 连接掉线 | 等自动重连（最多 30 秒），重连失败再 @项目管理 |
| 配对风暴 | 检查对端 text_id 是否更新，清理 `_rate_limits/` 和 `pending` 目录 |

- 自检搞不定时才在工作群 @项目管理 求助
- 项目管理不替代各成员排查，只派活和协调
- 各成员排查结论需附带证据（日志截图、配置输出）
