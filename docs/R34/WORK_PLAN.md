## 🟢 自动化管线

### 角色分工

| 角色 | 成员 | 职责 |
|:----:|:----:|:-----|
| 🏗️ 架构师 | 架构师 | 技术方案编写 |
| 💻 开发工程师 | 开发工程师 | 编码实现 |
| 🔍 审查工程师 | 审查工程师 | 代码审查 |
| 🦐 测试工程师 | 测试工程师 | 点名报道 + Dev 测试 |
| 🦸 项目管理 | 项目管理 | 建工作室 + Dev 部署 + 合并归档 |

### Step 1 — 建工作室 🦸 项目管理

| 动作 | 说明 |
|:-----|:------|
| 创建 R34 开发工作室 | 命名 `R34-dev`，指定全员加入 |

### Step 2 — 点名报道 🦐 测试工程师

| 动作 | 说明 |
|:-----|:------|
| 主持点名，全员确认就位 | 回复「到」确认在线 |
| 超时（3分钟）默认到齐 | 统一切换活跃频道到工作室 |

### Step 3 — 技术方案 🏗️ 架构师

| 动作 | 产出 |
|:-----|:-----|
| 根据需求文档出技术方案 | `docs/R34/R34-tech-plan.md` |
| 工作室全员讨论评审 | 评审意见汇总记录 |

**内容要求：**
- 需求 A：`workspace_reset` 消息类型定义 + 服务端处理逻辑（权限检查、状态判断、广播范围、日志）
- 需求 B：ACK 附加 delivery 字段（total/sent/offline/targets）
- 向后兼容方案
- 测试要点

### Step 4 — 编码实现 💻 开发工程师

| 动作 | 产出 |
|:-----|:-----|
| 按技术方案编码 | 推 `dev` 分支（`r34-rehearsal`） |

改动范围：仅 `server/handler.py`

### Step 5 — 代码审查 🔍 审查工程师

| 动作 | 产出 |
|:-----|:-----|
| 审查代码，确认实现符合设计 + 向后兼容 | `docs/R34/R34-review.md` |

> **不通过：** 出审计文档，dev 分支修复后重新审查。

### Step 6 — Dev 部署 🦸 项目管理

| 动作 | 说明 |
|:-----|:------|
| 部署 dev 容器 | `ws-im-dev.datahome73.com` |

> ⚠️ 需求 A 需 admin 角色测试，dev 环境需配置至少一个 admin。

### Step 7 — Dev 测试 🦐 测试工程师

| 动作 | 产出 |
|:-----|:-----|
| 在 dev 环境执行测试用例 | `docs/R34/R34-test-report.md` |

测试要点：
| 需求 | 用例 | 预期 |
|:----:|:-----|:------|
| A-T1 | 管理员对活跃工作室发 reset | 所有成员收到 `force: true` 广播 |
| A-T2 | 管理员对 CLOSING 工作室发 reset | 返回 error |
| A-T3 | 非管理员对工作室发 reset | 权限不足 error |
| B-T1 | 消息到有 3 人在线的工作室 | ACK delivery.sent=3 |
| B-T2 | 消息到有 2 人在线 + 1 人离线的工作室 | ACK delivery.sent=2, offline=1 |
| B-T3 | 限速时发消息 | 收到 error，不收到 ack |
| B-T4 | 无前缀消息发大厅 | 收到 error |

### Step 8 — 合并部署 & 归档 🦸 项目管理

| 子步骤 | 动作 | 产出 |
|:------:|:-----|:-----|
| 8a | 合并 `dev` → `main` | main 分支更新 |
| 8b | 部署正式容器 + 关闭工作室 + 归档文档 | 轮次闭环 |

---

## 注意事项

1. 需求 A、B 同文件（`handler.py`），编码可串行实施，建议先 A 后 B
2. 需求 B 新增字段完全向后兼容，旧 Gateway 忽略新字段
3. Test report 中 A-T1/A-T4 需要真实 admin 角色 + 工作室环境
4. 测试前确保 dev 环境有 admin 配置
