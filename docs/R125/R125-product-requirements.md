## 2. 修复项

### 修复 A — 删除死代码（F-1）

> **来源：** R124 QA F-1
> **预计：** 删 ~98 行

**问题：** 文件末尾（L2950-L3068）残留了旧版 `_handle_reject` 和 `_archive_pipeline` 函数定义。这两个函数在文件中部（L3499 和 L3839）已有升级版实现（带 `PS.COMPLETED` 守卫等增强）。

**旧版与新版对比：**

| 函数 | 旧版位置 | 新版位置 | 差异 |
|:-----|:--------|:--------|:-----|
| `_handle_reject` | L2951 | L3499 | 旧版缺 `PS.COMPLETED` + `stuck` 状态守卫 |
| `_archive_pipeline` | L3000 | L3839 | 旧版有 `_notify_pm`（L3039），新版缺（→ B-8） |

**处理方式：**
- 删除 L2950-L3042 区间（旧版 `_handle_reject` + `_archive_pipeline` 完整函数定义）
- 保留 `_find_archive`（L3044）和 `_fmt_ts`（L3060）—— 这两个被 `_handle_hash_status` 调用（L4021/L4031），**不是死代码**
- 清理对应的 `# ═══` 节标记

**风险控制：** 仅删除旧版定义，不修改新版函数逻辑。删除后搜索 `def _handle_reject` / `def _archive_pipeline` 确认每个函数仅 1 处定义。

**验收标准：**
- [ ] A-1：`_handle_reject` 在文件中仅剩余 1 处定义（L3499）
- [ ] A-2：`_archive_pipeline` 在文件中仅剩余 1 处定义（L3839）
- [ ] A-3：`_find_archive`（L3044）和 `_fmt_ts`（L3060）未被连带删除
- [ ] A-4：`##status` 命令仍能正确查询归档管线（依赖 `_find_archive`）
- [ ] A-5：所有 `# ═══` 节标记对齐，无断裂

### 修复 C — _archive_pipeline 归档通知（B-8）

> **来源：** R124 QA B-8
> **预计：** +3 行

**问题：** 活跃版 `_archive_pipeline`（L3839-L3888）归档写入成功后**没有调用 `_notify_pm`**。旧版（L3000-L3041）有 L3039 行 `await _notify_pm(ctx, len(ctx.steps), "archived", "")`，但该旧版即将被删除（修复 A）。

**修复：** 在活跃版 `_archive_pipeline` 的 `try` 块内，归档写入成功后加上：

```python
await _notify_pm(ctx, len(ctx.steps), "archived",
    f"📦 {round_name} 管线已完成并归档")
```

**验收标准：**
- [ ] C-1：`_archive_pipeline` 归档写入成功分支调用 `_notify_pm`
