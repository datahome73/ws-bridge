## 详细分析

### 1. `!` 命令路由范围受限 🔴

**现象：**
`handle_broadcast` 中 `!` 前缀命令的解析和调度仅在 `_admin` 频道分支（≈行 1608-1649）。即使 `step_complete` 被注册为 `workspace_scope: True`，发到工作频道的 `!step_complete` 只会被当成普通文本广播给全员，服务端不会执行 `_cmd_step_complete`。

**影响：**
- 工作区成员无法在工作频道直接交接，必须通过原始 WS 连 `_admin` 频道执行
- 小爱通过 Hermes agent 发消息到工作室时，`!` 命令走 adapter 通道，更加不会触发服务端解析
- 整条管线依赖 `!step_complete` 驱动状态机，但该命令实际不可达

**修复方向：**
在 handler.py 的工作频道路由分支（`channel != p.LOBBY and resolved_workspace` 块）中，在广播前增加 `!` 命令拦截，调用 `_parse_command` + `_check_command_permission` + 执行 handler。或：在工作频道路由分支中，检测到 `!` 前缀时转发到 `_admin` 频道处理。

**参考位置：** `server/handler.py` ≈行 1608-1649（_admin 分支） vs 1650+（工作频道广播分支）

### 2. `!step_complete` 权限 🔴

**现象：**
注册为 `min_role: 3`（工作区管理员或全局管理员），普通 member 角色成员无法执行。工作区里的小开（arch）、泰虾（member）都无法自驱调用。

**修复：**
R48 已改为 `min_role: 1`（commit `adb4a93`），待部署上线后所有认证成员均可执行。

### 3. 角色映射缺失（F-16） 🟡

**现象：**
`PIPELINE_STEP_MAP` 硬编码 `arch/dev/review/qa/admin` 五角色。`_cmd_pipeline_start` 从 `auth.get_users()` 按 `role` 字段过滤成员加入工作区。但现有 agent 注册时的角色均为默认 `member`，管线点名 `review`/`qa`/`dev` 时匹配不到任何成员。

**R48 实际影响：**
- `!step_complete step2` 后点名 dev → 找不到 dev 角色成员
- `!step_complete step3` 后点名 review → 找不到 review 角色成员
- `!step_complete step4` 后点名 qa → 找不到 qa 角色成员
- 每步只能靠外部手动通知对应人员接手

**修复方向（见 TODO F-16）：**
用 Agent Card 模式让各 agent 自行声明能力/角色，服务端持久化角色映射到运维数据层（非代码层），`!pipeline_start` 从持久化数据中按需拉取。

### 4. 管线状态不同步（F-17） 🟡

**现象：**
R48 实际工作进度：
- Step 2 技术方案 ✅（小开产出）
- Step 3 编码 ✅（commit 7a299a9）
- Step 4 审查 ✅（11/11 通过）
- Step 5 测试 ✅（25/25 通过）

但管线状态直到手动在 `_admin` 调用 `!step_complete` 之前一直卡在 step2。因为各角色完成了工作但没有调用 `!step_complete`——这不是恶意，而是：
- 不知道需要调用
- 权限不够调用不了
- 工作频道调了也没用（#1）

**修复方向：**
- #1 和 #2 修好后，成员可自驱 `!step_complete`
- 或建立超时检测：Step 超时后自动催办/升级
- 或建立状态一致性核查：定期对比 git commit 与 pipeline state

### 5. 开发与部署分离 🟢

**已知约束：** 开发过程中不部署（大宏原则）。R48 的权限修复（#2）虽然已 commit，但必须等 Step 5 测试全部通过、Step 6 统一部署后才能生效。这本身不是 bug，但理解这个约束有助于规划多轮次依赖的修复顺序。

### 6. Step 完成后无 TG 通知 🟡

**现象：**
R48 各 Step 实际完成后，大宏没有收到任何主动通知，需要主动来问「到了哪个 step」。

**R48 方向 B 已修：** Step 6 完成时服务端向 `_admin` 频道写入 `🔔 [PIPELINE_COMPLETE]` 完结消息，PM 看到后 TG DM 大宏。但中间 Step（2/3/4/5）的完成仍无主动 TG 通知。

**修复方向：**
- 中间 Step 完成后，也能触发 TG 通知给大宏
- 或 PM 在工作群看到进度后主动汇报
