## 逐项评估

### Bug A — 🔴 核心改动，方案扎实

- 根因链完整（A1 内存态 + A2 轮询盲区 + A3 异常吞没）✅
- `localStorage` 双重保险设计合理：即时恢复（无网络）→ 异步 API 验证 ← 兜底 ✅
- 4 处改动坐标精确到行号，可读性强 ✅
- ⚠️ `init()` 改动量较大（替换 L390-401 为 ~40 行），需确保 **API 错误分支** 降级时 `restoredTab2` 逻辑覆盖所有路径，避免出现既无 localStorage 又无 API 数据的空状态

### Bug B — 🟡 防御性修复，方案诚实

- 根因定性准确：**非代码 Bug**，是 Docker volume 配置问题 ✅
- 前端自愈（401 降级 + WS 断连检测）在代码层面是最佳可作范围 ✅
- ⚠️ `validate_token()` 改动偏轻（仅 +debug log），确认是否需要在 `web_viewer.py` 增加 session 文件缺失时的 `try-reload` 重试（当前方案因 `__main__.py` 被排除而未做）

### Bug C — 🟢 体验优化，但要注意范围边界

- 区分「暂无消息」vs「加载失败」是正确方向 ✅
- ⚠️ 但 C 的根因是**数据卷导致 SQLite 重建**（历史群数据真实丢失），前端区分提示只能改善信息呈现，**不能恢复数据**。建议在 PRD 中标注此边界
