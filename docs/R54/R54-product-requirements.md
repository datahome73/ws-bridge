## 1. 问题背景

### 1.1 R53 回顾

R53 完成了 ACK 确认制点名与派活，解决了通信可靠性问题：
- `!step_complete` 和 `!step_handoff` 的工作室通知定向发送
- 消息发送超时重试（最多 3 次、4 秒超时）
- 发起者和工作室成员 ID 双重去重
- 2 个并行工作室的 ACK 序列化隔离

但 ACK 确认制只解决了「消息送达」问题，没有解决「谁来推进 step」的问题。

### 1.2 自动驾驶方向

大宏在 R53 完成后提出了**自动驾驶管线协作设计原则**（`docs/R53/R53-self-driving-pipeline-design.md`），核心思想：

> **任何人都可以推进 step。** 共享 WORK_PLAN + step 状态 + git，相邻 step 互为审查，无产出则退回。

### 1.3 R54 目标

R54 作为设计评审轮次，目标是将 R53 的设计原则细化、评审、固化为可执行的 5 项技术方向和完整的行为规则体系，为 R55 编码实现做准备。

## 2. 方向说明

### 方向 A：放开 `!step_complete` 角色校验 🔴 P0

| 项目 | 说明 |
|:-----|:------|
| **当前** | `!step_complete stepN` 只允许 step 负责人执行 |
| **目标** | 允许任意成员推进任一处于 **pending** 的 step |
| **约束** | 仅允许推进 pending step；不可跳过当前活跃 step；2 秒序列化缓冲 |
| **服务端系统层** | ✅ — 纯规则校验 |

### 方向 B：新增 `!step_reject stepN --reason <原因>` 🔴 P0

| 项目 | 说明 |
|:-----|:------|
| **当前** | Step N 产出有问题时没有正式退回机制 |
| **目标** | 新增 `!step_reject` 命令，退回上一 step 并附理由 |
| **约束** | 退回必须附 reason；退回后 pipeline_status 显示退回状态；单 step 最多退回 2 轮 |
| **服务端系统层** | ✅ — 纯规则状态机操作 |

### 方向 C：基于 git 的自动验证 🟡 P1

| 项目 | 说明 |
|:-----|:------|
| **当前** | `!step_complete stepN --output <sha>` 不会验证 commit 是否存在 |
| **目标** | `!step_complete` 收到 `--output` 参数时，自动检查远程 git 是否存在该 commit |
| **约束** | 检查 `origin/dev`；`--output` 可选；超时 10 秒降级为警告 |
| **纯规则逻辑** | ✅ |

### 方向 D：Step 状态关联可视化 🟡 P1

| 项目 | 说明 |
|:-----|:------|
| **当前** | `!pipeline_status` 只显示当前活跃 step |
| **目标** | 显示各 step 状态：✅ ▶ 🔄 ⏳；被退回 step 附退回次数和理由 |
| **约束** | 状态从持久化层读取 |
| **纯规则逻辑** | ✅ |

### 方向 E：自动驾驶模式开关 🟢 P2

| 项目 | 说明 |
|:-----|:------|
| **当前** | 所有管线按传统模式运行 |
| **目标** | 每个管线可设 `auto`（自动驾驶）或 `manual`（传统模式），默认 `auto` |
| **约束** | 模式存储在管线 state 中；特定行为仅在 auto 模式下生效 |
| **纯规则逻辑** | ✅ |

## 3. 行为规则体系

以下为 R54 设计评审确认的 6 条行为规则，将在 R55 中正式写入 WORK_PLAN：

### 规则 1：共享上下文，主动推进

任何人都可以推进 step。不需要等 step 负责人主动调 `!step_complete`。只要通过 git 确认远程有产出（`git ls-remote origin dev` 有新 commit），任意角色都可以直接推进管线。

### 规则 2：相邻审查机制

| 当前 Step | 你要检查 | 如果没产出 | 如果有缺陷 | 如果 OK |
|:-----------|:---------|:-----------|:-----------|:--------|
| Step N+1 | Step N 的产出 | `!step_reject stepN --reason "无新产出"` | `!step_reject stepN --reason "缺陷说明"` | 直接推进到 Step N+2 或开始工作 |

### 规则 3：退回必附理由，最多 2 轮

退回必须附具体原因。同一 step 最多退回 2 轮，第 3 次自动升级给 PM 协调。

### 规则 4：始终有人在干活

| 场景 | 行为 |
|:-----|:------|
| Step N 退回修正 | 修正完推 git，推进到下一 step |
| 10 分钟无推进 | PM 主动介入核查 |
| 卡点无法解决 | PM → 项目负责人（TG 私聊） |

### 规则 5：不抢占活跃 Step

自动驾驶 ≠ 抢活。当前活跃 step 的负责人正在工作中时，其他人不要覆盖推进。只有在负责人已完成产出但不调 `!step_complete` 时，其他人再主动推进。

### 规则 6：Step 交接不回声

Step 交接产生的系统消息，仅被 @mention 的目标 bot 回复 ACK。其他角色零输出。
