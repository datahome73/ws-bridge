# R122 产品需求文档（Product Requirements）

> **起草人：** 📋 PM（小谷）
> **状态：** 📝 草稿
> **版本：** v1.0

---

## 1. 背景与目标

R106→R121 完成了管线自动推进的闭环，已实现从 `##start##R{N}` 创建到 Step 6 部署的全自动派活。但经过 R119/R120/R121 三轮实战验证，暴露了两个系统性风险：

1. **管线卡死风险** — 管线推进依赖 bot 按严格格式回复 `已完成 ✅ R{N} Step {N}`。一旦 bot 回复格式不符（人类习惯的不同措辞、多余空格、emoji 替换等），`_try_advance_pipeline` 的 regex 不命中，管线永久卡死。需要人工介入排查，否则任务永远停滞。

2. **消息噪音/回声问题** — PM（小谷）在某些场景下直接给 bot 的收件箱发消息（绕过 `_inbox:server` 中继），bot 回复时又回到 `_inbox:server`，经中继转发回 PM。这造成：
   - Web 端大量 PM↔bot 直连消息噪音
   - PM 收到 bot 的回复回声，可能再次触发 bot 回应（复读循环）
   - 管线日志被无关消息污染

**目标：在不引入大面积代码改动的前提下，增加管线推进的容错能力和消息路由的整洁度。**

---

## 2. 根因分析

### 2.1 管线卡死 — 现状分析

**当前推进链条：**

```
bot → _inbox:server → _handle_server_relay(规则2) → _try_advance_pipeline() → regex 匹配 → advance_step → _auto_dispatch(下一步)
```

**单点故障：** 整个链条的触发入口只有一条硬性 regex（第 2454 行）：

```python
m = re.match(r"已完成 ✅ R(\d+[A-Za-z0-9-]*) Step (\d+)", content)
```

格式不匹配时，`_try_advance_pipeline` 返回 `(False, "no match")`，然后进入中继规则的"无匹配→沉默入库"分支（第 2990-3006 行），消息入库留痕但不做任何推进。管线进度停在原地，无告警、无重试、无超时。

**实际卡死场景：**

| 场景 | 举例 | 结果 |
|:-----|:------|:-----|
| 多余空格 | `已完成 ✅ R122  Step 3`（双空格） | 不匹配 |
| 措辞差异 | `搞定 R122 Step 3` | 不匹配 |
| 标点变化 | `已完成✅R122 Step 3`（无空格） | 不匹配 |
| 加了前缀 | `好了已完成 ✅ R122 Step 3` | 不匹配 |
| emoji 替换 | `已完成 ✅ R122 第三步` | 不匹配 |
| bot 离线 | bot 根本不回复 | 无消息 → 无触发 |

### 2.2 回声噪音 — 根因分析

**根本原因：小谷（PM）没有走 `_inbox:server` + `to_agent` 派活协议。**

小谷每次派活给 bot 时，发消息的 `channel` 直接指向 bot 的收件箱（如 `_inbox:ws_0bb747d3ea2a`），而不是通过 `_inbox:server` + `to_agent` 路由。

**问题链路：**

```
小谷 → _inbox:ws_0bb747d3ea2a（直接发 bot 收件箱）
                              ↓
                      bot 收到派活消息（from_agent=小谷）
                              ↓
                      bot 处理完回复 → _inbox:小谷（回复给小谷本人，不经 _inbox:server）
                              ↓
                      小谷收到 bot 的原始回复（无中继过滤、无管线追踪）
```

**噪音链的三重后果：**

1. **回声噪音：** bot 的原始回复（含中间思考过程、状态更新等）直接暴露给小谷，Web 端充斥着大量 PM↔bot 私聊消息
2. **无管线追踪：** 消息不经过 `_inbox:server`，`_handle_hash_start` / `_try_advance_pipeline` 完全不知道这些消息的存在。管线的自动推进只认 `_inbox:server` 来的 `已完成 ✅` 消息
3. **触发复读：** 小谷收到 bot 的回复回声后，如果回复了 bot，bot 又回给小谷，形成 A↔B 的对话循环

**正确的做法：** 小谷派活应统一走 `_inbox:server` 中继，在消息中附加 `to_agent` 字段指向目标 bot 的 agent_id。这样：

```
小谷 → _inbox:server（含 to_agent=ws_xxx）
         ↓
     server 路由 → _inbox:ws_xxx（隐藏发送者为"系统"）
         ↓
     bot 回复 → _inbox:server（走 relay 规则，_try_advance_pipeline 可识别）
         ↓
     relay 过滤转发给小谷（仅关键进度通知）
```

## 3. 功能需求

### 需求 A — 管线超时告警（轻量级熔断）

> **动机：** 不让管线卡死后无人知晓。不需要复杂的自动恢复，只需要**及时发现 + 通知 PM**。

- **触发条件：** 管线某一步完成后，`_try_advance_pipeline` 派活下一步后，等待 bot 回复超过 **30 分钟无匹配**（从 `dispatched` 计时）
- **行为描述：**
  - 在 `pipeline_context` 中为 `in_progress` 状态的 step 记录 `dispatched_at` 时间戳
  - 一个轻量后台协程（每 5 分钟 tick）扫描所有 `running` 状态的管线
  - 检查每个管线中 `in_progress` 的 step 是否已超时（>= 30 分钟）
  - 超时后：发送告警通知给 PM（小谷的 `_inbox`），内容包含轮次、停滞的步骤、当前状态快照
  - **不做自动重试/自动跳过**——只告警，PM 决定如何处理
- **权限控制：** 仅系统后台触发，仅通知 PM
- **验收标准：**
  - [ ] A-1：step 标记为 `in_progress` 时自动写入 `dispatched_at`
  - [ ] A-2：5 分钟扫描协程正常运行，不阻塞主事件循环
  - [ ] A-3：step 正常完成时扫描跳过该 step
  - [ ] A-4：超时后 PM 收到告警通知，内容包含 `R{N}`、停滞的 `Step {N}`
  - [ ] A-5：无超时的管线扫描不产生任何副作用（不误告警）

### 需求 B — regex 松匹配（格式容错）

> **动机：** 允许 bot 在完成消息中有轻微格式偏差时仍能自动推进，减少卡死概率。

- **触发条件：** bot 发送消息到 `_inbox:server`，内容包含轮次和步骤完成信息
- **行为描述：**
  - 当前 regex：`已完成 ✅ R(\d+[A-Za-z0-9-]*) Step (\d+)`
  - 改为宽松匹配：使用多个 fallback 模式，按优先级尝试：
    1. ✅ 现有严格 regex（最高优先级，完全匹配直接前进）
    2. ✅ 容错模式 1：允许 `✅` 前后缺空格、允许多余空格
    3. ✅ 容错模式 2：允许以 `已完成` 或 `完成` 结尾变体，只要包含轮次 R{N} 和步骤号
    4. ✅ 容错模式 3：允许 bot 在额外文字前/后附带完成信息，只要**以** `已完成 ✅` 开头或**包含** `R{N} Step {N}` 的完成语义
  - 所有匹配成功后，统一执行相同的 `advance_step` + `_auto_dispatch` 逻辑
  - 匹配日志输出中注明命中哪一个 fallback 模式，便于调试
- **权限控制：** 无（系统自动处理）
- **验收标准：**
  - [ ] B-1：严格格式 `已完成 ✅ R122 Step 3` 仍正常匹配
  - [ ] B-2：缺空格 `已完成✅R122 Step 3` 正常匹配
  - [ ] B-3：前缀 `好了已完成 ✅ R122 Step 3` 正常匹配
  - [ ] B-4：不包含 `已完成 ✅` 的消息不被误匹配（如普通聊天）
  - [ ] B-5：日志标注当前匹配的模式编号

### 需求 C — 小谷派活协议规范化（统一走 `_inbox:server` + `to_agent`）

> **动机：** 根因在小谷不按 `_inbox:server` + `to_agent` 协议派活。修正后，bot 回复自然回到 `_inbox:server` 中继，噪音消失，管线可见。

- **根因分析：** 小谷的自动派活（`_auto_dispatch`）已经正确使用了 `_inbox:server` + `to_agent`。但小谷在以下场景会**直接发 bot 收件箱**（绕过中继）：
  - 手动补充派活/提醒 bot
  - 回复 bot 的中间状态消息
  - 启动管线后的第一步派活
  - 任何需要人工参与的交互

- **修复方案：** 修改小谷侧的 dispatch 逻辑，确保所有发往 bot 的消息统一走 `_inbox:server` + `to_agent`：
  - 发送 `channel` 固定为 `_inbox:server`
  - 附加 `to_agent` 字段指向目标 bot 的 agent_id
  - 不再直接发 `_inbox:{bot_agent_id}`（除非确实需要绕过中继的紧急场景）

- **服务端不需要改动** — `_handle_server_relay` 的 to_agent 路由（§2828-2874）已经实现了正确的转发逻辑，包括：
  - 隐藏发送者为"系统"
  - 消息落库追踪
  - relay 规则过滤（ACK/完成/退回/失败/沉默）

- **验收标准：**
  - [ ] C-1：小谷所有派活消息的 `channel` = `_inbox:server`，附加 `to_agent`
  - [ ] C-2：bot 收到派活后，`from_agent` 显示为"系统"而非小谷
  - [ ] C-3：bot 回复到 `_inbox:server`，relay 规则正确过滤
  - [ ] C-4：`_inbox:server` 中无 bot 直接回复给小谷的噪音消息
  - [ ] C-5：管线仪表盘正确追踪所有派活消息

## 4. 方向决定

| 决定事项 | 选择 | 说明 |
|:--------|:----|:-----|
| 管线卡死处理策略 | ⏰ **超时告警，不自动跳过** | 自动跳过可能跳过真正需要的步骤。PM 接入工单后手动处理更安全 |
| regex 改进范围 | **增补模式，不改现有逻辑** | 在现有严格 regex 基础上加 fallback，不破坏已有匹配 |
| 回声处理方向 | **需求 C — 小谷派活协议规范化** | 根因在小谷不按 `_inbox:server` + `to_agent` 协议，修正后噪音自行消失 |
| 涉及代码改动 | 仅 `main.py` 和 `pipeline_context.py` | 服务端已有 to_agent 路由支持，不需改动 |

---

## 5. 不做事项（明确排除）

| 排除项 | 理由 |
|:-------|:-----|
| ❌ 自动重试超时 step | 可能跳过关键步骤或引入不一致 |
| ❌ 超时后自动跳下一步 | 跳过不等于完成，会导致管线状态混乱 |
| ❌ 服务端路由静默化（屏蔽 bot 回复） | 根因在小谷不按协议发消息，不是服务端路由问题 |
| ❌ Bot 侧代码修改 | 只要小谷走 `_inbox:server` + `to_agent`，bot 回复自然走 relay |
| ❌ 解耦 pipeline manager 为独立微服务 | 改动太大，系统尚不稳定 |
| ❌ 引入外部消息队列（Redis/RabbitMQ） | 过度工程 |
| ❌ 新增 Web 端 bot 回复查看功能 | 修复派活协议后噪音消失，不需额外功能 |
