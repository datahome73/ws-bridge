> **版本：** v0.2（基于项目负责人反馈更新）
> **状态：** 📝 草稿（待审核）
> **产品经理：** 🧐 PM
> **日期：** 2026-06-26
> **本轮改动范围：** 仅第①类（服务器代码），扩展管线看门狗与超时机制

## 1. 背景与痛点

### 当前管线卡死场景

R42 上线了 `!pipeline_start` + `!step_complete` 自动接力系统，管线不再靠人工 @mention 推进。但存在多个关键缺口：

**缺口 A — Step 超时无感知：** 管线依赖责任人主动调用 `!step_complete`。一旦该角色忘记、卡壳、离线，整个管线永久挂起。

```
时间线：
  pipeline_start → Step 2 技术方案 → 架构师出方案 ...
    ↓ 推 dev
  Step 2 完成（但架构师忘了执行 !step_complete）
    ↓  ⏳ 无人点名下一棒
  所有角色进入「等信号→无人发信号→继续等」死循环
    ↓ 🔴 F-11 Hot Standby 死锁
  PM 不人工巡检就无法发现
```

**缺口 B — 点名响应无反馈：** 即使 `!step_complete` 被正确调用，下一角色可能因为频道未切换或离线而收不到点名消息。点名者不知道下一棒是否已收到，管线可能在「点名已发出但无人响应」的状态下静默停滞。

### 已有基础

| 已有能力 | 状态 | 说明 |
|:---------|:----:|:------|
| `_PIPELINE_STATE` | ✅ | 内存中记录活跃管线名称、当前 Step、ws_id、启动时间 |
| `!pipeline_status` | ✅ | 可查询活跃 Step 进度，但需人工手动调用 |
| `_admin` 频道 | ✅ | 常驻频道，可接收系统级通知 |
| `!step_complete` | ✅ | Step 完成标记 + 自动点名下一人 |
| Step 映射表 | ✅ | `server/config.py` 中 `PIPELINE_STEP_MAP` 可配置 |

### 不是问题的情况

- `!step_complete` 被正常调用且下一角色在线 → ✅ 自动接力正常
- 初始点名阶段 3 分钟超时 → ✅ 已有 `_rollcall_timeout` 处理未确认成员

## 2. 预期体验

### 改进后

```
Step 2 技术方案启动
  → 架构师出方案 → 推 dev → 调用 !step_complete ✅ 正常流转
    ↓
  自动点名 dev-bot（附上下文）
    ↓
  ✅ dev-bot 收到并开始工作 → 正常推进
  … 但如果 dev-bot 未响应 …
    ↓
  10分钟后看门狗检测：Step 3 超时？不，Step 2 刚完成，还没到超时
  … 但架构师注意到 dev-bot 没动静 …
    ↓
  架构师：检查 → dev-bot 离线 → TG DM 项目负责人
    ↓
  项目负责人：收到 code 块消息，复制粘贴转发给 dev-bot 处理
    ↓
  dev-bot 恢复正常 → 管线继续

如果没有缺口 A 的看门狗兜底：
  … 架构师完成 Step 2 但忘了 !step_complete …
    ↓
  Step 2 启动已 >6h？不，刚完成
  但 !step_complete 没被调用 → Step 2 仍然活跃
    ↓
  再等 10 分钟 → 仍然没动静
  … 从 Step 2 启动算起超过 6 小时 …
    ↓
  ⚠️ 看门狗触发：_admin 频道告警「R43 Step 2 已挂起超 6h」
    ↓
  PM 收到通知，介入协调
```

### 关键变化

1. **不再静默挂起** — 任何 Step 挂起超过配置时间，一定有通知到 PM
2. **PM 超时感知** — 不靠人工巡检 `!pipeline_status`，看门狗主动推
3. **点名响应可观测** — 点名者可在合理时间内判断下一角色是否已收到消息
4. **交接有逃生通道** — 下一角色离线/频道未切换时，点名者通过项目负责人协调处理
5. **不影响正常流程** — Step 按时完成时看门狗无输出，正常接力不受影响

## 3. 需求详述

### 方向 A — 看门狗定时器 🟢 高

新增一个后台异步任务，定期扫描所有活跃管线，检测当前 Step 是否超时。

| 行为 | 说明 |
|:-----|:------|
| 扫描周期 | **每 10 分钟**扫描一次活跃管线（硬编码，非配置） |
| 检测条件 | 当前 Step 的 `started_at` 时间戳 + 该 Step 的超时阈值 → 判断是否超时 |
| 触发时机 | 服务启动时自动启动看门狗；服务关闭时自动终止 |
| 触发后行为 | 仅做「超时判断 + 通知」，不做自动跳过/重派 |

> 技术方案（看门狗启动时机、生命周期管理、避免重复通知）由架构师决定。

### 方向 B — Step 超时配置 🟢 高

为每个 Step 配置可自定义的超时阈值，存放在 `PIPELINE_STEP_MAP` 中扩展：

| 配置项 | 类型 | 默认值 | 说明 |
|:-------|:----:|:------:|:-----|
| `timeout_hours` | float | 按角色不同 | 该 Step 允许的最大挂起小时数 |
| `escalation` | str | `"notify_pm"` | 超时后的升级行为 |

**初始默认值（运行 2 轮后根据实际数据调优）：**

| Step | 角色 | 默认超时（小时） | 理由 |
|:----:|:----:|:----------------:|:------|
| 技术方案 | arch-bot | 6h | 需求已定，出方案不应超过半天 |
| 编码 | dev-bot | 12h | 编码含调试，需要更长时间 |
| 代码审查 | review-bot | 4h | 审查应快速闭环 |
| 测试验证 | qa-bot | 6h | 含部署 + 测试执行 |
| 合并部署归档 | admin-bot | 2h | 纯操作类，应快速完成 |

> **关于超时值：** 当前初始值偏保守。前 2 轮运行期间项目负责人会通过 Web 端观察 + TG 协调，积累数据后再调优。

### 方向 C — 超时升级通知 🟡 中

当看门狗检测到 Step 超时时，向 `_admin` 频道发送一条告警消息。

**通知格式示例：**
```
⚠️ R43 管线超时告警
  Step: 技术方案（Step 2）
  责任人: arch-bot
  已挂起: 7 小时（超时阈值: 6h）
  启动时间: 2026-06-25 14:30
  建议操作: 联系 arch-bot 或考虑换人
```

**通知规则：**

| 规则 | 说明 |
|:-----|:------|
| 首次超时 | 立即发送告警 |
| 重复通知 | 首次超时后，**每 30 分钟重复一次**（如果仍未完成），避免静默反复 |
| 解除通知 | 当 `!step_complete` 被调用，Step 标记完成时 → 自动发送「✅ R43 Step 2 已恢复」解除告警 |
| 通知目标 | `_admin` 频道（PM 可见） |

**重复通知间隔的推理：** 看门狗每 10 分钟扫描一次确认超时，首次超时即告警。如果 Step 在预计完成时间（10 分钟）内没完成，30 分钟后再补一次提醒，此时距 Step 启动已约 40 分钟——足够判断是否异常。

> 技术方案（通知去重、重复间隔配置）由架构师决定。

### 方向 D — 交接响应机制 🟡 中

`!step_complete` 调用 `!rollcall_next` 点名下一角色后，如果下一角色长时间无响应，当前角色需要一条逃生通道。

**核心问题：** 点名消息已发出，但下一角色可能因为：
- 频道未切换过来（活跃频道未指向工作室）
- 机器人离线 / 服务异常
- 消息队列积压未处理

而点名者（上一 Step 的完成者）无法判断下一人是否已收到。

**处理流程：**

```
!step_complete → 点名下一角色 → 返回点名确认
  ↓
当前角色观察：下一角色是否在合理时间内开始工作？
  ├─ ✅ 正常开始 → 继续
  └─ ⏳ 长时间无响应（~5 分钟无动静）
       → 检查一下（确认是否离线或频道问题）
       → 如有问题 → TG DM 项目负责人，附 code 块消息
```

**对服务端的要求：** `!step_complete` 返回消息中需明确告知点名已发出、目标角色是谁。辅助当前角色判断「点名已送达」。

**对流程的要求：** 当前角色需要在自己完成后关注下一角色的响应情况。看到点名返回确认后，等待片刻观察。如果下一角色迟迟未开始工作，主动检查 + 通过项目负责人协调。

**转发给项目负责人的消息必须用 code 块封装：**

```
把下面这段话转发给 arch-bot：

```
🏗️ arch-bot，R43 Step 2 技术方案已点名给你，
但没有收到你的响应，请检查活跃频道是否已切换到工作室。
如果离线请联系项目管理恢复。
```
```

> 此方向侧重流程规范，服务端改动较小（仅需在 `!step_complete` 返回值中增强点名确认信息）。具体 code 块消息模板在管线运行中由当前角色自主撰写。

## 4. 架构原则

### 4.1 纯服务端系统层，零 token 消耗

所有超时检测、通知逻辑放在 `handler.py` 的后台任务中。AI Agent 不参与看门狗逻辑。

### 4.2 仅通知，不自动操作

看门狗只做检测和通知，不做自动跳过/重派/关闭等操作。PM 收到通知后人工决策。这是 `!step_complete` 命令的基本哲学延续——Step 流转由角色自主触发，系统不强制推进。

### 4.3 向后兼容

- 无活跃管线时，看门狗空扫描，零输出
- 超时配置可省略（省略则使用默认值）
- 不影响旧流程（人工 @mention 接力）

### 方向 A — 看门狗定时器

| # | 验收标准 | 优先级 |
|:-:|:---------|:------:|
| A-1 | 服务启动后看门狗自动启动，**每 10 分钟**扫描一次活跃管线 | 🔴 P1 |
| A-2 | 无活跃管线时，看门狗扫描零输出 | 🟢 P3 |
| A-3 | 看门狗检测到 Step 超时时，生成超时告警（不直接发送，由方向 C 处理） | 🔴 P1 |
| A-4 | 看门狗不会对同一 Step 发送重复告警（不重复判断标志记录） | 🟡 P2 |
| A-5 | 看门狗在服务器停止时自动终止 | 🟢 P3 |

### 方向 B — Step 超时配置

| # | 验收标准 | 优先级 |
|:-:|:---------|:------:|
| B-1 | `PIPELINE_STEP_MAP` 中每个 Step 可配置 `timeout_hours` | 🔴 P1 |
| B-2 | `PIPELINE_STEP_MAP` 中每个 Step 配置了 `escalation` 字段 | 🟡 P2 |
| B-3 | 未配置 `timeout_hours` 的 Step 使用默认超时值 | 🟡 P2 |

### 方向 C — 超时升级通知

| # | 验收标准 | 优先级 |
|:-:|:---------|:------:|
| C-1 | Step 超时时，`_admin` 频道收到格式完整的告警消息 | 🔴 P1 |

| C-4 | 首次超时后，**每 30 分钟**重复通知一次（如仍未完成） | 🟡 P2 |
| C-5 | 通知使用纯文本格式（不含 Markdown 代码块），兼容 `_admin` 频道广播语义 | 🟡 P2 |

### 方向 D — 交接响应机制

| # | 验收标准 | 优先级 |

| D-2 | 当前角色在点名发出后可判断下一角色是否已响应（通过观察工作室活跃度） | 🟢 P3 |
