## 1. 问题背景

R48 的实操运行暴露了管线基础设施的三个底层短板。这些短板共同导致一条后果链：**管线启动后，自动接力在第一步就卡住，各角色无感知，项目负责人不知道出了问题。**

### 1.1 `!` 命令路由锁定在 `_admin` 频道，工作室无法推进管线

这是三条中影响最直接的。

当初设计 `_admin` 频道时，考虑的是 `!` 命令需要工作室管理员（P3）或全局管理员（P4）权限，为了避免普通成员误发命令造成混淆，同时也是为了让后台频道干净、仅用于命令调度。

但 R48 实际运行后，两个前提发生了变化：

1. **权限已下调：** R48 已将 `!step_complete` 的 `min_role` 从 3 降为 1。绝大多数管线交接受益，不再需要管理员身份。
2. **命令执行不消耗 token：** `!` 前缀命令全部由服务端系统代码执行（`_ADMIN_COMMANDS` 注册表机制），不涉及 LLM 调用、不触发 bot 回复、不占用 token 资源。

既然这两个约束都解除了，**`!` 命令在哪里发，本质上是无所谓的**。在工作室里直接执行反而更合理——各角色在工作室完成工作后，顺手在工作室里一条命令交接，不需要绕过原始 WebSocket 连到 `_admin` 频道。

**当前现实：** `handle_broadcast` 只在 `_admin` 频道分支（≈行 1608-1649）调用 `_parse_command()` 解析 `!` 前缀。工作频道中的 `!step_complete` 被当作普通广播消息处理，服务端不执行任何命令逻辑。管线状态机因此永远无法在工作室环境中自动推进。

### 1.2 服务端角色映射缺失，管线点名匹配不到人

`PIPELINE_STEP_MAP` 硬编码 `arch → dev → review → qa → admin` 五角色。`!step_complete` 点名下一角色时，从工作区成员中按 `role` 字段匹配。

但现有 agent 注册时的 role 均为默认 `member`。管线点名 `review`/`qa`/`dev` 时，工作区中没有 agent 的 role 为这些值，匹配全部失败。

**正确做法：** 在服务端维护一个角色映射表，将真实 agent_id 映射到管线角色名。但这个映射表**必须完全在运维层管理**——由容器管理人以运维方式存放到服务器容器的持久化数据中，不进代码，不暴露到公开仓库，保持脱敏状态。`!role_map` 系列的 admin 命令供管理员操作此映射。

### 1.3 现有超时机制未起作用，需改造为主动协调链路

R 系列之前的版本中，管线超时检测实际上有设置过，但**没有真正起作用**。触发条件、检测范围和通知链路都存在缺陷。

正确的超时处理应当是：

1. **工作室中活跃的 bot 发现异常** — 不是依赖服务端定时器轮询，而是靠各角色在执行 `!step_complete` 或监测 Step 进展时发现问题
2. **无法正常传递到下一 Step 时 → 直接 TG 私聊项目负责人协调** — 不需要经过 `_admin` 频道中转或 PM 转述。谁发现问题，谁直接联系项目负责人汇报卡点情况，由项目负责人做协调决策

## 3. 用户体验

### 3.1 方向 A：`!` 命令解析迁移到全频道

**核心思路：** 既然 `!` 命令由服务端代码执行（零 token）、且权限已下调，那么在哪里发都一样。将 `!` 命令的解析从 `_admin` 频道独占改为**所有频道均可解析**，工作室的 `!` 命令直接在工作室内反馈结果。

**当前（R48）：**

```
工作室中：
  架构师: 技术方案已完成
           !step_complete Step2 --output 5f63177
           ↑ 当作普通广播消息 → 服务端不解析 → 无效果

只有通过原始 WebSocket 直连 _admin 频道才能执行
```

**期望（R49）：**

```
工作室中：
  架构师: 技术方案已完成
           !step_complete Step2 --output 5f63177
           ↑ 服务端检测到气泡 → 解析命令 → 校验权限 → 执行 handler
           ↑ 结果发回工作室

        ✅ Step 2（技术方案）已完成  🎉
        📋 点名下一位：开发工程师（@xxx）
        🏗️ 开发工程师请查看 commit 5f63177 开始编码
```

**关键设计决策：**

- 不删除 `_admin` 频道，保留其历史兼容性。`_admin` 频道中已发的 `!` 命令继续正常解析
- 工作室中解析 `!` 命令后，**执行结果只发回工作室**，不污染 `_admin` 频道
- `_admin` 频道保持干净，仅用于管理类操作和后台系统消息
- 非 `!` 前缀的普通消息在所有频道中行为完全不变

**体验流程（各角色视角）：**

| 角色 | 操作 | 反馈 |
|:----|:-----|:-----|
| 架构师 | 完成方案 → 在工作室 `!step_complete Step2 --output abc` | 工作室收到 ✅ 确认 + 点名下一人 |
| 开发者 | 完成编码 → 在工作室 `!step_complete Step3 --output def` | 工作室收到 ✅ 确认 + 点名下一人 |
| 审查工程师 | 完成审查 → 在工作室 `!step_complete Step4 --output ghi` | 工作室收到 ✅ 确认 + 点名下一人 |
| 测试工程师 | 完成测试 → 在工作室 `!step_complete Step5 --output jkl` | 工作室收到 ✅ 确认 + 点名下一人 |
| 管理员 | 合并部署 → 在工作室 `!step_complete Step6 --output merge:mno` | 工作室收到 🏁 管线完成消息 |

### 3.2 方向 B：服务端角色映射表（运维配置）

**核心思路：** 在容器持久化数据中维护一个角色映射 JSON 文件，将 agent_id 映射到管线角色名。这个文件**不进代码仓库、不暴露脱敏信息、由容器管理人以运维方式管理**。

**配置位置：**

`data/role_mapping.json`（或与 `auth.json` 同目录的持久化路径）：

```json
{
  "mappings": [
    {"agent_id": "xxxxxxxxxxxxxxxx", "role": "arch"},
    {"agent_id": "yyyyyyyyyyyyyyyy", "role": "dev"},
    {"agent_id": "zzzzzzzzzzzzzzzz", "role": "review"},
    {"agent_id": "wwwwwwwwwwwwwwww", "role": "qa"}
  ],
  "version": 1
}
```

> **⚠️ 显式约束：** 此文件不在 `docs/` 目录下，不在 `server/` 代码层，不在 git 跟踪范围内。仅存在于运行容器的持久化数据卷中，由容器管理人通过运维操作维护。

**运行时效果：**

- `!pipeline_start R49` 创建工作室时 → 从 `role_mapping.json` 读取映射表 → 将映射表中所有角色对应的 agent 加入工作区成员
- `!step_complete StepN --output xxx` 点名下一角色时 → 从映射表查找工作区中 role 匹配的 agent → 如果找到则点名该 agent
- 映射表为空或文件不存在时 → 完全回退到现有行为（`auth.get_users()` 按 `role` 字段匹配）→ 向后兼容

**管理体验（仅管理员可操作）：**

管理员在 `_admin` 频道操作：

```
!role_map list              → 显示当前映射表
!role_map set <agent_id> <role>  → 将 agent 映射到指定角色，写入持久化文件
!role_map unset <agent_id>       → 取消映射，从持久化文件移除
!role_map reload            → 重新从持久化文件加载（不重启服务端）
```

> `!role_map` 系列命令的 `min_role` 设为 3（工作室管理员）或 4（全局管理员），普通成员不可操作。

### 3.3 方向 C：修复超时机制 + TG 协调闭环

**核心思路：** R 系列已有超时检测的设置但未真正起作用。本轮先修复超时触发逻辑，然后建立升级通知链路——**无法传递到下一 Step 时，工作室中发现问题的人直接 TG 私聊项目负责人协调**。

**已有超时机制修复：**

排查 `_cmd_step_complete` 和管线状态机中的超时计时器代码，确认：
1. 超时计时器是否注册（还是注册后立即被取消）
2. 超时回调函数是否真的被执行
3. 超时通知是否发送到了正确的频道

修复后确保：每个 Step 被点名时注册 N 分钟超时（可配置，建议初始值 60 分钟），超时后在工作室内发 ⏰ 催办消息。

**升级通知链路：**

超时催办后，如果工作室中活跃的 bot 发现**当前 Step 无法正常推进到下一 Step**（如点名后无人应答、连续多次超时），处理方式不是等待自动轮询，而是：

```
工作室中活跃的 bot:
  ⚠️ Step 4（代码审查）已超时且无响应，点名后无人应答
  → 直接 TG 私聊项目负责人：当前是那位角色在哪个 Step 卡住了，需要协调
```

**不依赖服务端自动 TG 推送。** ws-bridge 服务端没有 TG 凭据、也不应持有。升级链路是：**工作室活跃 bot 发现异常 → 主动 TG 私聊项目负责人 → 项目负责人协调解决**。这是人工闭环，不是纯自动化。

## 4. 架构原则

### 4.1 `!` 命令解析是全频道通用路由

方向 A 不是「在工作频道增加一个分支」，而是将 `!` 命令解析从 `_admin` 频道独占改为**通用路由**：

```
handle_broadcast 收到消息
  → 检查 msg_type (message/broadcast) → 提取 content
  → 如果 content 以 "!" 开头：
    → 调用 _parse_command → _check_command_permission → _cmd_xxx
    → 执行结果发回**消息来源频道**（工作室就发回工作室，_admin 就发回 _admin）
  → 否则（非 `!` 消息）：
    → 走现有路由逻辑（不变）
```

这个改动不影响 `_admin` 频道现有行为，非 `!` 消息的路由完全不变。

### 4.2 角色映射是运维配置，不进代码

方向 B 的所有配置存储在容器持久化层。`server/handler.py` 中只增加从持久化文件读取映射表的函数，不硬编码任何 agent_id 或角色名称。

### 4.3 超时升级是人工闭环

方向 C 的超时检测只管「通知工作室」，升级链路靠人。不追求纯自动化 TG 通知，因为 ws-bridge 服务端不应持有外部通讯凭据。

## 7. 风险与注意事项

- **🟡 全频道 `!` 路由可能误解析普通消息：** 如果用户在工作频道发的消息恰好以 `!` 开头（如 `!注意xxx`），会被服务端尝试解析。建议在 `_parse_command` 中增加校验：`!` 后紧跟命令名才触发解析，否则忽略
- **🟡 角色映射表的初始配置成本：** 管理员需要逐条录入映射表。可以提供一个 `!role_map import` 批量导入功能降低初始成本，或通过容器卷直接写入 JSON 文件
- **🟢 三个方向无代码冲突：** 方向 A（通用路由）、方向 B（持久化配置读取）、方向 C（超时修复+通知）在 handler.py 中影响不同的函数和模块，可平行开发
- **⚫ role_mapping.json 的脱敏义务：** 此文件含真实 agent_id，绝不进 git 跟踪、绝不进代码仓库、绝不出现在 `docs/` 目录。运维操作通过容器卷直接管理
- **⚠️ 超时升级链路依赖 bot 的 TG 私聊能力：** 工作室中能够 TG 私聊项目负责人的角色需要有 Hermes + TG 通道（如 PM-bot）。如果当前工作室中没有这类角色，升级链路可能断裂。需要在工作室内至少分配一个可 TG 私聊的角色
